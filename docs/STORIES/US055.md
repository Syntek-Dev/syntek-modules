# US055 — `@syntek/ui-settings`: Settings Pages & Tenant Admin UI

**Last Updated**: 10/03/2026 **Version**: 0.2.0 **Status**: To Do **Epic**: Epic 7 — Web Frontend
Features

## User Story

As a developer, I want a settings UI package so that every Syntek application has consistent
end-user settings pages (profile, security, billing, notifications), a `useSettings` hook to consume
tenant settings in any component, and a `TenantSettingsPanel` admin component for tenant admins to
manage tenant-wide settings — all wired to the GraphQL API from `syntek-settings` (US016) via
`@syntek/api-client`.

## What It Exposes

### `@syntek/ui-settings/hooks` — Tenant settings consumption

```typescript
import { useSettings, useTenantSettingsAdmin } from "@syntek/ui-settings/hooks";

// Consumer hook — read a tenant setting value in any component
const maxMembers = useSettings("max_team_members");
// Returns: typed value (int, bool, str, etc.) as returned by the GraphQL tenantSettings query.
// Handles loading state, caching, and re-fetching on mutation.

// Options:
const value = useSettings("max_team_members", {
  defaultValue: 10, // fallback while loading or if key not found
  suspense: false, // boolean — enable React Suspense (default false)
  refetchInterval: 0, // number (ms) — 0 = no polling (default)
});

// Admin hook — read and write tenant settings (requires manage_tenant_settings permission)
const { settings, updateSetting, resetSetting, isLoading } = useTenantSettingsAdmin();
// settings: TenantSettingEntry[] — all keys with value, type, description, group, isOverridden
// updateSetting(key, value): mutates via updateSetting GraphQL mutation
// resetSetting(key): resets to schema default via resetSetting mutation
```

### `@syntek/ui-settings/admin` — Admin management components

```typescript
import { TenantSettingsPanel, DesignTokensEditor } from '@syntek/ui-settings/admin';

// Grouped settings management panel — lists all registered settings with inline edit
// Groups settings by 'group' field from schema; renders appropriate input per type
// (toggle for bool, number input for int/float, text for str, colour picker for CSS tokens)
<TenantSettingsPanel
    onSaveSuccess?: () => void
    onSaveError?: (error: Error) => void
    readOnly?: boolean           // boolean — show values without edit controls
/>

// Design token override editor — colour pickers, font selectors, preview panel
<DesignTokensEditor
    previewComponent?: ReactNode  // live preview of the token changes
    onSaveSuccess?: () => void
/>
```

### `@syntek/ui-settings/pages` — End-user settings pages

```typescript
import {
    SettingsShell,
    ProfileSettings,
    SecuritySettings,
    NotificationPreferences,
    BillingSettings,
} from '@syntek/ui-settings/pages';

// SettingsShell — sidebar navigation wrapping all settings sections
// Each section loads data independently; unsaved changes in one section do not affect others
<SettingsShell sections={['profile', 'security', 'notifications', 'billing']} />

// Individual sections can be used standalone:
<ProfileSettings />       // name, email, avatar
<SecuritySettings />      // change password, MFA management, active sessions
<NotificationPreferences /> // per-channel toggles (calls syntek-notifications API)
<BillingSettings />       // delegates to @syntek/ui-payments components
```

**Key rules:**

- `useSettings(key)` reads from the `tenantSettings` GraphQL query. It returns the resolved value
  for the current tenant — the schema default if no override exists. It does NOT write.
- `useTenantSettingsAdmin` requires the user to have `manage_tenant_settings` permission (enforced
  at the GraphQL resolver — the hook simply calls the mutation and surfaces the error).
- `TenantSettingsPanel` renders input types automatically based on the `type` field in the
  `tenantSettings` response: `bool` → toggle, `int`/`float` → number input, `url`/`email`/`str` →
  text input, `json` → code editor.
- `DesignTokensEditor` requires `DESIGN_TOKENS_ENABLED: True` on the backend. If the `designTokens`
  query returns null, the editor renders a disabled placeholder.
- `SecuritySettings` gates the password change and MFA management sections behind a
  re-authentication prompt (delegates to `@syntek/ui-auth`).
- All components accept `className` prop and use `@syntek/ui` design tokens — fully overridable.

## Acceptance Criteria

```gherkin
# --- useSettings hook ---

Given 'max_team_members' is declared in SYNTEK_SETTINGS SCHEMA with default 10
When useSettings('max_team_members') is called and tenant has no override
Then 10 is returned (correctly typed)

Given tenant has set 'max_team_members' to 25
When useSettings('max_team_members') is called
Then 25 is returned

Given useSettings is called with defaultValue: 99 and the GraphQL query is loading
When the component renders
Then 99 is returned during loading — no null/undefined flash

# --- useTenantSettingsAdmin ---

Given the current user has manage_tenant_settings permission
When useTenantSettingsAdmin().updateSetting('max_members', 50) is called
Then the updateSetting mutation fires, the cache updates, and the new value is reflected

Given the current user does not have manage_tenant_settings permission
When updateSetting is called
Then the GraphQL mutation returns PermissionDenied and the hook surfaces the error

Given resetSetting('max_members') is called for a key with a tenant override
When the mutation resolves
Then the key returns to its schema default and isOverridden is false

# --- TenantSettingsPanel ---

Given TenantSettingsPanel renders with settings grouped by 'group'
When the component mounts
Then settings are organised into labelled groups matching the schema

Given a bool setting renders in TenantSettingsPanel
When it renders
Then a toggle switch is shown

Given an int setting renders
When it renders
Then a number input is shown with correct min/max if validators declare bounds

Given a user changes a setting value and saves
When onSaveSuccess fires
Then the panel reflects the updated value without a full page reload

Given readOnly is true
When TenantSettingsPanel renders
Then all inputs are disabled — no mutations can be triggered

# --- DesignTokensEditor ---

Given '--color-primary' is set to '#FF6B00' for the current tenant
When DesignTokensEditor renders
Then the colour picker shows #FF6B00

Given the user changes '--color-primary' to '#0057FF'
When save is triggered
Then updateDesignToken mutation fires and previewComponent reflects the new colour

Given DESIGN_TOKENS_ENABLED is False on the backend
When DesignTokensEditor renders
Then a disabled placeholder is shown explaining design tokens are not enabled

# --- SettingsShell / ProfileSettings ---

Given SettingsShell renders with ['profile', 'security', 'notifications', 'billing']
When the user navigates between sections
Then each section loads its data independently — no cross-section state bleed

Given a user updates their profile and the save mutation succeeds
When the save completes
Then a success toast is shown and the profile data updates without a page reload

Given a user navigates to SecuritySettings
When the change password form is shown
Then the user is prompted to re-authenticate before the form is enabled

Given a user updates their notification preferences
When the preferences are saved
Then the syntek-notifications API is called and the UI reflects the new state

# --- Accessibility ---

Given TenantSettingsPanel renders
When a screen reader reads the form
Then all inputs have associated labels and error messages are announced via aria-live
```
