## User Story

As a developer, I want a CalDav and CardDav client module so that I can sync calendar events and
contacts with any CalDav/CardDav server — defaulting to the Radicale instance on
syntek-infrastructure but configurable for any compliant server — with per-tenant credential
isolation and retry on connection failure.

## Architecture

`syntek-caldav` is a generic CalDav/CardDav client. The default server is the Radicale instance on
syntek-infrastructure (NixOS/Hetzner), but `SYNTEK_CALDAV['SERVER_URL']` can point to any RFC 4791 /
RFC 6352 compliant server (Nextcloud, Baikal, Apple Calendar Server, Google Calendar via CalDav
endpoint, Fastmail, etc.). `syntek-scheduling-calendar` (US110) uses this module to push confirmed
appointments to CalDav. Consuming projects that need contact sync (CardDav) configure
`SYNTEK_CALDAV['CARDDAV_ENABLED'] = True`.

## Configuration

```python
SYNTEK_CALDAV = {
    # CalDav server — defaults to Radicale on syntek-infrastructure
    'SERVER_URL': env('CALDAV_SERVER_URL'),           # e.g. https://cal.example.com/
    'USERNAME': env('CALDAV_USERNAME'),
    'PASSWORD': env('CALDAV_PASSWORD'),               # Encrypted at rest via syntek-pyo3

    # Per-tenant calendars — resolved at runtime; key = tenant slug, value = calendar path
    'TENANT_CALENDAR_MAP': {},   # Optional override; default = auto-discover per tenant

    # CardDav (contact sync) — disabled by default
    'CARDDAV_ENABLED': False,
    'CARDDAV_ADDRESS_BOOK_PATH': env('CARDDAV_PATH', default=''),

    # Sync behaviour
    'SYNC_INTERVAL_MINUTES': 15,      # Celery Beat poll interval
    'RETRY_COUNT': 3,
    'RETRY_BACKOFF_SECONDS': [30, 120, 300],

    # iCal export — used by syntek-scheduling-calendar
    'ICAL_PRODUCT_ID': '-//Syntek//Scheduling//EN',
    'ICAL_TIMEZONE': 'Europe/London',
}
```

## Acceptance Criteria

```gherkin
Scenario: Fetch and upsert calendar events
Given SERVER_URL points to a Radicale instance
When sync_calendar(tenant) runs
Then events from the tenant's CalDav calendar are fetched and upserted into the local CalDavEvent model using UID as the idempotency key

Scenario: Push new event to CalDav server
Given a new CalDavEvent is created locally
When the outbound sync task runs
Then the event is pushed to the CalDav server as a VEVENT component

Scenario: Update event on CalDav server
Given a local event is updated
When the outbound sync task runs
Then the VEVENT on the server is updated (PUT with matching UID)

Scenario: Delete event from CalDav server
Given a local event is deleted
When the outbound sync task runs
Then the VEVENT is deleted from the server (DELETE request)

Scenario: Retry on server unreachable
Given the CalDav server is unreachable
When a sync is attempted
Then the failure is logged, the sync is retried per RETRY_BACKOFF_SECONDS, and no uncaught exception is raised

Scenario: CardDav contact sync
Given CARDDAV_ENABLED = True
When sync_contacts(tenant) runs
Then contacts from the configured address book are fetched and upserted into the local CardDavContact model

Scenario: Non-Radicale CalDav server compatibility
Given SERVER_URL points to a non-Radicale CalDav server
When sync runs
Then the module behaves identically — server implementation is abstracted by RFC 4791 compliance
```
