## User Story

As a developer, I want a scheduling and booking module so that I can define configurable resources,
set availability with hard or soft booking modes, accept atomic bookings, manage recurring events,
enforce cancellation policies, and allow any project — a restaurant, vet, church hall, or service
provider — to offer online booking without any business-type assumptions baked in.

## Architecture

`syntek-scheduling-core` is domain-agnostic. A "resource" may be a staff member, a room, a table,
equipment, or any bookable entity — defined by the consuming project. "Appointments" are the booked
instances. The module does not know or care whether the project is a restaurant or a vet surgery.
All business-type decisions (labels, booking modes, durations, policies) are driven by
`SYNTEK_SCHEDULING` configuration and per-resource settings.

Calendar export and sync is handled by `syntek-scheduling-calendar` (US110), which this module fires
domain events towards. Direct staff calendar push (e.g. Google Calendar API) is the responsibility
of the consuming project via `syntek-integrations-core` adapters — this module fires
`appointment.confirmed` and `appointment.cancelled` events on the bus, which consuming projects can
hook into.

## Configuration

```python
SYNTEK_SCHEDULING = {
    # Booking mode default (overridable per ResourceType)
    # 'hard' = exact start time (10:30, 11:00, etc.)
    # 'soft' = arrival window ("arrive between 10:00 and 11:00")
    'DEFAULT_BOOKING_MODE': 'hard',

    # Timezone — all datetimes stored UTC; displayed in this timezone unless user TZ is known
    'DEFAULT_TIMEZONE': 'Europe/London',

    # Availability windows
    'MIN_LEAD_TIME_MINUTES': 60,      # Minimum advance booking (0 = instant booking allowed)
    'MAX_ADVANCE_DAYS': 90,           # How far ahead bookings can be made

    # Slot defaults (overridable per ResourceType)
    'DEFAULT_SLOT_DURATION_MINUTES': 30,
    'DEFAULT_BUFFER_BEFORE_MINUTES': 0,
    'DEFAULT_BUFFER_AFTER_MINUTES': 0,
    'DEFAULT_ARRIVAL_WINDOW_MINUTES': 60,  # Used when booking_mode = 'soft'

    # Cancellation
    'CANCELLATION_WINDOW_HOURS': 24,   # Free cancellation within this window; 0 = no free window
    'CANCELLATION_FEE_ENABLED': False, # If True, cancellation outside window triggers a fee record

    # Waitlist
    'WAITLIST_ENABLED': True,
    'WAITLIST_OFFER_EXPIRY_MINUTES': 30,  # How long the waitlisted booker has to claim the slot

    # Calendar export — which providers to surface in the API response
    # Consuming project's frontend renders "Add to Calendar" buttons for each listed provider
    'CALENDAR_EXPORT_PROVIDERS': ['ical', 'caldav'],
    # Options: 'ical' | 'caldav' | 'google' | 'outlook' | 'apple'

    # Notifications
    'CONFIRMATION_NOTIFICATION': True,   # Send on booking confirmed
    'REMINDER_NOTIFICATION': True,       # Send reminder before appointment
    'REMINDER_HOURS_BEFORE': [24, 1],    # Hours before appointment to send reminders
    'CANCELLATION_NOTIFICATION': True,   # Notify on cancellation

    # Resource types the consuming project uses (informational — drives admin UI labels)
    'RESOURCE_TYPES': ['staff'],  # e.g. ['staff', 'room', 'table', 'equipment']
}
```

## Acceptance Criteria

```gherkin
Scenario: Hard booking mode — exact time slots
Given a ResourceType with booking_mode = 'hard' and slot_duration = 30
When availableSlots(resourceId, dateRange) is queried
Then only unbooked 30-minute slots within the resource's availability rules are returned — no past, no already-booked, no within-lead-time slots

Scenario: Soft booking mode — arrival windows
Given a ResourceType with booking_mode = 'soft' and arrival_window = 60
When an available window is booked
Then an Appointment is created with window_start and window_end rather than an exact start time, and the window is blocked for other bookings

Scenario: Atomic slot reservation — no double-booking
Given two simultaneous booking requests arrive for the same slot
When both are processed concurrently
Then exactly one succeeds and the other receives a SlotUnavailableError — no double-booking (DB-level SELECT FOR UPDATE)

Scenario: Waitlist creation and notification
Given WAITLIST_ENABLED = True and a fully-booked slot
When a booker joins the waitlist
Then a WaitlistEntry is created and the booker is notified via syntek-notifications when the slot becomes available

Scenario: Waitlist offer expiry
Given a waitlist offer is made and WAITLIST_OFFER_EXPIRY_MINUTES = 30
When 30 minutes pass without the waitlisted booker claiming the slot
Then the offer expires and the next waitlist entry is offered

Scenario: Free cancellation within window
Given CANCELLATION_WINDOW_HOURS = 24 and a cancellation is made within 24 hours of the appointment
When the cancellation is processed
Then the appointment status transitions to cancelled, the slot is released, waitlist is notified, and appointment.cancelled is fired on the bus

Scenario: Cancellation fee outside window
Given a cancellation is made outside the free window and CANCELLATION_FEE_ENABLED = True
When the cancellation is processed
Then a CancellationFeeRecord is created for the consuming project to process via its payment module

Scenario: Recurring event expansion
Given a recurring RecurrenceRule is defined
When individual occurrence dates are computed
Then RRULE expansion generates correct dates respecting timezone and any individual occurrence exceptions

Scenario: Appointment reminders
Given REMINDER_HOURS_BEFORE = [24, 1] and an appointment is confirmed
When each reminder threshold is reached
Then a reminder notification is dispatched via syntek-notifications
```

## Dependencies

- US010 — `syntek-tenancy` (per-tenant resource and availability scoping)
- US015 — `syntek-tasks` (async reminder Celery tasks)
- US019 — `syntek-notifications` (confirmation, reminder, cancellation, waitlist notifications)
- US090 — `syntek-bus` (fires `appointment.booked`, `appointment.confirmed`,
  `appointment.cancelled`, `appointment.rescheduled`)
- US110 — `syntek-scheduling-calendar` (calendar export URL generation, CalDav sync — fires events
  consumed by US110)
