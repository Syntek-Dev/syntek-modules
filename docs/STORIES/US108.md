## User Story

As a developer, I want an automation platform bridge so that I can send Syntek domain events to n8n
or Zapier workflows and receive trigger webhooks from those platforms — enabling workflow automation
without writing custom integration code for every third-party service.

## Architecture

**n8n is the recommended platform** for the Syntek ecosystem. n8n is open-source, self-hostable on
the same Hetzner/NixOS infrastructure as the rest of the stack, and has a broad connector library
including accounting services (Xero, QuickBooks, FreeAgent, Sage, Harvest) that lack clean REST
APIs. It aligns with Syntek's self-hosting and FOSS philosophy.

**Zapier is fully supported** as an alternative for consuming projects whose teams already use
Zapier. Both platforms operate via webhooks — the module behaves identically for either. The
`PLATFORM` setting determines which signature verification algorithm is applied to inbound webhooks.

For accounting services without a direct API (e.g. Sage 50 desktop, older FreeAgent tiers), the
recommended workflow is: Syntek → n8n webhook trigger → n8n workflow → accounting service (via n8n's
native node). This avoids building fragile screen-scraping adapters inside Syntek.

## Configuration

```python
SYNTEK_INTEGRATIONS_AUTOMATION = {
    # 'n8n' or 'zapier' — determines signature verification algorithm
    # n8n uses HMAC-SHA256; Zapier uses their proprietary header verification
    'PLATFORM': 'n8n',

    # Inbound webhook — n8n/Zapier sends events TO Syntek (trigger webhooks)
    'INBOUND_WEBHOOK_SECRET': env('AUTOMATION_WEBHOOK_SECRET'),
    'INBOUND_WEBHOOK_PATH': '/webhooks/automation/',

    # Outbound webhook URLs — Syntek sends domain events TO n8n/Zapier
    # Map of Syntek event slug → webhook URL
    # Event slugs are dot-separated: 'payment.succeeded', 'donation.completed', etc.
    'OUTBOUND_HOOKS': {
        # 'payment.succeeded': 'https://n8n.example.com/webhook/abc123',
        # 'invoice.paid': 'https://n8n.example.com/webhook/def456',
    },

    # Retry failed outbound deliveries
    'OUTBOUND_RETRY_COUNT': 3,
    'OUTBOUND_RETRY_BACKOFF_SECONDS': [10, 60, 300],

    # Optional HMAC secret for outbound requests (n8n can verify these)
    'OUTBOUND_HMAC_SECRET': env('AUTOMATION_OUTBOUND_SECRET', default=''),
}
```

## Acceptance Criteria

```gherkin
Given PLATFORM = 'n8n' and a 'payment.succeeded' event is fired via syntek-bus
When the outbound dispatcher runs
Then an HTTP POST is made to the configured OUTBOUND_HOOKS['payment.succeeded'] URL with the event payload and HMAC-SHA256 signature header

Given OUTBOUND_HOOKS has no entry for a fired event
When the dispatcher checks the mapping
Then no outbound request is made and no error is raised

Given an inbound webhook POST arrives from n8n at INBOUND_WEBHOOK_PATH
When HMAC-SHA256 signature is verified against INBOUND_WEBHOOK_SECRET
Then the payload is accepted and a AutomationWebhookReceived event is fired via syntek-bus for consuming apps to handle

Given PLATFORM = 'zapier' and an inbound webhook arrives
When Zapier's signature verification runs
Then the same inbound flow is triggered with Zapier's verification algorithm

Given an outbound delivery fails
When the retry schedule runs
Then delivery is retried at the configured backoff intervals and a AutomationDeliveryFailed record is created after all retries are exhausted

Given OUTBOUND_HMAC_SECRET is set
When an outbound request is sent
Then a X-Syntek-Signature header is included so the receiving n8n/Zapier workflow can verify the request origin
```
