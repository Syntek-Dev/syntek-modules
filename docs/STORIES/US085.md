# US085 — `@syntek/ui-flags`: Feature Flag UI Package

**Last Updated**: 10/03/2026 **Version**: 0.1.0 **Status**: To Do **Epic**: Epic 7 — Web Frontend
Features

## User Story

As a developer, I want a React package for feature flags so that any component can gate behaviour
behind a flag with a `useFlag` hook, `<FlagGate>` renders children conditionally, and tenant admins
can manage all flags from a `FlagsAdmin` panel — all wired to the GraphQL API from `syntek-flags`
(US017) without boilerplate.

## What It Exposes

### `@syntek/ui-flags/hooks` — Flag evaluation

```typescript
import { useFlag, useFlags } from "@syntek/ui-flags/hooks";

// Single flag — returns boolean (evaluated for current user + tenant)
const isEnabled = useFlag("new_checkout_flow");

// Options:
const isEnabled = useFlag("new_checkout_flow", {
  defaultValue: false, // boolean — returned while loading (default false)
  suspense: false, // boolean — enable React Suspense (default false)
  refetchInterval: 0, // number (ms) — 0 = no polling; set to re-evaluate on interval
});

// Multiple flags in one query — avoids N+1 GraphQL calls
const flags = useFlags(["new_checkout_flow", "beta_reporting", "maintenance_mode"]);
// Returns: Record<string, boolean>
```

### `@syntek/ui-flags/components` — Declarative flag gating and admin UI

```typescript
import { FlagGate, FlagsAdmin, FlagBadge } from '@syntek/ui-flags/components';

// Render children only when flag is enabled
<FlagGate flag="new_checkout_flow">
    <NewCheckout />
</FlagGate>

// Render fallback when flag is disabled
<FlagGate flag="new_checkout_flow" fallback={<LegacyCheckout />}>
    <NewCheckout />
</FlagGate>

// Render while loading (avoids layout shift)
<FlagGate flag="new_checkout_flow" loading={<Skeleton />}>
    <NewCheckout />
</FlagGate>

// Invert — render children when flag is DISABLED
<FlagGate flag="maintenance_mode" invert>
    <AppContent />
</FlagGate>

// Admin panel — lists all flags for the current tenant, allows enable/disable and
// rollout percentage editing. Requires manage_flags permission.
<FlagsAdmin
    onSaveSuccess?: () => void
    onSaveError?: (error: Error) => void
    readOnly?: boolean
/>

// Small badge — shows flag state visually (useful in admin/debug UIs)
<FlagBadge flag="new_checkout_flow" />
// Renders: "new_checkout_flow ✓ enabled" | "new_checkout_flow ✗ disabled"
// Only render this in admin/debug contexts — do not expose flag names to end users
```

**Key rules:**

- `useFlag` calls the `flagStatus(name)` GraphQL query. The evaluated result is for the current user
  and tenant — the same evaluation hierarchy as the backend `is_enabled()` function.
- `useFlags` batches all named flags into a single `featureFlags` GraphQL query filtered by name —
  it does not fire N separate queries.
- `FlagGate` never throws when a flag is unknown — it renders the fallback (or null) and logs a
  console warning in development. This mirrors the backend `is_enabled` contract (unknown = false).
- `FlagsAdmin` requires `manage_flags` permission. The GraphQL mutations will return
  `PermissionDenied` for unpermissioned users — the hook surfaces this as an error, not a crash.
- `FlagBadge` is intentionally a debug/admin component — it exposes the flag name visually. Do not
  use it in customer-facing UIs.
- All components accept `className` prop and use `@syntek/ui` design tokens.

## Acceptance Criteria

```gherkin
# --- useFlag hook ---

Given 'new_checkout_flow' is enabled for the current user
When useFlag('new_checkout_flow') is called
Then true is returned

Given 'new_checkout_flow' is disabled for the current user
When useFlag('new_checkout_flow') is called
Then false is returned

Given the flagStatus query is loading
When useFlag is called with defaultValue: false
Then false is returned during loading — no undefined flash

Given 'unknown_flag' does not exist in the registry
When useFlag('unknown_flag') is called
Then false is returned (no error thrown; console warning in dev)

# --- useFlags hook ---

Given three flags are passed to useFlags
When the hook resolves
Then a single GraphQL query is fired (not three separate queries)

Given flags: ['flag_a', 'flag_b'] where flag_a is enabled and flag_b is disabled
When useFlags resolves
Then { flag_a: true, flag_b: false } is returned

# --- FlagGate component ---

Given flag 'new_checkout_flow' is enabled
When <FlagGate flag="new_checkout_flow"><Child /></FlagGate> renders
Then <Child /> is rendered

Given flag 'new_checkout_flow' is disabled and no fallback is provided
When FlagGate renders
Then nothing is rendered (null)

Given flag 'new_checkout_flow' is disabled and fallback={<Legacy />} is provided
When FlagGate renders
Then <Legacy /> is rendered

Given loading={<Skeleton />} is provided and the flag query is in flight
When FlagGate renders
Then <Skeleton /> is rendered — no layout shift

Given invert={true} and flag 'maintenance_mode' is enabled
When FlagGate renders
Then children are NOT rendered (inverted behaviour)

Given invert={true} and flag 'maintenance_mode' is disabled
When FlagGate renders
Then children ARE rendered

# --- FlagsAdmin component ---

Given FlagsAdmin renders and the user has manage_flags permission
When the component mounts
Then all flags for the current tenant are listed with name, description, enabled state,
and rollout percentage

Given a tenant admin toggles 'new_checkout_flow' from disabled to enabled
When save fires
Then the updateFlag mutation runs and the panel reflects the new state

Given rolloutPercentage is edited from 20 to 50
When save fires
Then the updateFlag mutation runs with rolloutPercentage: 50

Given readOnly is true
When FlagsAdmin renders
Then all controls are disabled — no mutations can be triggered

Given the user does not have manage_flags permission
When updateFlag is called via FlagsAdmin
Then a PermissionDenied error is surfaced in the UI — no crash

Given resetFlag is triggered for a flag with a tenant override
When the mutation resolves
Then the flag returns to its static definition default and isOverridden is false

# --- FlagBadge component ---

Given 'new_checkout_flow' is enabled
When FlagBadge renders
Then "new_checkout_flow ✓ enabled" is shown

Given 'new_checkout_flow' is disabled
When FlagBadge renders
Then "new_checkout_flow ✗ disabled" is shown

# --- Accessibility ---

Given FlagsAdmin renders
When a screen reader reads the panel
Then all toggles have associated labels and state changes are announced via aria-live
```
