## User Story

As a developer, I want a security validator module so that all Syntek applications enforce
consistent, configurable input validation, XSS sanitisation, and CSRF protection at the GraphQL
layer — with field validation schemas, a composable `@validateInput` directive, and automatic
mutation sanitisation — all controlled from `settings.py` without touching module source.

## Configuration Contract

All behaviour is controlled via `SYNTEK_SECURITY_VALIDATOR` in `settings.py`. Validated at startup
in `AppConfig.ready()`.

```python
SYNTEK_SECURITY_VALIDATOR = {
    # --- Schema registry ---
    'SCHEMAS': {},                          # dict[str, str | callable] — custom schemas merged with built-ins
                                             # str value = regex pattern; callable = validator function(value) -> bool
                                             # Built-in names cannot be overridden — startup raises ImproperlyConfigured

    # --- XSS sanitisation ---
    'XSS_ENABLED': True,
    'XSS_ALLOWED_TAGS': [],                 # list[str] — empty = strip all HTML (correct for JSON/GraphQL APIs)
    'XSS_ALLOWED_ATTRIBUTES': {},           # dict[str, list[str]] — tag -> list of allowed attrs
    'XSS_STRIP_COMMENTS': True,

    # --- CSRF ---
    'CSRF_GRAPHQL_EXEMPT_OPERATIONS': [],   # list[str] — GraphQL operation names exempt from CSRF enforcement
    'CSRF_COOKIE_HTTPONLY': False,          # bool — must be False for SPA CSRF token injection
    'CSRF_COOKIE_SAMESITE': 'Lax',         # 'Strict' | 'Lax' | 'None'

    # --- Validation error behaviour ---
    'VALIDATION_ERROR_CODE': 'VALIDATION_ERROR',
    'VALIDATION_LOG_FAILURES': True,
    'VALIDATION_BLOCK_ON_FAILURE': True,    # False = pass through with warning (security risk — documented)

    # --- Content sanitisation ---
    'SANITISE_GRAPHQL_INPUTS': True,        # bool — opt-OUT default; auto-sanitises all string mutation inputs
                                             # Set False only if you have benchmarked overhead and have explicit
                                             # per-field sanitisation in place.

    # --- Multi-schema validation ---
    'ALLOW_MULTI_SCHEMA_VALIDATION': True,  # bool — allow @validateInput(schema=[...]) on a single field
                                             # Set False to enforce one schema per field (strict mode)

    # --- File upload validation ---
    'FILE_UPLOAD_ENABLED': True,
    'FILE_UPLOAD_ALLOWED_MIME_TYPES': [
        'image/jpeg', 'image/png', 'image/webp', 'application/pdf',
    ],
    'FILE_UPLOAD_MAX_SIZE_BYTES': 10_485_760,   # 10 MB
    'FILE_UPLOAD_VALIDATE_MAGIC_BYTES': True,    # requires python-magic / libmagic
}
```

**Key rules:**

- `SCHEMAS` merges with built-in registry. Consumer names cannot collide with built-ins — startup
  raises `ImproperlyConfigured`.
- `XSS_ALLOWED_TAGS: []` is the correct default for JSON/GraphQL APIs. Rich text fields must
  explicitly permit tags.
- `SANITISE_GRAPHQL_INPUTS: True` is a defence-in-depth layer and does not replace explicit
  `@validateInput` directives.
- `FILE_UPLOAD_VALIDATE_MAGIC_BYTES: True` requires `python-magic`. Missing package raises
  `ImproperlyConfigured` at startup.
- `VALIDATION_BLOCK_ON_FAILURE: False` in production is a security risk and must be documented as
  such.
- No database migrations — module is entirely stateless.

## Built-in Validation Schemas

Thirteen built-in schema names are available for use with `@validateInput`:

```text
'email'              — RFC 5321 email format
'phone_e164'         — E.164 international phone format (+1234567890)
'phone_gb'           — GB phone format (including landline variants)
'url'                — HTTP/HTTPS URL with valid hostname
'url_https_only'     — HTTPS URL only
'slug'               — URL-safe slug (alphanumeric, underscore, hyphen)
'uuid'               — RFC 4122 UUID (all versions)
'ipv4'               — IPv4 address
'ipv6'               — IPv6 address
'postcode_gb'        — UK postcode (format A9A 9AA)
'hex_colour'         — Hex colour code (#RRGGBB or #RRGGBBAA)
'iso_date'           — ISO 8601 date (YYYY-MM-DD)
'iso_datetime'       — ISO 8601 datetime with timezone
```

## @validateInput Directive Usage

```python
# Single schema (always supported)
email: Annotated[str, validateInput(schema="email")]

# Multi-schema (requires ALLOW_MULTI_SCHEMA_VALIDATION: True, which is the default)
# All schemas must pass; first failure short-circuits with schema name in error message
website: Annotated[str, validateInput(schema=["url_https_only", "custom:myapp_url"])]

# Custom schema via regex (in settings.py SCHEMAS dict)
SYNTEK_SECURITY_VALIDATOR['SCHEMAS']['custom:myapp_url'] = r'^https://myapp\.com/.*'

# Custom schema via callable
def validate_internal_url(value: str) -> bool:
    return value.startswith('https://internal.myapp.com/')

SYNTEK_SECURITY_VALIDATOR['SCHEMAS']['custom:internal_url'] = validate_internal_url
```

## Acceptance Criteria

```gherkin
# --- Settings validation ---

Given `SCHEMAS` contains a name that collides with a built-in (e.g. 'email')
When `AppConfig.ready()` runs
Then `ImproperlyConfigured` is raised — consumer schemas cannot override built-ins

Given `FILE_UPLOAD_VALIDATE_MAGIC_BYTES` is True and python-magic is not installed
When `AppConfig.ready()` runs
Then `ImproperlyConfigured` is raised — magic byte validation requires python-magic

Given `ALLOW_MULTI_SCHEMA_VALIDATION` is False and a field declares schema=["email", "url"]
When `AppConfig.ready()` runs
Then `ImproperlyConfigured` is raised — multi-schema validation is disabled

# --- Built-in schema validation ---

Given a valid email 'user@example.com' is validated with schema 'email'
When validation runs
Then it passes

Given an invalid email 'user@invalid' is validated with schema 'email'
When validation runs
Then it fails with VALIDATION_ERROR code

Given a valid E.164 phone '+1234567890' is validated with schema 'phone_e164'
When validation runs
Then it passes

Given a valid GB phone '020 1234 5678' is validated with schema 'phone_gb'
When validation runs
Then it passes

Given a valid HTTPS URL is validated with schema 'url_https_only'
When validation runs
Then it passes

Given an HTTP URL (not HTTPS) is validated with schema 'url_https_only'
When validation runs
Then it fails

Given a valid UK postcode 'SW1A 1AA' is validated with schema 'postcode_gb'
When validation runs
Then it passes

# --- Custom regex schema ---

Given a custom schema defined as regex r'^ABC-\d{3}$' in SCHEMAS
When a value 'ABC-123' is validated against it
Then it passes

Given a custom schema as regex and a value 'XYZ-123' is validated
When validation runs
Then it fails

# --- Custom callable schema ---

Given a custom schema defined as callable `lambda v: len(v) > 5`
When a value with length 10 is validated
Then it passes

Given a custom schema as callable returning False
When validation runs
Then it fails with VALIDATION_ERROR code

# --- @validateInput single schema ---

Given a GraphQL mutation with email: Annotated[str, validateInput(schema="email")]
When a valid email is submitted
Then the resolver receives the value and executes normally

Given a GraphQL mutation with @validateInput(schema="email")
When an invalid email is submitted
Then a ValidationError is raised with VALIDATION_ERROR_CODE before the resolver runs

# --- @validateInput multi-schema ---

Given `ALLOW_MULTI_SCHEMA_VALIDATION` is True and a field has schema=["email", "url"]
When both schemas pass
Then the resolver is reached

Given schema=["email", "url"] and the first schema fails
When validation runs
Then ValidationError is raised with the first failing schema name in the message

Given schema=["email", "url"] and the first passes but second fails
When validation runs
Then ValidationError is raised with the second (failing) schema name in the message

# --- GraphQL input sanitisation ---

Given `SANITISE_GRAPHQL_INPUTS` is True
When a GraphQL mutation receives string input '<script>alert("xss")</script>'
Then the value is sanitised before the resolver is called

Given `SANITISE_GRAPHQL_INPUTS` is False
When a mutation receives malicious string input
Then the resolver receives the raw unsanitised value

# --- XSS sanitisation ---

Given `XSS_ALLOWED_TAGS` is []
When a string '<p>hello</p>' is sanitised
Then all HTML tags are stripped; output is 'hello'

Given `XSS_ALLOWED_TAGS` is ['p', 'strong']
When '<p><strong>hello</strong></p>' is sanitised
Then the tags are preserved

Given `XSS_ALLOWED_TAGS` is ['p'] and 'data-custom' is not in XSS_ALLOWED_ATTRIBUTES
When '<p data-custom="value">text</p>' is sanitised
Then the attribute is removed; output is '<p>text</p>'

Given `XSS_ALLOWED_TAGS` is ['p'] and 'class' is in XSS_ALLOWED_ATTRIBUTES['p']
When '<p class="danger">text</p>' is sanitised
Then the class attribute is preserved

Given `XSS_STRIP_COMMENTS` is True
When '<p>text<!-- comment --></p>' is sanitised
Then the HTML comment is removed

Given `XSS_STRIP_COMMENTS` is False
When '<p>text<!-- comment --></p>' is sanitised
Then the comment is preserved (uncommon; for legacy HTML migration)

# --- Validation logging ---

Given `VALIDATION_LOG_FAILURES` is True
When a field validation fails
Then the failure is logged via syntek-logging with context (schema, value, resolver)

Given `VALIDATION_LOG_FAILURES` is False
When a field validation fails
Then no log entry is created

# --- Validation error behaviour ---

Given `VALIDATION_BLOCK_ON_FAILURE` is True
When validation fails
Then a ValidationError is raised; the mutation is not executed

Given `VALIDATION_BLOCK_ON_FAILURE` is False
When validation fails
Then the request passes through with a warning logged; the resolver receives the raw input
(security risk — documented)

# --- File upload validation ---

Given `FILE_UPLOAD_ENABLED` is True and MIME type 'image/jpeg' is in ALLOWED_MIME_TYPES
When an image/jpeg file is uploaded
Then it is accepted

Given MIME type 'application/exe' is not in ALLOWED_MIME_TYPES
When an exe file is uploaded
Then a ValidationError is raised before storage

Given `FILE_UPLOAD_MAX_SIZE_BYTES` is 10_485_760 (10 MB)
When a file of 11 MB is uploaded
Then a ValidationError is raised for size limit exceeded

Given `FILE_UPLOAD_VALIDATE_MAGIC_BYTES` is True
When a file with .jpg extension but actual PNG magic bytes is uploaded
When magic bytes do not match Content-Type
Then a ValidationError is raised for magic byte mismatch

Given `FILE_UPLOAD_VALIDATE_MAGIC_BYTES` is False
When a file with mismatched magic bytes is uploaded
Then magic byte validation is skipped; only MIME allowlist and size are checked

# --- CSRF protection ---

Given `CSRF_GRAPHQL_EXEMPT_OPERATIONS` is ['GetPublicData']
When a GET-style query named 'GetPublicData' is executed without a CSRF token
Then it is allowed

Given a GraphQL mutation not in CSRF_GRAPHQL_EXEMPT_OPERATIONS
When the request lacks a valid CSRF token
Then a 403 Forbidden is returned

Given a valid CSRF token is present in the X-CSRFToken header
When a mutation is executed
Then the token is validated and the mutation proceeds

Given `CSRF_COOKIE_HTTPONLY` is False
When the CSRF token cookie is set
Then the cookie is readable by JavaScript for SPA token injection

Given `CSRF_COOKIE_SAMESITE` is 'Lax'
When a cross-site GET request includes the CSRF cookie
Then the cookie is sent; SameSite=Lax allows top-level navigation

Given `CSRF_COOKIE_SAMESITE` is 'Strict'
When a cross-site GET request includes the CSRF cookie
Then the cookie is not sent (SameSite=Strict blocks all cross-site requests)
```
