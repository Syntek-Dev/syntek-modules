## User Story

As a developer, I want a calendar export and sync module so that I can generate iCal files, push
confirmed appointments to a CalDav server, and produce configurable "Add to Calendar" URLs for any
provider the consuming project chooses to surface — without hardcoding any specific calendar
platform.

## Architecture

`syntek-scheduling-calendar` listens to `appointment.confirmed` and `appointment.cancelled` events
on the bus and handles all calendar-related output. It generates iCal VEVENT data, pushes to CalDav
via `syntek-caldav` (the default and recommended sync path), and produces "Add to Calendar" URLs for
Google Calendar, Outlook, and Apple Calendar (these are deep-link URL schemes — no API credentials
needed). The consuming project's frontend renders whichever provider buttons are listed in
`SYNTEK_SCHEDULING['CALENDAR_EXPORT_PROVIDERS']`. No calendar provider is bundled or assumed.

## Configuration

```python
SYNTEK_SCHEDULING_CALENDAR = {
    # CalDav push — pushes confirmed appointments to CalDav server (via syntek-caldav)
    'CALDAV_PUSH_ENABLED': True,
    'CALDAV_PUSH_CALENDAR_PATH': env('SCHEDULING_CALENDAR_PATH', default=''),

    # iCal download — serves .ics files for manual import
    'ICAL_DOWNLOAD_ENABLED': True,

    # "Add to Calendar" URL generation
    # These produce deep-link URLs — no API credentials required
    # Consuming project's frontend renders buttons for each enabled provider
    'ADD_TO_CALENDAR_PROVIDERS': ['ical', 'google', 'outlook', 'apple'],

    # CalDav subscription — a persistent subscription URL the user can add to any CalDav client
    'CALDAV_SUBSCRIPTION_ENABLED': True,
    'CALDAV_SUBSCRIPTION_BASE_URL': env('CALDAV_SUBSCRIPTION_BASE_URL', default=''),

    # iCal metadata
    'ICAL_ORGANISER_NAME': env('ICAL_ORGANISER_NAME', default=''),
    'ICAL_ORGANISER_EMAIL': env('ICAL_ORGANISER_EMAIL', default=''),
}
```

## Acceptance Criteria

```gherkin
Scenario: Push confirmed appointment to CalDav
Given an appointment transitions to confirmed
When the appointment.confirmed bus event is received
Then a VEVENT is generated and pushed to the CalDav server via syntek-caldav (if CALDAV_PUSH_ENABLED = True)

Scenario: Delete cancelled appointment from CalDav
Given an appointment is cancelled
When the appointment.cancelled bus event is received
Then the corresponding VEVENT is deleted from the CalDav server

Scenario: Generate add-to-calendar URLs with provider filtering
Given ADD_TO_CALENDAR_PROVIDERS = ['ical', 'google', 'outlook', 'apple']
When calendarExportLinks(appointmentRef) is queried
Then four URLs are returned — one iCal download, one Google Calendar add link, one Outlook add link, one Apple Calendar link — all pre-filled with the appointment details

Scenario: Provider-agnostic calendar export
Given ADD_TO_CALENDAR_PROVIDERS = ['ical', 'caldav']
When calendarExportLinks is queried
Then only iCal download and CalDav subscription URLs are returned — no Google or Outlook links

Scenario: CalDav subscription URL generation
Given CALDAV_SUBSCRIPTION_ENABLED = True
When a user requests their personal booking calendar URL
Then a unique per-user subscription URL is returned that they can add to any CalDav-compatible app (Apple Calendar, Thunderbird, Google Calendar's "Other calendars" URL import, etc.)

Scenario: iCal RFC 5545 compliance
Given an iCal download is requested for a confirmed appointment
When the .ics endpoint is called
Then a valid RFC 5545 VCALENDAR file is returned with correct VEVENT, DTSTART, DTEND, SUMMARY, DESCRIPTION, and UID
```
