## User Story

As a developer, I want a pre-configured Celery task queue so that any Syntek application can
dispatch background work to priority lanes with automatic retries, dead-letter routing, cron-style
scheduling, and chunked fan-out for high-volume operations — with per-task rate limits, incremental
progress reporting, and a GraphQL API for task status — all controlled from `settings.py` without
touching module source.

**Must Have**

## Design Decisions

### Cron jobs vs Celery

**Celery Beat with `django-celery-beat` is the only scheduler needed.** It provides full crontab
syntax, interval scheduling, and dynamic schedule management — all stored in the Django database
(cluster-safe, restart-safe). There is no benefit to running a separate cron system alongside it.
Schedules can be defined in `SYNTEK_TASKS['BEAT_SCHEDULE']` (static) or created programmatically via
the `PeriodicTask` model (dynamic, e.g. tenant-specific recurring jobs).

In a multi-worker deployment, run **one Beat process** and **N worker processes**. The
`DatabaseScheduler` from `django-celery-beat` uses database-level locking — multiple Beat instances
would double-fire schedules.

### Priority queues

Four lanes, each independently sized. Workers are configured per lane so a flood of `low`-priority
bulk tasks cannot starve `critical` or `high` tasks.

| Lane       | Intended use                                                       |
| ---------- | ------------------------------------------------------------------ |
| `critical` | Security alerts, auth token invalidation, payment webhook dispatch |
| `high`     | Individual notification sends, user-facing operations              |
| `default`  | General application tasks                                          |
| `low`      | Bulk/chunked operations, report generation, large email batches    |

### Chunking and fan-out

High-volume operations (email, SMS, push, in-app notifications) must never be dispatched as a single
monolithic task. The fan-out pattern splits work into chunks and dispatches each chunk as a separate
Celery task on the `low` queue. This:

- Keeps individual task execution times bounded and predictable
- Allows Celery to distribute chunks across all available workers
- Enables partial failure and retry without re-processing the whole batch
- Prevents a single bulk job from saturating the queue

The `chunk_task` decorator (and `ChunkableMixin` on `SyntekTask`) implements this automatically.
Consumers configure `DEFAULT_CHUNK_SIZE` and per-type overrides in `SYNTEK_TASKS['CHUNK_SIZES']`.

### Workload balancing

Macro balancing: priority lanes ensure critical work is always served first. Micro balancing:
`TASK_RATE_LIMITS` (per-task Celery rate limit strings) prevent any single task type from exhausting
downstream resources (SMTP rate limits, SMS API quotas, DB write throughput). Consumer must set rate
limits that match their provider's limits.

### GraphQL API for task status

Any task that a user initiates from the frontend (bulk import, report generation, bulk email) must
be trackable. US015 provides a `TaskResult` model and a `taskStatus(taskId: ID!)` GraphQL query. The
frontend polls this query until the task reaches `SUCCESS` or `FAILURE`. A companion
`@syntek/ui-tasks` frontend package (US084) provides the `useTaskStatus` hook and UI components
built on this query.

## Configuration Contract

All behaviour is controlled via `SYNTEK_TASKS` in `settings.py`. Validated at startup in
`AppConfig.ready()`.

```python
SYNTEK_TASKS = {

    # --- Broker and result backend ---
    'BROKER_URL': env('CELERY_BROKER_URL'),         # str — Redis/Valkey URL (required)
    'RESULT_BACKEND': env('CELERY_RESULT_BACKEND'), # str — Redis/Valkey URL; can be same as broker
    'RESULT_EXPIRES': 3600,                          # int (seconds) — Celery result TTL in backend

    # --- Priority queues ---
    # Each queue can have an independent worker concurrency setting.
    # Concurrency = number of parallel task slots on that queue's workers.
    'QUEUES': {
        'critical': {'concurrency': 4},
        'high':     {'concurrency': 8},
        'default':  {'concurrency': 8},
        'low':      {'concurrency': 4},
    },
    'DEFAULT_QUEUE': 'default',     # str — queue used when no queue is specified on a task

    # --- Retry policy (defaults — overridable per task via SyntekTask class attributes) ---
    'MAX_RETRIES': 3,               # int — maximum number of automatic retries before DLQ
    'RETRY_BACKOFF_BASE': 60,       # int (seconds) — base delay; doubles each retry (60s, 120s, 240s)
    'RETRY_JITTER': True,           # bool — add ±20% randomness to backoff to avoid thundering herd

    # --- Dead-letter queue ---
    'DLQ_ENABLED': True,            # bool — route exhausted tasks to DLQ instead of discarding
    'DLQ_QUEUE': 'dead-letter',     # str — queue name for dead-letter tasks
    'DLQ_RETAIN_DAYS': 30,          # int — auto-delete DLQ entries older than this

    # --- Chunking (fan-out for bulk operations) ---
    'DEFAULT_CHUNK_SIZE': 100,      # int — default number of items per chunk task
    'CHUNK_SIZES': {                 # dict[str, int] — per-type overrides
        'email':  50,                # email tasks: 50 per chunk (SMTP/SES rate-limit aware)
        'sms':    20,                # SMS: smaller chunks, stricter provider limits
        'push':   500,               # push notifications: FCM/APNs accept large batches
        'in_app': 200,               # in-app: DB writes, can go larger
    },

    # --- Rate limits (per-task Celery rate limit format: 'N/s', 'N/m', 'N/h') ---
    # Applied per worker process. Set to match your downstream provider's limits.
    'TASK_RATE_LIMITS': {},          # dict[str, str] — task dotted name → Celery rate limit string
                                      # Example:
                                      # 'syntek_notifications.tasks.send_email_chunk': '100/m'

    # --- Scheduled tasks (Celery Beat) ---
    # Uses django-celery-beat's DatabaseScheduler — schedules stored in Django DB.
    # BEAT_SCHEDULE entries are merged with any @periodic_task decorator schedules.
    'BEAT_SCHEDULE': {},             # dict — same format as Celery BEAT_SCHEDULE
                                      # Example:
                                      # 'cleanup-expired-sessions': {
                                      #     'task': 'syntek_auth.tasks.cleanup_sessions',
                                      #     'schedule': crontab(hour=3, minute=0),
                                      # }

    # --- Timeouts ---
    'TASK_SOFT_TIME_LIMIT': 300,    # int (seconds) — raises SoftTimeLimitExceeded (catchable)
    'TASK_TIME_LIMIT': 360,         # int (seconds) — hard kill after this regardless of state

    # --- Task serialisation ---
    'TASK_SERIALIZER': 'json',       # str — 'json' only; pickle is banned for security reasons
    'RESULT_SERIALIZER': 'json',

    # --- TaskResult model (for GraphQL task status API) ---
    'STORE_RESULTS': True,           # bool — persist TaskResult model for status queries
    'STORE_RESULTS_TTL_DAYS': 7,     # int — auto-delete TaskResult records older than this
    'STORE_RESULTS_PROGRESS': True,  # bool — allow tasks to report incremental progress (0–100)

    # --- Monitoring ---
    'PROMETHEUS_METRICS_ENABLED': True,   # bool — expose /metrics queue depth + task counters
    'TASK_TRACK_STARTED': True,           # bool — emit STARTED state (needed for progress tracking)
}
```

**Key rules:**

- `BROKER_URL` and `RESULT_BACKEND` are required and must come from environment variables.
- `QUEUES` dict keys are the canonical queue names. Worker processes must be started with
  `-Q <queue-name>` to serve each lane. Deployment documentation must show the recommended worker
  command per lane.
- `TASK_SERIALIZER: 'json'` is enforced — `pickle` is explicitly blocked at startup because it
  allows arbitrary code execution via crafted task messages. Attempting to configure `pickle` raises
  `ImproperlyConfigured`.
- `CHUNK_SIZES` overrides `DEFAULT_CHUNK_SIZE` for the named type. Types are convention strings —
  the task author is responsible for naming their task type consistently.
- `BEAT_SCHEDULE` entries are merged with schedules registered via `@periodic_task`. On conflict
  (same task name), `BEAT_SCHEDULE` wins.
- `STORE_RESULTS: True` creates a `TaskResult` table (one migration). The GraphQL `taskStatus` query
  reads from this table, not from the Celery result backend, so results survive broker restarts.
- `DLQ_RETAIN_DAYS` is enforced by a Beat-scheduled cleanup task registered automatically when
  `DLQ_ENABLED: True`.

## Task API

### `SyntekTask` base class

All module tasks extend `SyntekTask`. Consumer tasks should also extend it for the retry, logging,
and DLQ behaviour.

```python
from syntek_tasks import SyntekTask

class SendWelcomeEmail(SyntekTask):
    name = 'myapp.send_welcome_email'
    queue = 'high'
    max_retries = 3
    retry_backoff_base = 60
    retry_jitter = True
    task_type = 'email'     # matches CHUNK_SIZES key for auto chunk sizing

    def run(self, user_id: int) -> None:
        ...
```

### `@syntek_task` decorator

Shorthand for registering a function as a `SyntekTask`:

```python
from syntek_tasks import syntek_task

@syntek_task(queue='high', max_retries=3, task_type='email')
def send_welcome_email(user_id: int) -> None:
    ...
```

### Chunked fan-out

The `dispatch_chunked` helper splits an iterable of items and dispatches chunk tasks automatically:

```python
from syntek_tasks import dispatch_chunked

# Dispatches send_email_chunk in batches of CHUNK_SIZES['email'] (default 50)
# All chunks go to the 'low' queue.
# Returns a list of task IDs for progress tracking.
task_ids = dispatch_chunked(
    task=send_email_chunk,
    items=recipient_ids,       # list or queryset
    task_type='email',         # resolves chunk size from SYNTEK_TASKS['CHUNK_SIZES']
    parent_task_id=parent_id,  # optional — links chunks to a parent TaskResult
)
```

### Progress reporting

Tasks that support progress update their `TaskResult` record via the `update_progress` method:

```python
class GenerateReport(SyntekTask):
    def run(self, report_id: int) -> None:
        total = get_total_rows(report_id)
        for i, row in enumerate(get_rows(report_id)):
            process_row(row)
            if i % 100 == 0:
                self.update_progress(current=i, total=total)
```

### `@periodic_task` decorator

Registers a Beat-scheduled task with a crontab or interval:

```python
from syntek_tasks import periodic_task
from celery.schedules import crontab

@periodic_task(schedule=crontab(hour=3, minute=0), queue='low')
def nightly_cleanup() -> None:
    ...
```

## GraphQL API

Provided by `syntek_tasks.graphql`. Requires `syntek-tasks` in `INSTALLED_APPS`.

### `TaskResult` model fields

| Field          | Type             | Description                                      |
| -------------- | ---------------- | ------------------------------------------------ |
| `task_id`      | UUID             | Celery task ID                                   |
| `task_name`    | str              | Dotted task name                                 |
| `status`       | enum             | `PENDING` / `STARTED` / `SUCCESS` / `FAILURE`    |
| `progress`     | int (0–100)      | Incremental progress; null when not applicable   |
| `result`       | JSON             | Task return value on success; null otherwise     |
| `error`        | str              | Exception message on failure; null otherwise     |
| `created_at`   | datetime         | When the task was dispatched                     |
| `completed_at` | datetime or null | When the task reached SUCCESS or FAILURE         |
| `tenant_id`    | UUID             | Tenant that owns this task result                |
| `user_id`      | UUID or null     | User who initiated the task (null for scheduled) |

### GraphQL query

```graphql
query TaskStatus($taskId: ID!) {
  taskStatus(taskId: $taskId) {
    taskId
    status
    progress
    result
    error
    createdAt
    completedAt
  }
}
```

Tenancy-scoped: a user can only query task results belonging to their tenant. Returns `null` for
unknown or cross-tenant task IDs (not an error — prevents task ID enumeration).

## Acceptance Criteria

```gherkin
# --- Settings validation ---

Given BROKER_URL is not set
When AppConfig.ready() runs
Then ImproperlyConfigured is raised

Given TASK_SERIALIZER is 'pickle'
When AppConfig.ready() runs
Then ImproperlyConfigured is raised — pickle is banned for security reasons

Given TASK_SOFT_TIME_LIMIT is greater than TASK_TIME_LIMIT
When AppConfig.ready() runs
Then ImproperlyConfigured is raised — soft limit must be less than hard limit

# --- Priority queues ---

Given a task is dispatched to the 'critical' queue and tasks are waiting in 'default'
When Celery workers are running with correct queue bindings
Then the 'critical' task is processed before any 'default' task

Given a task has queue = 'high'
When it is dispatched via .delay() or .apply_async()
Then it appears in the 'high' queue, not 'default'

Given DEFAULT_QUEUE is 'default'
When a task with no explicit queue is dispatched
Then it appears in the 'default' queue

# --- Retry with backoff ---

Given a task fails and MAX_RETRIES is 3 and RETRY_BACKOFF_BASE is 60
When the first failure occurs
Then the task is retried after approximately 60 seconds (±jitter)

Given a task fails on its second retry
When the retry fires
Then the delay is approximately 120 seconds (doubled from first retry, ±jitter)

Given RETRY_JITTER is True
When two identical tasks both fail simultaneously
Then their retry delays differ (jitter applied independently)

Given a task exhausts all retries
When the final failure occurs
Then it is routed to the DLQ queue with full task name, args, kwargs, and exception preserved

Given DLQ_ENABLED is False
When a task exhausts all retries
Then it is discarded — not routed to DLQ

# --- Dead-letter queue ---

Given a task is in the DLQ
When the management command 'replay_dlq --task-id <id>' runs
Then the task is re-dispatched to its original queue with original arguments

Given DLQ_RETAIN_DAYS is 30 and a DLQ entry is 31 days old
When the scheduled cleanup task runs
Then the DLQ entry is deleted

# --- Chunked fan-out ---

Given dispatch_chunked is called with 500 recipient IDs and task_type='email'
When CHUNK_SIZES['email'] is 50
Then 10 chunk tasks are dispatched, each with exactly 50 IDs

Given the last batch has fewer items than the chunk size
When dispatch_chunked completes
Then the final chunk task receives the remainder (not padded)

Given dispatch_chunked is called with task_type='sms' and CHUNK_SIZES has no 'sms' key
When chunks are dispatched
Then DEFAULT_CHUNK_SIZE is used

Given one chunk task fails and exhausts retries
When the DLQ receives it
Then the other chunk tasks continue unaffected

# --- Rate limits ---

Given TASK_RATE_LIMITS includes 'myapp.send_email_chunk': '100/m'
When more than 100 send_email_chunk tasks complete in one minute
Then subsequent executions are deferred by the Celery rate limiter

# --- Celery Beat / scheduled tasks ---

Given a @periodic_task with schedule=crontab(hour=3, minute=0) is registered
When Celery Beat triggers at 03:00
Then the task executes and a TaskResult record is created

Given BEAT_SCHEDULE defines the same task name as a @periodic_task decorator
When Beat schedules are loaded
Then the BEAT_SCHEDULE entry takes precedence

Given django-celery-beat's DatabaseScheduler is active
When the Beat process restarts
Then all registered schedules resume correctly from the database

# --- Progress reporting ---

Given a task calls self.update_progress(current=50, total=100)
When the GraphQL taskStatus query is polled
Then progress is 50

Given a task completes successfully
When taskStatus is queried
Then status is SUCCESS and completedAt is set

Given a task fails after all retries
When taskStatus is queried
Then status is FAILURE and error contains the exception message

# --- GraphQL task status ---

Given a task is dispatched and STORE_RESULTS is True
When taskStatus(taskId: "abc-123") is queried by the owning user
Then status, progress, result, error, createdAt are returned

Given a user queries taskStatus for a task belonging to a different tenant
When the query resolves
Then null is returned — no cross-tenant leakage

Given STORE_RESULTS is False
When taskStatus is queried
Then null is returned for all task IDs

Given STORE_RESULTS_TTL_DAYS is 7 and a TaskResult is 8 days old
When the cleanup task runs
Then the TaskResult record is deleted

# --- Logging and request ID propagation ---

Given syntek-logging is installed and a request with request_id 'req-abc' dispatches a task
When the task runs on a Celery worker
Then 'req-abc' appears in all log output from that task execution

Given TASK_TRACK_STARTED is True
When a task begins execution
Then status transitions to STARTED before run() is called

# --- Timeouts ---

Given TASK_SOFT_TIME_LIMIT is 300 and a task runs for 301 seconds
When the soft time limit triggers
Then SoftTimeLimitExceeded is raised inside the task (catchable for cleanup)

Given TASK_TIME_LIMIT is 360 and a task ignores the soft limit and runs for 361 seconds
When the hard time limit triggers
Then the task process is killed and status is set to FAILURE

# --- Prometheus metrics ---

Given PROMETHEUS_METRICS_ENABLED is True
When /metrics is requested
Then queue depth per lane and task success/failure/retry counters are present
```
