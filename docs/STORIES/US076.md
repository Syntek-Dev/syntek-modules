## Background and Rationale

### SSO Provider Policy

Social login providers that do not **enforce** MFA at the platform level represent an elevated risk
for credential compromise. If a user's Google, Facebook, Instagram, or LinkedIn account is hacked
(phishing, credential stuffing, SIM swap), an attacker gains immediate access to any application
that trusts that provider — even if the application itself has strong auth controls.

The Syntek platform therefore only permits OAuth SSO providers where MFA is either:

- **Enforced by the platform with no opt-out** (e.g. Defguard — MFA is mandatory by design), or
- **Enforceable at the organisation or application level before the OAuth flow completes** (e.g.
  GitHub, where an organisation can mandate MFA for all members before tokens are issued), or
- **MFA-gated by syntek-auth itself** — the provider is permitted through startup validation, but
  after the OAuth callback syntek-auth issues a short-lived `PendingOAuthSession` and requires the
  user to complete a local MFA challenge before a full session token pair is issued. This allows
  consumer social providers (Google, Facebook, etc.) to be used as an identity source while
  guaranteeing that the session is still protected by a second factor owned and verified by
  syntek-auth. See [US009] `completeSocialMfa` mutation.

Providers where MFA is a user-optional setting (Google, Facebook, Instagram, LinkedIn, Twitter/X,
Apple consumer accounts, Discord, Microsoft consumer MSA) are **MFA-gated** — they pass startup
validation but always require a local MFA challenge after the OAuth callback completes.

### Key Rotation Policy

AES-256-GCM encryption keys must be rotated every **90 days** to limit the blast radius of a key
compromise. Keys are versioned so existing ciphertext remains decryptable during the transition
period while new writes use the current key.

### Network Architecture Reference

> **Implementation responsibility**: `syntek-platform` and `syntek-templates`. Documented here as
> the canonical reference that all Syntek deployments must conform to.

All Syntek client deployments use a layered TLS model to ensure traffic is encrypted at every hop:

```text
User browser
    │ HTTPS (Edge TLS — Cloudflare certificate)
    ▼
Cloudflare Edge  ←──  Cloudflare Tunnel (encrypted tunnel, no open ports on origin)
    │ HTTPS (Origin TLS — Syntek-managed certificate)
    ▼
Nginx (reverse proxy, TLS termination at origin)
    │ Unix socket (internal only)
    ▼
Gunicorn (process manager — worker lifecycle, restarts, concurrency)
    │ ASGI (internal)
    ▼
Uvicorn workers (UvicornWorker class — ASGI protocol, async Django)
    │
    ▼
Django ASGI application
```

- **Edge TLS**: Cloudflare issues and manages the user-facing certificate. Always HTTPS.
- **Cloudflare Tunnel**: no inbound ports need to be open on the origin server. The origin initiates
  an outbound encrypted tunnel to Cloudflare — eliminates direct internet exposure.
- **Origin TLS**: a separate certificate (Cloudflare Origin CA or Let's Encrypt) encrypts traffic
  between Cloudflare and the Nginx origin. Prevents a Cloudflare-level compromise from exposing
  plaintext at the origin.
- **Nginx**: terminates TLS, forwards to Gunicorn via Unix socket, serves static/media files
  directly.
  - Syntek-managed client deployments: configured and maintained by `syntek-infrastructure`.
  - Self-hosted deployments (open-source users): an Nginx configuration template is provided by
    `syntek-platform` and `syntek-templates` for developers who host outside the Syntek managed
    infrastructure.
- **Gunicorn + Uvicorn workers**: Gunicorn manages process lifecycle; `UvicornWorker` handles the
  ASGI protocol inside each worker. Required for async Django views, WebSocket connections, and
  async GraphQL subscriptions. Startup configuration provided by
  `syntek-platform`/`syntek-templates`.

The `syntek-security` module is responsible for ensuring Django trusts the proxy headers set by
Nginx (`SECURE_PROXY_SSL_HEADER`, `USE_X_FORWARDED_HOST`, `SECURE_SSL_REDIRECT`).

```gherkin
# --- SSO Provider Allowlist ---

Given `SYNTEK_AUTH` is configured with an OAuth provider
When `AppConfig.ready()` validates the configuration
Then it checks the provider against the MFA-enforcing allowlist; any provider not on the list
raises `ImproperlyConfigured` with a clear message naming the blocked provider

Given the MFA-enforcing allowlist is checked at startup
When an operator configures Google, Facebook, Instagram, LinkedIn, Twitter/X, Apple (consumer),
Discord, or Microsoft (consumer MSA) as an OAuth provider
Then startup passes — these providers are MFA-gated, not blocked; after the OAuth callback
syntek-auth issues a PendingOAuthSession requiring local MFA before tokens are issued

Given a user authenticates via an MFA-gated provider (e.g. Google, Facebook)
When the oidcCallback mutation processes the provider's callback
Then a PendingOAuthSession is created with a TTL of OAUTH_MFA_PENDING_TTL seconds (default 600)
and an OidcCallbackPayload is returned with mfa_pending: True and a pending_token;
no access_token or refresh_token is issued at this stage

Given a PendingOAuthSession with a valid pending_token exists for a user who has MFA configured
When the completeSocialMfa mutation is called with a valid MFA proof
Then the PendingOAuthSession row is deleted (single-use), a full TokenPair is issued,
and success: True is returned

Given a PendingOAuthSession has expired (past its expires_at timestamp)
When the completeSocialMfa mutation is called with the expired pending_token
Then the mutation returns success: False with error_code: "pending_token_expired"

Given a user authenticates via an MFA-gated provider for the first time (no MFA configured)
When the oidcCallback mutation processes the provider's callback
Then OidcCallbackPayload is returned with mfa_pending: True and mfa_setup_required: True;
the user must complete MFA enrolment before completeSocialMfa can issue a full session

Given a user authenticates via an allowed provider (e.g. GitHub)
When the OAuth callback is processed
Then the session is issued — no additional MFA check is needed for GitHub as 2FA is mandatory
at the platform level for all GitHub accounts

Given an operator wants to add a self-hosted identity provider (e.g. Defguard, Authentik,
Keycloak)
When the provider is added to `SYNTEK_AUTH['OAUTH_ALLOWED_PROVIDERS']`
Then it is accepted if and only if the provider's OIDC/OAuth metadata includes MFA enforcement
claims or the operator explicitly certifies MFA enforcement via a settings flag

# --- Encryption Key Rotation ---

Given encryption keys are in use for AES-256-GCM field encryption
When 90 days have elapsed since a key was last rotated
Then the key rotation task triggers automatically, generates a new key, and updates the active
key reference in the secrets store

Given a key has been rotated
When a read request arrives for a row encrypted with the previous key
Then the system decrypts successfully using the previous key (identified by key version prefix
on the ciphertext) — no data loss occurs during transition

Given a key rotation is in progress
When a write request arrives
Then the new key is used for all new encryptions — old ciphertexts are not immediately
re-encrypted (lazy rotation pattern)

Given a key rotation completes
When all rows have been re-encrypted with the new key (background migration)
Then the previous key is retired and removed from the active keyring

Given a key rotation fails mid-process
When the error is detected
Then the previous key remains active, the failure is logged via syntek-logging, and an alert
is raised — no data is left in an inconsistent state

# --- Proxy Trust Settings ---

Given `SYNTEK_SECURITY` is installed and a reverse proxy is in use
When the Django application starts
Then `SECURE_PROXY_SSL_HEADER`, `USE_X_FORWARDED_HOST`, and `SECURE_SSL_REDIRECT` are set
correctly in the `SYNTEK_SECURITY` defaults so the application trusts the proxy's TLS termination
```

## Key Versioning Scheme

Each ciphertext produced by `encrypt_field` / `encrypt_fields_batch` is prefixed with a 2-byte key
version identifier before base64 encoding:

```text
stored value = base64( version_bytes(2) || nonce(12) || ciphertext || tag(16) )
```

The active key version is stored in the secrets manager. On read, the version bytes are stripped,
the correct key is resolved from the keyring, and decryption proceeds. On rotation, the version
counter increments. This scheme is backwards-compatible — rows encrypted with `v1` remain readable
after rotation to `v2`.

> **Note**: The key versioning prefix is a change to the ciphertext layout defined in US006. US006
> must be updated to support an optional `key_version: u16` parameter or to read/write the prefix
> transparently.

### Policy Revision — Social OAuth MFA Gating (US009)

The original "blocked" category has been reclassified to **MFA-gated**. Providers in that set
(Google, Facebook, Instagram, LinkedIn, Twitter/X, Apple, Discord, Microsoft consumer MSA) now pass
startup validation but trigger a `PendingOAuthSession` flow in `oidcCallback`. The user must
complete a local MFA challenge via the `completeSocialMfa` mutation before a full `TokenPair` is
issued. This is implemented in US009 via:

- `backends/allowlist.py` — `MFA_GATED_PROVIDERS` set + `is_mfa_gated_provider()` helper
- `models/oauth_pending.py` — `PendingOAuthSession` model (UUID pk, user FK, provider, expires_at)
- `services/oauth_mfa.py` — `issue_oauth_pending_session()` + `complete_oauth_mfa()`
- `types/auth.py` — `OidcCallbackPayload` (replaces `AuthPayload` for `oidcCallback`)
- `mutations/oidc.py` — updated `oidcCallback` return type + new `completeSocialMfa` mutation
- Optional setting: `SYNTEK_AUTH['OAUTH_MFA_PENDING_TTL']` (default 600 s, range 60–3600)
