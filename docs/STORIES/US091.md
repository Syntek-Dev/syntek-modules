## User Story

As a developer, I want a legal document management module so that I can publish versioned Terms &
Conditions, Privacy Policies, Cookie Policies, and any other legal documents — tracking which users
have accepted which version, enforcing re-acceptance when new versions are published, and
integrating with GDPR erasure.

## MoSCoW Priority

**Must Have**

## Acceptance Criteria

```gherkin
Given a new version of the Terms & Conditions is published with `requires_acceptance = True`
When an authenticated user makes a request to any acceptance-guarded endpoint
Then a `LegalReacceptanceRequired` error is returned with the document type and current version ID

Given a user accepts the current version of a legal document
When the acceptance mutation is called
Then a UserLegalAcceptance record is created with user FK, document version FK, accepted_at,
ip_hash, and method — and the event `user.accepted_legal_doc` is fired via syntek-events

Given a user requests their data via SAR (Subject Access Request)
When the GDPR export runs
Then all UserLegalAcceptance records for that user are included in the export

Given an erasure request is processed for a user
When the erasure handler runs
Then UserLegalAcceptance records are retained (contractual evidence) but the ip_hash field is
anonymised — timestamps and version references are preserved

Given a project consumer configures `SYNTEK_LEGAL` with custom document types
When the module starts
Then only those configured document types are active; standard types can be renamed or disabled

Given a legal document is queried without authentication
When the public endpoint is called
Then the current active version content is returned — legal documents are publicly readable
```

## Architecture Decision — Legal vs GDPR

`syntek-legal` manages legal **documents** (contractual). `syntek-gdpr` manages data **processing
consent** (GDPR). They are separate modules with a clean integration point:

| Module         | Concern                                     | Legal Basis             |
| -------------- | ------------------------------------------- | ----------------------- |
| `syntek-gdpr`  | Processing consent, SAR, erasure, retention | GDPR Articles 6, 17, 20 |
| `syntek-legal` | T&Cs, Privacy Policy, legal doc versioning  | Contract / Statutory    |

A `ConsentRecord` in `syntek-gdpr` can optionally reference a `LegalDocumentVersion` ID (e.g. "I
consent to the Privacy Policy version 3.2"). The reference is stored as a string — no FK across
module boundaries.

## Data Model

### `LegalDocumentType`

Configuration model — one row per document type. Pre-populated from `SYNTEK_LEGAL` settings.

| Field           | Type      | Notes                                                                  |
| --------------- | --------- | ---------------------------------------------------------------------- |
| `slug`          | SlugField | Identifier: `terms`, `privacy`, `cookie`, `data-processing`, or custom |
| `display_name`  | CharField | Human-readable name, overridable via settings                          |
| `requires_auth` | bool      | True = only authenticated users can accept                             |
| `public_read`   | bool      | True = content readable without login                                  |
| `is_active`     | bool      |                                                                        |

### `LegalDocumentVersion`

Immutable — once published, a version cannot be edited, only superseded.

| Field                 | Type      | Notes                                                           |
| --------------------- | --------- | --------------------------------------------------------------- |
| `document_type`       | FK        | `LegalDocumentType`                                             |
| `version`             | CharField | Semantic version string, e.g. `3.2.0`                           |
| `content_html`        | TextField | Rendered HTML content (sanitised via syntek-security-validator) |
| `content_markdown`    | TextField | Canonical source — rendered to HTML on publish                  |
| `effective_from`      | DateField | Date this version becomes active                                |
| `published_at`        | datetime  | When `is_published` was set True                                |
| `is_published`        | bool      | Immutable once True                                             |
| `requires_acceptance` | bool      | If True, authenticated users must re-accept before continuing   |
| `published_by`        | FK        | Auth user who published                                         |
| `tenant`              | FK (null) | Null = global; set = tenant-specific override                   |

### `UserLegalAcceptance`

Immutable record of each acceptance event. Never updated, only appended.

| Field              | Type      | Notes                                            |
| ------------------ | --------- | ------------------------------------------------ |
| `user`             | FK        | Auth user                                        |
| `document_type`    | FK        | `LegalDocumentType`                              |
| `document_version` | FK        | `LegalDocumentVersion`                           |
| `accepted_at`      | datetime  | UTC timestamp                                    |
| `ip_hash`          | CharField | SHA-256 of IP — anonymised on GDPR erasure       |
| `user_agent_hash`  | CharField | SHA-256 of UA string                             |
| `method`           | CharField | `registration` / `prompted` / `settings` / `api` |
| `tenant`           | FK (null) | Tenant context of acceptance                     |

### `LegalReacceptanceRequired`

In-memory exception raised by `check_acceptance()`. Not a model. Caught by a Django middleware or
GraphQL extension and converted to a typed error in the API response.

## Configurable Document Types

All document types are configured via `SYNTEK_LEGAL`. Consuming projects can add, rename, disable,
or replace any type:

```python
SYNTEK_LEGAL = {
    'DOCUMENT_TYPES': [
        {
            'slug': 'terms',
            'display_name': 'Terms & Conditions',
            'requires_auth': True,
            'public_read': True,
        },
        {
            'slug': 'privacy',
            'display_name': 'Privacy Policy',
            'requires_auth': False,
            'public_read': True,
        },
        {
            'slug': 'cookie',
            'display_name': 'Cookie Policy',
            'requires_auth': False,
            'public_read': True,
        },
        {
            'slug': 'data-processing',
            'display_name': 'Data Processing Agreement',
            'requires_auth': True,
            'public_read': False,
        },
        # Custom types — any slug is valid
        {
            'slug': 'acceptable-use',
            'display_name': 'Acceptable Use Policy',
            'requires_auth': True,
            'public_read': True,
        },
    ],
    'ENFORCEMENT_MODE': 'error',   # 'error' | 'redirect' | 'none' (for testing)
    'ACCEPTANCE_GRACE_PERIOD_DAYS': 7,  # days after new version before enforcement kicks in
    'GDPR_ANONYMISE_IP_ON_ERASURE': True,
}
```
