## User Story

As a developer building a multi-tenant application, I want schema-per-tenant isolation so that no
tenant's data is ever accessible from another tenant's request — with full control over domain
resolution strategy, tenant-to-tenant relationship rules, provisioning behaviour, and isolation
mode, all configured from `settings.py`.

## Configuration Contract

All behaviour is controlled via `SYNTEK_TENANCY` in `settings.py`. Validated at startup in
`AppConfig.ready()`.

```python
SYNTEK_TENANCY = {
    # --- Isolation ---
    'ISOLATION': 'schema',              # 'schema' | 'row'
                                        # 'schema' — each tenant gets a dedicated PostgreSQL schema
                                        # 'row'    — all tenants share tables; rows tagged with tenant_id
    'PUBLIC_SCHEMA_NAME': 'public',     # str — name of the shared Postgres schema
    'SCHEMA_PREFIX': 'tenant_',         # str — prefix for auto-generated schema names (schema isolation only)

    # --- Domain resolution ---
    'DOMAIN_STRATEGY': 'subdomain',     # 'subdomain' | 'custom_domain' | 'path'
                                        # 'subdomain'     — tenant resolved from Host header subdomain (e.g. acme.example.com)
                                        # 'custom_domain' — tenant resolved from full custom domain (e.g. acme.io)
                                        # 'path'          — tenant resolved from URL path prefix (e.g. /t/acme/)
    'SUBDOMAIN_BASE': '',               # str — required when DOMAIN_STRATEGY is 'subdomain' (e.g. 'example.com')
    'ALLOW_CUSTOM_DOMAINS': False,      # bool — tenants may register their own domain after DNS verification
    'CUSTOM_DOMAIN_VERIFICATION': 'dns_txt',  # 'dns_txt' | 'http_file' — verification method for custom domains

    # --- Tenant relationships ---
    'TENANT_RELATIONSHIPS': False,      # bool — enable tenant-to-tenant linking
    'RELATIONSHIP_TYPES': ['parent_child', 'partner', 'reseller'],
                                        # list[str] — permitted relationship types (requires TENANT_RELATIONSHIPS: True)
    'CROSS_TENANT_QUERIES': False,      # bool — allow explicitly authorised read-only queries across linked tenants
                                        #        (never cross-tenant writes; always requires explicit grant)

    # --- Provisioning ---
    'AUTO_PROVISION': False,            # bool — automatically create schema on first request if tenant record exists
    'PROVISIONING_STRATEGY': 'async',   # 'sync' | 'async' — async requires syntek-tasks
    'SEED_ON_PROVISION': True,          # bool — run seed fixtures after schema creation
    'ADMIN_ON_PROVISION': True,         # bool — create a default admin user in the new tenant schema

    # --- Limits ---
    'MAX_TENANTS': 0,                   # int — maximum number of active tenants (0 = unlimited)
}
```

**Key rules:**

- `DOMAIN_STRATEGY: 'subdomain'` requires `SUBDOMAIN_BASE` to be set — validated at startup.
- `TENANT_RELATIONSHIPS: False` makes `RELATIONSHIP_TYPES` and `CROSS_TENANT_QUERIES` inert.
- `CROSS_TENANT_QUERIES: True` only permits reads authorised by an explicit `CrossTenantGrant`
  record — never implicit.
- `PROVISIONING_STRATEGY: 'async'` requires `syntek-tasks` to be installed.
- `ISOLATION: 'row'` does not create Postgres schemas; `SCHEMA_PREFIX` and `PUBLIC_SCHEMA_NAME` are
  ignored.

## Acceptance Criteria

```gherkin
# --- Settings validation ---

Given `DOMAIN_STRATEGY` is 'subdomain' and `SUBDOMAIN_BASE` is empty
When `AppConfig.ready()` runs
Then `ImproperlyConfigured` is raised — SUBDOMAIN_BASE is required for subdomain strategy

Given `PROVISIONING_STRATEGY` is 'async' and `syntek-tasks` is not in INSTALLED_APPS
When `AppConfig.ready()` runs
Then `ImproperlyConfigured` is raised — async provisioning requires syntek-tasks

Given `CROSS_TENANT_QUERIES` is True and `TENANT_RELATIONSHIPS` is False
When `AppConfig.ready()` runs
Then `ImproperlyConfigured` is raised — cross-tenant queries require tenant relationships to be enabled

# --- Domain resolution ---

Given `DOMAIN_STRATEGY` is 'subdomain' and `SUBDOMAIN_BASE` is 'example.com'
When a request arrives with Host header 'acme.example.com'
Then the active tenant is resolved as the tenant with slug 'acme'

Given `DOMAIN_STRATEGY` is 'custom_domain'
When a request arrives with Host header 'acme.io' mapped to a verified custom domain
Then the active tenant is resolved from the custom domain record

Given `DOMAIN_STRATEGY` is 'path'
When a request arrives with path '/t/acme/dashboard'
Then the active tenant is resolved as the tenant with slug 'acme' and the path prefix is stripped before routing

Given `ALLOW_CUSTOM_DOMAINS` is False
When a tenant attempts to register a custom domain via the mutation
Then the mutation returns an error — custom domains are not enabled for this deployment

Given `ALLOW_CUSTOM_DOMAINS` is True and `CUSTOM_DOMAIN_VERIFICATION` is 'dns_txt'
When a tenant registers a custom domain
Then a DNS TXT record challenge is returned; the domain is only activated after the challenge is verified

# --- Isolation ---

Given `ISOLATION` is 'schema' and a request arrives for tenant 'acme'
When the TenancyMiddleware runs
Then the PostgreSQL search path is set to 'tenant_acme' for the duration of that request only

Given a cross-tenant query is attempted programmatically without a CrossTenantGrant
When the ORM executes it
Then it is blocked and raises `CrossTenantQueryError`

Given two concurrent requests arrive for different tenants
When both are processed simultaneously
Then each operates entirely within its own schema with no data leakage between them

# --- Tenant relationships ---

Given `TENANT_RELATIONSHIPS` is False
When the linkTenants mutation is called
Then an error is returned — tenant relationships are not enabled for this deployment

Given `TENANT_RELATIONSHIPS` is True and `RELATIONSHIP_TYPES` includes 'parent_child'
When a parent tenant creates a child tenant relationship
Then the relationship is stored and the parent can query child tenant summary data (read-only, explicit grant only)

Given `CROSS_TENANT_QUERIES` is True and a CrossTenantGrant exists between tenant A and tenant B
When tenant A queries a resource explicitly included in the grant
Then the query succeeds and returns tenant B's data for that resource only

Given `CROSS_TENANT_QUERIES` is True but no CrossTenantGrant exists
When tenant A attempts to query tenant B's data
Then `CrossTenantQueryError` is raised — implicit cross-tenant access is never permitted

# --- Provisioning ---

Given `PROVISIONING_STRATEGY` is 'async'
When a new tenant is created via the provisionTenant mutation
Then the provisioning task is queued and the mutation returns immediately with a status of 'provisioning'

Given a new tenant provisioning task completes
When the task finishes
Then the tenant schema exists, all migrations have run, seed data is applied (if SEED_ON_PROVISION),
and a default admin user is created (if ADMIN_ON_PROVISION)

Given a tenant is suspended
When any request arrives for that tenant
Then a 403 is returned with a suspension notice and no data is accessible

# --- Limits ---

Given `MAX_TENANTS` is 10 and 10 active tenants exist
When the provisionTenant mutation is called
Then it is rejected with a clear error — maximum tenant count reached
```
