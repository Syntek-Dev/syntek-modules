# US081 — `syntek-security-secrets`: Provider-Agnostic Secrets Management

**Last Updated**: 10/03/2026 **Version**: 0.1.0 **Status**: To Do **Epic**: Epic 3 — Core Backend
Modules

## User Story

As a developer, I want a provider-agnostic secrets management module so that Syntek applications can
load credentials from a plain `.env` file in development and from OpenBao, Vault, AWS, GCP, or Azure
in production — with typed environment variable validation, in-process caching, automatic secret
rotation detection, and built-in Django database connection recycling — all controlled from
`settings.py` without touching module source.

## MoSCoW Priority

**Must Have**

## Positioning vs django-environ

This module is a **replacement** for `django-environ`, not a complement. `django-environ` only
parses and coerces env vars from a `.env` file — it has no provider abstraction, no rotation, and no
secrets manager. `syntek-security-secrets` covers the full credential lifecycle from `.env` in dev
to OpenBao in production. Projects using `django-environ` should migrate; there is no benefit to
running both.

## Configuration Contract

All behaviour is controlled via `SYNTEK_SECRETS` in `settings.py`. Validated at startup in
`AppConfig.ready()`.

```python
SYNTEK_SECRETS = {
    # --- Provider selection ---
    'PROVIDER': 'env',              # 'env' | 'openbao' | 'vault' | 'aws' | 'gcp' | 'azure' | 'custom'
                                    # 'env'     — reads from os.environ / .env file (default, dev)
                                    # 'openbao' — OpenBao (open-source Vault fork, recommended production)
                                    # 'vault'   — HashiCorp Vault (legacy / enterprise)
                                    # 'aws'     — AWS Secrets Manager
                                    # 'gcp'     — GCP Secret Manager
                                    # 'azure'   — Azure Key Vault
                                    # 'custom'  — bring your own provider class

    # --- Environment variable validation (applies to ALL providers) ---
    'REQUIRED_ENV_VARS': [],        # list[str] — env var names that must be present at startup
    'TYPED_ENV_VARS': {},           # dict[str, dict] — typed schema for env vars
                                    # Format: 'VAR_NAME': {'type': 'str|int|bool|url|email', 'required': True}
                                    # 'DATABASE_URL': {'type': 'url', 'required': True},
                                    # 'MAX_CONNECTIONS': {'type': 'int', 'required': False, 'default': 20},
                                    # 'DEBUG': {'type': 'bool', 'required': False, 'default': False},

    # --- OpenBao / HashiCorp Vault (provider: 'openbao' or 'vault') ---
    'VAULT_URL': env('VAULT_URL', default=''),
    'VAULT_TOKEN': env('VAULT_TOKEN', default=''),          # token auth
    'VAULT_ROLE_ID': env('VAULT_ROLE_ID', default=''),      # AppRole auth
    'VAULT_SECRET_ID': env('VAULT_SECRET_ID', default=''),
    'VAULT_MOUNT': 'secret',
    'VAULT_PATH_PREFIX': '',
    'VAULT_NAMESPACE': '',          # HCP Vault enterprise only
    'VAULT_TLS_VERIFY': True,
    'VAULT_TLS_CERT': env('VAULT_TLS_CERT', default=''),    # mTLS client cert path
    'VAULT_TLS_KEY': env('VAULT_TLS_KEY', default=''),      # mTLS client key path

    # --- AWS Secrets Manager (provider: 'aws') ---
    # AWS credentials from boto3 chain ONLY — never in this dict
    'AWS_REGION': env('AWS_REGION', default='eu-west-2'),
    'AWS_SECRET_PREFIX': '',

    # --- GCP Secret Manager (provider: 'gcp') ---
    # GCP credentials from ADC (GOOGLE_APPLICATION_CREDENTIALS) ONLY — never in this dict
    'GCP_PROJECT': env('GCP_PROJECT', default=''),
    'GCP_SECRET_PREFIX': '',

    # --- Azure Key Vault (provider: 'azure') ---
    # Azure credentials from DefaultAzureCredential chain ONLY — never in this dict
    'AZURE_VAULT_URL': env('AZURE_VAULT_URL', default=''),
    'AZURE_SECRET_PREFIX': '',

    # --- Custom provider ---
    'CUSTOM_PROVIDER_CLASS': '',    # dotted import path to class implementing SecretProvider ABC
    'CUSTOM_PROVIDER_OPTIONS': {},  # dict — passed as kwargs to provider __init__

    # --- Rotation and live reload ---
    'ROTATION_ENABLED': False,
    'ROTATION_POLL_INTERVAL': 300,      # seconds
    'ROTATION_KEYS': [],                # empty = poll all keys
    'ROTATION_ON_CHANGE': None,         # dotted path to callback(key: str, new_value: str) -> None
    'ROTATION_DB_RECYCLE': True,        # bool — built-in DB connection recycling on rotation
                                         # When True and rotated key matches ROTATION_DB_KEY_PATTERNS,
                                         # calls django.db.connections.close_all() automatically
    'ROTATION_DB_KEY_PATTERNS': [       # list[str] — key name substrings triggering DB recycling
        'DB_PASS', 'DATABASE_PASS', 'DATABASE_PASSWORD', 'DB_PASSWORD',
    ],

    # --- In-process caching ---
    'CACHE_SECRETS': True,
    'CACHE_TTL': 300,               # seconds; set lower than ROTATION_POLL_INTERVAL when rotation enabled
                                     # In-process only — secrets must never go into Redis/shared cache
}
```

**Key rules:**

- `PROVIDER: 'env'` requires no extra packages. Others require optional extras:
  `syntek-security-secrets[openbao]`, `[vault]`, `[aws]`, `[gcp]`, `[azure]`.
- Cloud credentials NEVER go in the `SYNTEK_SECRETS` dict — enforced by settings validator at
  startup.
- `TYPED_ENV_VARS` runs for ALL providers (even Vault/OpenBao may need local env vars like
  `VAULT_URL`).
- `CACHE_SECRETS: True` uses module-level in-process dict. TTL eviction by timestamp comparison. NOT
  Redis.
- `ROTATION_ENABLED: True` spawns a daemon thread at startup. Each worker polls independently — no
  distributed lock.
- Built-in DB recycling: when a rotated key name matches any pattern in `ROTATION_DB_KEY_PATTERNS`
  (case-insensitive), calls `django.db.connections.close_all()`. Both built-in recycling and
  `ROTATION_ON_CHANGE` run on the same event — not mutually exclusive.
- `secret_rotated` Django signal is also fired on every rotation for signal-based listeners.

## Provider Abstraction

```python
class SecretProvider(ABC):
    def get(self, key: str) -> str: ...         # raises SecretNotFoundError or SecretProviderError
    def list_keys(self) -> list[str]: ...        # for rotation polling; may return [] if provider doesn't support enumeration
    def health_check(self) -> bool: ...          # default implementation calls list_keys()
```

**Provider implementations:**

- **`env`**: Reads from `os.environ`. Supports `.env` file via `python-dotenv` if installed
  (optional).
- **`openbao`**: OpenBao (Linux Foundation open-source Vault fork, wire-compatible with Vault API).
  Uses `hvac` or `openbao-client` library. AppRole and token auth both supported. KV v2 secrets
  engine.
- **`vault`**: Identical to openbao except supports HCP Vault `VAULT_NAMESPACE` (enterprise only).
- **`aws`**: AWS Secrets Manager. Uses `boto3`. Credentials from standard chain (env vars,
  `~/.aws/credentials`, IAM roles). Never hardcode `AWS_ACCESS_KEY_ID` in settings.
- **`gcp`**: GCP Secret Manager. Uses `google-cloud-secret-manager`. Credentials from Application
  Default Credentials (ADC).
- **`azure`**: Azure Key Vault. Uses `azure-keyvault-secrets` + `azure-identity`
  (DefaultAzureCredential chain).
- **`custom`**: Dynamically imports `CUSTOM_PROVIDER_CLASS`, passes `CUSTOM_PROVIDER_OPTIONS` as
  kwargs.

## Public API

```python
from syntek_security_secrets import get_secret, rotate_secret

# Get a secret from the configured provider (with caching if enabled)
db_password = get_secret('DATABASE_PASSWORD')

# Manually trigger rotation of a specific secret
rotate_secret('DATABASE_PASSWORD')
```

Then django.db.connections.close_all() is NOT called

Given ROTATION_DB_RECYCLE is False
When any secret rotates, including a DB password
Then django.db.connections.close_all() is NOT called

Given an exception occurs during rotation polling
When the exception is caught
Then it is logged, and polling continues on the next interval

Given each Django process has ROTATION_ENABLED set to True
When secrets are rotated
Then each process polls independently — no distributed lock is used

# --- Signal handling ---

Given a listener is registered for secret_rotated signal
When a secret rotates
Then the signal is fired and the listener is called with sender, key, old_value, new_value

Given multiple listeners are registered for secret_rotated
When a secret rotates
Then all listeners are called in registration order

# --- Health check ---

Given health_check() is called on any provider
When the provider is reachable
Then True is returned

Given health_check() is called and the provider is unreachable
When a connectivity error occurs
Then False is returned
