# US090 — `syntek-bus`: Internal Domain Event Bus

**Last Updated**: 10/03/2026 **Version**: 0.1.0 **Status**: To Do **Epic**: Epic 4 — Backend Feature
Modules

## User Story

As a developer, I want an internal domain event bus so that all Syntek modules can fire typed domain
events that other modules and consuming projects can subscribe to — decoupling producers from
consumers without polling or tight coupling.

## MoSCoW Priority

**Must Have**

## Acceptance Criteria

```gherkin
Given a module fires a domain event via the event dispatcher
When a synchronous subscriber is registered for that event type
Then the subscriber receives the event payload in the same request/transaction cycle

Given a module fires a domain event via the event dispatcher
When an asynchronous subscriber is registered for that event type
Then a Celery task is enqueued and the subscriber is called outside the request cycle

Given an event is fired and a subscriber raises an exception
When the dispatcher handles the failure
Then the exception is logged and other subscribers are still called — one bad subscriber cannot
block the chain

Given `syntek-webhooks` is installed
When a domain event is fired
Then the webhook dispatcher automatically receives the event and delivers it to all registered
outbound endpoints for that event type — no manual wiring required

Given a module registers its event types at startup
When the event registry is queried
Then all registered event types with their schema and owning module are listed

Given an event history record exists for a fired event
When an admin queries `domainEventHistory`
Then the record includes event type, payload hash, source module, fired_at, and subscriber results
```

## Architecture

`syntek-bus` is the internal domain event bus. It is distinct from `syntek-webhooks`:

| Layer             | Module            | Purpose                                              |
| ----------------- | ----------------- | ---------------------------------------------------- |
| **Domain events** | `syntek-bus`      | In-process typed events between modules (this story) |
| **HTTP delivery** | `syntek-webhooks` | Outbound signed HTTP delivery to external endpoints  |

`syntek-webhooks` subscribes to `syntek-bus` automatically when both are installed. Consuming
projects never need to wire the two together manually.

## Domain Events Fired by Syntek Modules

Each installed module registers its own event types. Examples:

| Module                       | Event Type                    | Payload Fields                                 |
| ---------------------------- | ----------------------------- | ---------------------------------------------- |
| `syntek-notifications-core`  | `notification.sent`           | notification_id, channel, recipient_id         |
| `syntek-notifications-core`  | `notification.failed`         | notification_id, channel, error, attempt       |
| `syntek-notifications-email` | `email.bounced`               | recipient_email_hash, bounce_type, campaign_id |
| `syntek-email-marketing`     | `campaign.send_started`       | campaign_id, recipient_count, send_batch_id    |
| `syntek-email-marketing`     | `campaign.send_completed`     | campaign_id, delivered, bounced, failed        |
| `syntek-email-marketing`     | `subscriber.joined`           | list_id, subscriber_id                         |
| `syntek-email-marketing`     | `subscriber.unsubscribed`     | list_id, subscriber_id, method                 |
| `syntek-email-marketing`     | `subscriber.bounced.hard`     | subscriber_id, campaign_id                     |
| `syntek-auth`                | `user.registered`             | user_id, method                                |
| `syntek-auth`                | `user.login`                  | user_id, ip_hash, mfa_used                     |
| `syntek-payments`            | `payment.succeeded`           | order_id, amount_pence, currency               |
| `syntek-payments`            | `payment.failed`              | order_id, error_code                           |
| `syntek-legal`               | `legal_doc.version_published` | doc_type, version_id, requires_acceptance      |
| `syntek-legal`               | `user.accepted_legal_doc`     | user_id, doc_type, version_id                  |

## Tasks

- [ ] **Red phase:** write failing tests — synchronous dispatch, async Celery dispatch, subscriber
      exception isolation, webhook auto-subscription, event registry listing, event history
      recording
- [ ] Implement `DomainEvent` base dataclass — fields: `event_type` (dotted string),
      `source_module`, `tenant_id`, `fired_at`, `payload` (typed dict), `correlation_id` (UUID for
      tracing)
- [ ] Implement `EventRegistry` — modules call `registry.register(event_type, schema_class)` in
      `AppConfig.ready()`; singleton per process; queryable via GraphQL
- [ ] Implement `EventDispatcher` — `fire(event)` method; dispatches to all registered sync
      subscribers immediately; enqueues Celery tasks for all async subscribers
- [ ] Implement subscriber registration — `@subscribe(event_type, mode='sync'|'async')` decorator;
      modules register subscribers in `AppConfig.ready()`
- [ ] Implement subscriber exception isolation — each subscriber called in a try/except; failures
      logged via `syntek-logging` and do not propagate to caller
- [ ] Implement `DomainEventHistory` model — stores event type, source module, tenant FK, fired_at,
      payload_hash (SHA-256 of serialised payload), correlation_id; payload never stored in clear
- [ ] Implement Django ORM signal bridge — `orm_event(signal, sender, event_type)` helper wraps
      post_save/post_delete signals as domain events; reduces boilerplate in consuming modules
- [ ] Implement webhook auto-subscription — if `syntek-webhooks` is in `INSTALLED_APPS`, register a
      global async subscriber that calls `syntek-webhooks`' outbound dispatch for all events
- [ ] Implement `SYNTEK_EVENTS` settings schema (history_enabled toggle, history_retention_days,
      async_timeout, max_sync_subscribers per event)
- [ ] Implement Strawberry queries: `eventRegistry`, `domainEventHistory(event_type, days)`
- [ ] Create `syntek.manifest.toml` — module ID, settings keys, and `INSTALLED_APPS` entry
- [ ] **Green phase:** all event bus tests pass including cross-module isolation
