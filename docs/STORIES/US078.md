## User Story

As a developer, I want a React observability package so that I can capture frontend errors and
structured logs separately — crashes go direct to Sentry or Glitchtip for reliability, while
structured logs are shipped to the Django backend (or any custom endpoint) for server-side
enrichment and unified log storage — all configured via `SyntekLoggingProvider` with no changes to
individual components.

## MoSCoW Priority

**Must Have**

## Architecture: Hybrid Model

```text
Browser
  ├── Crashes & errors  →  errorBackend (direct to Sentry/Glitchtip)
  │                        Reliable — does not depend on your server being up.
  │
  └── Structured logs   →  logBackend
        'syntek'  →  POST /api/frontend-logs/ (Django enriches + forwards to Loki etc.)
        'custom'  →  POST to any endpoint you control (ELK, Datadog, etc.)
        'none'    →  disabled
```

Syntek deployments use `errorBackend="sentry"` (Glitchtip DSN) + `logBackend="syntek"`. Other
developers choose whatever fits their stack.

## Configuration Contract

```tsx
import { SyntekLoggingProvider } from "@syntek/ui-logging";

<SyntekLoggingProvider
  // ─── Error backend (direct, for reliability) ──────────────────────────────
  // Crashes and unhandled errors go here directly from the browser.
  // Does not depend on your server being up.

  errorBackend="sentry" // 'sentry' | 'console' | 'none'
  // 'sentry'  — Sentry SaaS or Glitchtip (same @sentry/react SDK)
  // 'console' — browser console only (development)
  // 'none'    — disable error tracking

  // Sentry / Glitchtip options (errorBackend: 'sentry')
  dsn={process.env.NEXT_PUBLIC_SENTRY_DSN} // string — Sentry or Glitchtip DSN
  environment={process.env.NEXT_PUBLIC_ENV}
  release={process.env.NEXT_PUBLIC_RELEASE}
  tracesSampleRate={0.1} // number 0.0–1.0
  profilesSampleRate={0.0} // number 0.0–1.0
  sendDefaultPii={false} // boolean
  // ─── Log backend (structured logs) ────────────────────────────────────────
  // Structured log statements from useLogger go here.
  // Recommended: 'syntek' — posts to Django, enriched with tenant + user context.

  logBackend="syntek" // 'syntek' | 'custom' | 'none'
  // 'syntek' — POST to /api/frontend-logs/ on the Django backend.
  //            JWT attached automatically from the active session.
  //            Base URL inherited from @syntek/api-client config.
  // 'custom' — POST to any endpoint you control.
  // 'none'   — disable structured log shipping.

  // Custom endpoint options (logBackend: 'custom')
  logEndpoint="" // string — full URL to POST log batches to
  logHeaders={{}} // Record<string, string> — additional request headers
  logBatchSize={50} // number — entries buffered before POST
  logFlushInterval={10} // number (seconds) — max time before flushing
  // ─── Shared options ────────────────────────────────────────────────────────

  logLevel="warn" // 'debug' | 'info' | 'warn' | 'error'
  // Minimum level sent to logBackend.
  // errorBackend always captures errors regardless of this setting.

  // Error boundary
  errorFallback={<ErrorPage />} // ReactNode — rendered on unrecoverable render error
  // Receives { error, resetError } as props if a component.

  captureConsole={false} // boolean — intercept console.error and send to errorBackend
  ignoreErrors={[
    // list of strings or regex — suppress matching errors
    "ResizeObserver loop",
    /ChunkLoadError/,
  ]}
>
  <App />
</SyntekLoggingProvider>;
```

**Key rules:**

- `errorBackend` and `logBackend` are independent — either can be set to `'none'` without affecting
  the other.
- `errorBackend: 'sentry'` requires `dsn`. Missing DSN logs a warning and disables the backend
  gracefully — the app does not crash.
- `logBackend: 'syntek'` requires `@syntek/api-client` to be configured. The JWT is attached
  automatically from the active session; unauthenticated log entries are submitted without a token
  (accepted or rejected depending on `FRONTEND_LOGS_REQUIRE_AUTH` on the server).
- `logBackend: 'custom'` requires `logEndpoint`. Missing endpoint logs a warning and disables the
  backend gracefully.
- `logLevel` applies only to `logBackend`. The `errorBackend` always captures unhandled errors and
  crashes.
- Glitchtip and Sentry use the same `@sentry/react` SDK — point `dsn` at your Glitchtip instance.
- Grafana is a visualisation layer, not a log target. Use `logBackend: 'syntek'` or `'custom'`
  pointing at Loki.

## Acceptance Criteria

```gherkin
# --- Configuration ---

Given `errorBackend` is 'sentry' and `dsn` is not provided
When the provider mounts
Then a console warning is logged and the error backend is disabled — the app does not crash

Given `logBackend` is 'custom' and `logEndpoint` is empty
When the provider mounts
Then a console warning is logged and the log backend is disabled — the app does not crash

Given `errorBackend` is 'none' and `logBackend` is 'syntek'
When the provider mounts
Then structured logs are shipped via Django but no error tracking is active

# --- Error boundary ---

Given an unrecoverable React render error occurs
When the error boundary catches it
Then `errorFallback` is rendered and the error is sent to `errorBackend`

Given `errorFallback` receives `resetError`
When the user triggers reset
Then the error boundary resets and attempts to re-render

# --- errorBackend: sentry ---

Given `errorBackend` is 'sentry' and an unhandled error occurs
When the error boundary or global handler catches it
Then the error is sent directly to Sentry/Glitchtip with stack trace, release, and environment

Given `dsn` points to a Glitchtip instance
When an error is captured
Then it appears in Glitchtip — no code change needed versus Sentry SaaS

Given `sendDefaultPii` is False
When an error is sent
Then no user email, IP, or session cookie is in the payload

# --- logBackend: syntek ---

Given `logBackend` is 'syntek' and the user is authenticated
When the useLogger hook emits a log entry at or above `logLevel`
Then it is POSTed to /api/frontend-logs/ with the JWT attached in the Authorization header

Given `logBackend` is 'syntek' and the user is not authenticated
When the useLogger hook emits a log entry
Then it is POSTed to /api/frontend-logs/ without an Authorization header

Given `logBatchSize` is 50 and 50 log entries have accumulated
When the buffer is full
Then all 50 are POSTed in a single request

Given `logFlushInterval` is 10 and fewer than `logBatchSize` entries have accumulated
When 10 seconds elapse
Then the buffer is flushed regardless

# --- logBackend: custom ---

Given `logBackend` is 'custom' and `logEndpoint` is 'https://logs.mycompany.com/ingest'
When the useLogger hook emits a log entry
Then it is POSTed to that endpoint with `logHeaders` applied

# --- logLevel ---

Given `logLevel` is 'warn'
When the useLogger hook emits a 'debug' or 'info' entry
Then it is suppressed — not sent to logBackend

Given `logLevel` is 'warn' and an unhandled error occurs
When the error boundary catches it
Then it is still sent to errorBackend — logLevel does not suppress error tracking

# --- captureConsole ---

Given `captureConsole` is True
When `console.error` is called anywhere in the app
Then the message is sent to errorBackend

# --- ignoreErrors ---

Given `ignoreErrors` includes /ChunkLoadError/
When a ChunkLoadError is thrown
Then it is not sent to errorBackend and does not trigger errorFallback

# --- useLogger hook ---

Given a component calls `const { warn, error } = useLogger('CheckoutPage')`
When `error('Payment failed', { orderId: '123' })` is called
Then the entry is sent to logBackend with 'CheckoutPage' as context
```
