## User Story

As a developer, I want a standalone consent management module so that I can define granular consent
purposes with legal bases, record consent events with full provenance, enforce consent before
processing, and support re-consent when purposes change — independently of any specific regulatory
regime.

## Architecture

`syntek-consent` is purpose-agnostic and regulation-agnostic. It provides the consent infrastructure
that `syntek-gdpr` and other modules build on. The link to specific regulations (GDPR legal basis,
CCPA opt-out) is made at the `syntek-gdpr` layer via `ConsentPurpose.legal_basis` configuration.

## Configuration

```python
SYNTEK_CONSENT = {
    'PURPOSES': [
        {
            'slug': 'essential',
            'display_name': 'Essential',
            'description': 'Required for the service to function.',
            'legal_basis': 'legitimate_interest',  # 'consent' | 'legitimate_interest' | 'contract' | 'legal_obligation' | 'vital_interests' | 'public_task'
            'required': True,           # Cannot be withdrawn
            'default': True,
        },
        {
            'slug': 'analytics',
            'display_name': 'Analytics',
            'description': 'Helps us understand how the service is used.',
            'legal_basis': 'consent',
            'required': False,
            'default': False,
        },
        {
            'slug': 'marketing',
            'display_name': 'Marketing',
            'description': 'Allows us to send personalised marketing.',
            'legal_basis': 'consent',
            'required': False,
            'default': False,
        },
        # Any additional purposes can be added by the consuming project
    ],
    'CONSENT_EXPIRY_DAYS': 365,          # 0 = never expires
    'REQUIRE_EXPLICIT_OPT_IN': True,      # True = no pre-ticked boxes
    'SIGNAL_ON_CHANGE': True,             # Fires syntek-bus event on any consent change
    'AUDIT_CONSENT_EVENTS': True,         # Writes to syntek-audit on every consent change
}
```

## Acceptance Criteria

```gherkin
Scenario 1: Consent record creation for configurable purpose
Given PURPOSES defines analytics with legal_basis = 'consent'
When a user gives consent for analytics
Then a ConsentRecord is created with purpose slug, user FK, given_at timestamp, version, IP hash, and user agent hash

Scenario 2: Consent state verification
Given a user has consented to analytics
When has_consent(user, 'analytics') is called
Then it returns True

Scenario 3: Consent enforcement via decorator
Given a user has not consented to marketing
When require_consent('marketing') decorator is applied to a view
Then the view returns a ConsentRequiredError without executing

Scenario 4: Purpose version invalidation
Given a purpose definition changes (new description or legal basis)
When the version increments
Then existing consent records for that purpose are marked as needs_renewal and has_consent() returns False until renewed

Scenario 5: Consent expiry enforcement
Given CONSENT_EXPIRY_DAYS = 365 and a consent record is older than 365 days
When has_consent() is called
Then it returns False and the user must re-consent

Scenario 6: Consent change event signalling
Given SIGNAL_ON_CHANGE = True and consent is withdrawn for marketing
When the withdrawal is recorded
Then a consent.withdrawn event is fired via syntek-bus with the purpose slug and user ID
```
