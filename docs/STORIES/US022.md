## User Story

As a developer, I want a site search module so that I can index my application's own content —
pages, records, products, posts — and let users search within the site with full-text matching,
facets, and autocomplete suggestions. The search provider (Elasticsearch, Meilisearch, or SQLite
FTS) is configurable via settings with no code changes required.

## Architecture — Site Search, Not Web Crawling

`syntek-search-core` indexes **content that already lives in the application's own database**. It is
a site search layer — equivalent to what Algolia or a WordPress search plugin does — not a web
crawler or general-purpose search engine.

Content is registered for indexing via `SYNTEK_SEARCH['REGISTERED_MODELS']`. When a registered model
instance is saved or deleted, the index is updated asynchronously via Celery. A full rebuild command
re-indexes all registered models in bulk.

```text
Model save / delete signal
  └─→ Celery task: update index document
        └─→ SearchBackend adapter
              └─→ Elasticsearch | OpenSearch | Meilisearch | SQLite FTS
```

The consuming project decides which models are searchable, which fields are indexed, which fields
are facets, and which fields carry extra relevance weight. The frontend queries via the GraphQL
`search` query — no direct search engine access from the frontend.

## Acceptance Criteria

```gherkin
Given a model is registered in SYNTEK_SEARCH['REGISTERED_MODELS']
When an instance is saved or updated
Then the search index document is updated asynchronously within 1 second via Celery

Given a model instance is deleted
When the deletion is committed
Then the corresponding search index document is removed

Given a search query is executed in the context of tenant A
When the search engine runs
Then results from tenant B's index are never returned — tenant isolation is enforced at the
index-naming level

Given a search query includes facet aggregations
When the response is returned
Then facet counts are accurate and match the filtered result set

Given autocomplete is enabled for a text field
When a partial string of 2+ characters is submitted
Then prefix suggestions are returned within 100ms

Given a search result is returned
When the result includes the matched document
Then the response includes a snippet of the matching content with matched terms highlighted

Given SYNTEK_SEARCH['BACKEND'] is set to 'meilisearch'
When the module initialises
Then it connects to the configured Meilisearch instance with no code changes elsewhere

Given SYNTEK_SEARCH['BACKEND'] is set to 'sqlite'
When the module initialises
Then search operates against an in-process SQLite FTS5 database — no external service required

Given a synonym is configured for a term
When a user searches for the original term
Then documents matching the configured synonym are also returned

Given a full rebuild is triggered via the management command
When the rebuild completes
Then all registered model instances are indexed and stale documents removed
```

## Configuration

All behaviour is controlled via `SYNTEK_SEARCH` in Django `settings.py`. Nothing is hardcoded.
Consuming projects configure the backend, registered models, field weights, facets, synonyms, and
stop words.

```python
SYNTEK_SEARCH = {
    # Backend: 'elasticsearch' | 'opensearch' | 'meilisearch' | 'sqlite'
    # sqlite = in-process SQLite FTS5 — no external service; ideal for dev / small sites
    'BACKEND': 'meilisearch',

    # Connection settings — used for elasticsearch / opensearch / meilisearch
    'URL': 'http://localhost:7700',     # Meilisearch default; ES/OS use 9200
    'API_KEY': env('SEARCH_API_KEY', default=''),  # Meilisearch master key or ES API key

    # Index prefix — all index names are prefixed to avoid collision in shared clusters
    'INDEX_PREFIX': 'myapp',

    # Async index update delay in seconds — batches rapid saves before dispatching Celery task
    'INDEX_UPDATE_DELAY': 1,

    # Autocomplete: minimum characters before suggestions are returned
    'AUTOCOMPLETE_MIN_CHARS': 2,

    # Snippet extraction: number of characters to return around matched terms
    'SNIPPET_LENGTH': 200,

    # Models to index — key = index name; value = index configuration
    'REGISTERED_MODELS': {
        'pages': {
            'model': 'myapp.models.Page',
            'fields': ['title', 'body', 'summary'],
            'boosts': {'title': 5.0, 'summary': 2.0, 'body': 1.0},
            'facets': ['category', 'author', 'published_year'],
            'autocomplete_fields': ['title'],
            'filter_queryset': 'myapp.search.published_pages_qs',  # callable(tenant) -> QuerySet
        },
        'products': {
            'model': 'myapp.models.Product',
            'fields': ['name', 'description', 'sku', 'tags'],
            'boosts': {'name': 4.0, 'sku': 3.0, 'tags': 2.0, 'description': 1.0},
            'facets': ['category', 'brand', 'in_stock'],
            'autocomplete_fields': ['name', 'sku'],
        },
    },

    # Synonyms — applied at query time (not index time)
    'SYNONYMS': {
        'tv': ['television', 'telly'],
        'phone': ['mobile', 'smartphone', 'handset'],
    },

    # Stop words — terms excluded from analysis
    'STOP_WORDS': ['the', 'a', 'an', 'and', 'or', 'but'],

    # SQLite FTS database path — only used when BACKEND = 'sqlite'
    'SQLITE_DB_PATH': BASE_DIR / 'search.db',
}
```

## Backend Adapter Protocol

Each backend implements the `SearchBackend` protocol. Consuming projects cannot add backends
directly — the four supported backends cover all deployment sizes:

| Backend         | Best For                                | External Service |
| --------------- | --------------------------------------- | ---------------- |
| `elasticsearch` | Large deployments, complex aggregations | Yes              |
| `opensearch`    | AWS deployments, ES-compatible          | Yes              |
| `meilisearch`   | Simple self-hosting, fast setup         | Yes              |
| `sqlite`        | Development, small sites, no infra      | No               |

`SearchBackend` protocol methods: `index(document)`, `bulk_index(documents)`, `delete(doc_id)`,
`query(query, filters, facets, page, tenant)`, `autocomplete(prefix, field, tenant)`,
`rebuild(model_slug, queryset)`.

All adapters normalise results to the same `SearchResult` dataclass — the GraphQL layer and React UI
never interact with the raw backend response.

## Data Model

### `SearchDocument` (index mirror — optional, configurable)

When `SYNTEK_SEARCH['MIRROR_TO_DB']` is `True`, a lightweight mirror of each indexed document is
stored in PostgreSQL. Used for debugging and for the SQLite backend's source-of-truth rebuild.
Disabled by default.

| Field        | Type      | Notes                                   |
| ------------ | --------- | --------------------------------------- |
| `index_name` | CharField | Corresponds to `REGISTERED_MODELS` key  |
| `doc_id`     | CharField | External index document ID              |
| `object_pk`  | CharField | Django model instance PK                |
| `content`    | JSONField | Indexed field values at last index time |
| `indexed_at` | datetime  |                                         |
| `tenant`     | FK (null) |                                         |
