## User Story

As a developer, I want a TimescaleDB integration module so that I can convert time-series Django
model tables into hypertables using custom migration operations, apply compression and retention
policies declaratively, and query time-bucketed aggregates — without changing any ORM query code,
and with graceful fallback to standard PostgreSQL when TimescaleDB is not installed.

## Architecture

TimescaleDB is a PostgreSQL extension. The Django ORM works completely unchanged — all `QuerySet`
operations, filters, and annotations continue to work on hypertables. What changes is the physical
storage: TimescaleDB automatically partitions the table into time-ordered chunks, making time-range
queries and inserts dramatically faster at scale.

`syntek-timescaledb` provides:

1. **`TimescaleHypertableMixin`** — a Django model mixin that declares a model as a hypertable
   candidate and stores its time column and chunk interval.
2. **Custom migration operations** — `CreateHypertable`, `AddCompressionPolicy`,
   `AddRetentionPolicy`, `CreateContinuousAggregate` — used in Django migration files exactly like
   `CreateModel` or `AddIndex`.
3. **`TimeBucket` Django ORM function** — wraps TimescaleDB's `time_bucket()` SQL function for use
   in `annotate()` calls.
4. **Graceful degradation** — when `SYNTEK_TIMESCALEDB['ENABLED'] = False` or the TimescaleDB
   extension is not present, all migration operations become no-ops and a startup warning is logged.
   Standard PostgreSQL performance applies.

**Primary use case in `syntek-tracking`**

The primary hypertable consumer is `TrackingPoint` in `syntek-tracking` (US116), which forms **Tier
2** of the three-tier tracking architecture. `TrackingPoint` stores sampled GPS pings (configurable
write rate — default 1 per 45 s) with a 90-day retention window. When
`SYNTEK_TRACKING['USE_TIMESCALEDB'] = True`, `CreateHypertable` partitions the table into daily
chunks, `AddCompressionPolicy` compresses chunks older than 7 days, and `AddRetentionPolicy`
replaces the Celery cleanup task. **Tier 3** (`RouteSnapshot`) is a standard PostgreSQL table — it
stores one permanent summary row per session (PostGIS `LineString`), not a time-series of rows, so
it has no hypertable benefit and must not be converted.

Modules that opt in declare `USE_TIMESCALEDB = True` in their own settings (e.g.
`SYNTEK_TRACKING['USE_TIMESCALEDB']`). The consuming project enables the extension once in their
PostgreSQL database and installs `syntek-timescaledb`. All other modules then pick it up through
their own settings flag.

## Configuration

```python
SYNTEK_TIMESCALEDB = {
    # Master enable/disable switch
    # When False, all CreateHypertable migration operations are no-ops (standard PostgreSQL)
    'ENABLED': True,

    # Default chunk time interval for new hypertables
    # TimescaleDB chunks data into this time window per partition
    # '1 day' recommended for high-frequency data (tracking, analytics)
    # '7 days' for lower-frequency data (audit logs, invoices)
    'DEFAULT_CHUNK_TIME_INTERVAL': '1 day',

    # Verify TimescaleDB extension is installed at startup; log a warning if missing
    'VERIFY_EXTENSION_ON_STARTUP': True,
}
```

## Migration Operations

```python
# In a Django migration file:
from syntek_timescaledb.operations import (
    CreateHypertable,
    AddCompressionPolicy,
    AddRetentionPolicy,
    CreateContinuousAggregate,
)

class Migration(migrations.Migration):
    dependencies = [...]

    operations = [
        # Convert table to hypertable on the 'recorded_at' column
        # chunk_time_interval overrides SYNTEK_TIMESCALEDB['DEFAULT_CHUNK_TIME_INTERVAL']
        CreateHypertable(
            model='TrackingPoint',
            time_column='recorded_at',
            chunk_time_interval='1 day',   # optional
        ),

        # Compress chunks older than 7 days to reduce storage
        AddCompressionPolicy(
            model='TrackingPoint',
            compress_after='7 days',
            segment_by=['session_id'],      # optional — columnar compression grouping
        ),

        # Drop chunks (rows) older than 30 days — replaces Celery retention tasks
        AddRetentionPolicy(
            model='TrackingPoint',
            drop_after='30 days',
        ),

        # Continuous aggregate — materialised hourly summaries
        CreateContinuousAggregate(
            name='tracking_point_hourly',
            query="""
                SELECT
                    time_bucket('1 hour', recorded_at) AS bucket,
                    session_id,
                    AVG(speed_kmh) AS avg_speed,
                    COUNT(*) AS point_count
                FROM syntek_tracking_trackingpoint
                GROUP BY bucket, session_id
            """,
            refresh_policy={
                'start_offset': '2 hours',
                'end_offset': '1 minute',
                'schedule_interval': '1 hour',
            },
        ),
    ]
```

## ORM Usage

```python
from syntek_timescaledb.functions import TimeBucket
from django.db.models import Avg, Count
from datetime import timedelta

# Hourly average speed per tracking session
TrackingPoint.objects.annotate(
    bucket=TimeBucket('1 hour', 'recorded_at')
).values('bucket', 'session_id').annotate(
    avg_speed=Avg('speed_kmh'),
    count=Count('id'),
).order_by('bucket')
```

## Acceptance Criteria

```gherkin
Scenario 1: Create hypertable from migration
Given SYNTEK_TIMESCALEDB['ENABLED'] = True and TimescaleDB extension is installed
When a migration containing CreateHypertable(model='TrackingPoint', time_column='recorded_at') runs
Then SELECT * FROM timescaledb_information.hypertables WHERE hypertable_name = 'syntek_tracking_trackingpoint' returns one row

Scenario 2: Chunk exclusion on time-range query
Given a hypertable exists for TrackingPoint
When 10,000 TrackingPoint records are inserted and a time-range query is executed
Then the query uses chunk exclusion (EXPLAIN shows only relevant chunks scanned, not the full table)

Scenario 3: Compression policy scheduling
Given AddCompressionPolicy(model='TrackingPoint', compress_after='7 days') is applied
When chunks older than 7 days are eligible
Then SELECT * FROM timescaledb_information.jobs WHERE application_name LIKE '%Compression%' returns a scheduled job for that table

Scenario 4: Retention policy automatic cleanup
Given AddRetentionPolicy(model='TrackingPoint', drop_after='30 days') is applied
When the TimescaleDB background worker runs
Then chunks entirely outside the retention window are dropped automatically — no Celery task required

Scenario 5: Graceful fallback when disabled
Given SYNTEK_TIMESCALEDB['ENABLED'] = False
When a migration containing CreateHypertable runs
Then the migration completes without error and the table remains a standard PostgreSQL table

Scenario 6: Extension check on startup
Given SYNTEK_TIMESCALEDB['VERIFY_EXTENSION_ON_STARTUP'] = True and TimescaleDB is not installed
When Django starts
Then a startup warning is logged: "TimescaleDB extension not found — falling back to standard PostgreSQL" and no exception is raised

Scenario 7: TimeBucket annotation
Given a TimeBucket('1 hour', 'recorded_at') annotation is used in a QuerySet
When the queryset executes
Then the generated SQL contains time_bucket('1 hour', recorded_at) and results are grouped into hourly buckets
```
