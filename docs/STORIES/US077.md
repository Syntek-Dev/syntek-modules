## User Story

As a developer, I want a caching module so that all Syntek backend modules share a consistent,
configurable Redis/Valkey cache layer — with connection management, key namespacing, cache
invalidation helpers, and optional cache warming — without each module managing its own Redis
connection.

**Must Have**

## Configuration Contract

All behaviour is controlled via `SYNTEK_CACHE` in `settings.py`. Validated at startup in
`AppConfig.ready()`. The module configures Django's `CACHES` setting automatically — consuming
modules do not set `CACHES` directly.

```python
SYNTEK_CACHE = {
    # --- Backend ---
    'BACKEND': 'valkey',              # 'valkey' | 'redis'
                                      # Valkey is the default — Redis-compatible fork maintained by the Linux Foundation.
                                      # Both are supported; the connection library (valkey-py / redis-py) is selected automatically.
    'URL': env('CACHE_URL'),          # str — connection URL (e.g. 'redis://localhost:6379/0' or 'valkey://...')
                                      #        Required — no default.

    # --- Connection pool ---
    'MAX_CONNECTIONS': 20,            # int — connection pool size
    'SOCKET_CONNECT_TIMEOUT': 5,      # int (seconds) — connection attempt timeout
    'SOCKET_TIMEOUT': 5,              # int (seconds) — read/write timeout

    # --- Cache defaults ---
    'DEFAULT_TIMEOUT': 300,           # int (seconds) — 0 = never expire; None = Django default (300s)
    'KEY_PREFIX': '',                 # str — namespace prefix for all cache keys (e.g. 'myapp')
                                      #        Prevents key collisions when multiple apps share one Redis instance.

    # --- Serialisation ---
    'SERIALIZER': 'json',             # 'json' | 'msgpack'
                                      # 'json'    — human-readable, slower, widely compatible
                                      # 'msgpack' — binary, faster, smaller; requires msgpack package

    # --- Compression ---
    'COMPRESSION': False,             # bool — gzip-compress values larger than COMPRESSION_THRESHOLD
    'COMPRESSION_THRESHOLD': 1024,    # int (bytes) — minimum value size to trigger compression

    # --- Cache warming ---
    'WARMUP_ON_STARTUP': False,       # bool — populate specified keys on Django startup
    'WARMUP_KEYS': [],                # list[str | callable] — cache keys or zero-argument callables
                                      #   returning (key, value, timeout) to warm on startup
}
```

**Key rules:**

- `BACKEND: 'valkey'` is the default and recommended option. Valkey is wire-compatible with Redis.
- `CACHE_URL` is always required — there is no in-process fallback. An unconfigured URL raises
  `ImproperlyConfigured`.
- `KEY_PREFIX` is applied to all keys written by any module using this cache — choose a value that
  is unique per deployment environment (e.g. `'prod'`, `'staging'`).
- `WARMUP_KEYS` entries that are callables are called at startup and must return
  `(key, value, timeout)`.
- Consuming modules (e.g. `syntek-security` for rate limiting, `syntek-auth` for lockout counters)
  obtain a Django cache instance via `django.core.cache.cache` — they do not manage their own Redis
  connections.

## Acceptance Criteria

```gherkin
# --- Settings validation ---

Given `SYNTEK_CACHE` is missing the 'URL' key
When `AppConfig.ready()` runs
Then `ImproperlyConfigured` is raised — CACHE_URL is required

Given `BACKEND` is 'valkey' and the valkey-py package is not installed
When `AppConfig.ready()` runs
Then `ImproperlyConfigured` is raised naming the missing package

Given `SERIALIZER` is not one of 'json' or 'msgpack'
When `AppConfig.ready()` runs
Then `ImproperlyConfigured` is raised

# --- Django cache wiring ---

Given `SYNTEK_CACHE` is correctly configured
When `AppConfig.ready()` runs
Then Django's `CACHES['default']` is set to the syntek-cache backend automatically — consuming
modules use `django.core.cache.cache` with no additional setup

# --- Backends ---

Given `BACKEND` is 'valkey'
When a cache write and read are performed
Then the value is stored and retrieved via Valkey using valkey-py

Given `BACKEND` is 'redis'
When a cache write and read are performed
Then the value is stored and retrieved via Redis using redis-py

# --- Key prefix ---

Given `KEY_PREFIX` is 'prod'
When any module writes to the cache with key 'user:123:lockout'
Then the actual Redis/Valkey key is 'prod:user:123:lockout'

# --- Serialisation ---

Given `SERIALIZER` is 'msgpack'
When a Python dict is written to the cache
Then the stored bytes are msgpack-encoded and correctly decoded on read

# --- Compression ---

Given `COMPRESSION` is True and `COMPRESSION_THRESHOLD` is 1024
When a value larger than 1024 bytes is written to the cache
Then it is gzip-compressed before storage and transparently decompressed on read

Given `COMPRESSION` is True and a value smaller than 1024 bytes is written
When the value is stored
Then it is not compressed — compression overhead on small values is avoided

# --- Cache invalidation helpers ---

Given a module calls `cache_invalidate_prefix('user:123')`
When the helper runs
Then all keys matching the prefix pattern are deleted from the cache in a single operation

# --- Cache warming ---

Given `WARMUP_ON_STARTUP` is True and `WARMUP_KEYS` contains a callable
When the Django application starts
Then the callable is called, its return value is stored in the cache under the specified key and timeout

Given `WARMUP_ON_STARTUP` is False
When the Django application starts
Then no cache warming occurs — `WARMUP_KEYS` is ignored

# --- Connection failure ---

Given the Redis/Valkey server is unreachable at startup
When `AppConfig.ready()` runs
Then a warning is logged via syntek-logging and startup continues — cache is not a hard dependency
(individual cache reads/writes will raise errors at runtime if the server remains unreachable)
```
