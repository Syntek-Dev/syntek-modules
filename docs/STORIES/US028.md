## User Story

As a developer, I want a report definition and query engine so that I can define role-gated reports
driven by configurable aggregate DB queries, cache results in Redis, and schedule report generation
via Celery Beat — with all export logic handled separately by `syntek-reporting-export`.

## Acceptance Criteria

```gherkin
Given a report definition specifies a `manager` role requirement
When a user without that role requests the report
Then a 403 is returned and no query is executed

Given a report definition specifies parameters (date_from, date_to, group_by)
When a user runs the report with valid parameter values
Then the query builder applies the parameters and returns correct aggregate results

Given a report query runs for the second time within the cache TTL
When the same parameter hash is requested
Then results are served from Redis without hitting the database

Given a report is scheduled for Monday at 08:00 UTC via a cron expression
When Celery Beat triggers the report task
Then the report is generated, persisted as a ReportRun, and a signal is emitted for downstream
consumers (e.g. syntek-reporting-export)

Given a report definition uses a multi-step aggregation pipeline
When the pipeline runs
Then intermediate results are composable and the final output matches manually verified test data

Given a report definition is updated
When the new definition is saved
Then a ReportVersion is created; existing ReportRun records reference the version in use at
generation time
```

## Configuration

```python
SYNTEK_REPORTING = {
    # --- Query engine ---
    'REPORT_REGISTRY': {},         # Dict[str, type[BaseReport]] — explicit registry; auto-discover if empty
    'QUERY_TIMEOUT_SECONDS': 30,   # Max wall-clock time for a single report query

    # --- Caching ---

    'CACHE_TTL_SECONDS': 300,      # Default cache TTL; overridable per report definition

    # --- Scheduling ---
    'SCHEDULER': 'celery_beat',    # 'celery_beat' | 'none'
    'SCHEDULE_TIMEZONE': 'UTC',    # Timezone for all cron expressions

    # --- Access control ---
    'REQUIRE_AUTHENTICATION': True,
    'DEFAULT_PERMISSION': '',      # Django permission string; empty = any authenticated user

    # --- Tenancy ---
    'TENANT_SCOPED': True,         # Isolate report definitions and runs per tenant

    # --- Integration ---
```

### Report Definition Schema

Report definitions are stored as tenant-scoped versioned JSON in the `ReportDefinition` model. Each
definition declares:

| Field            | Type          | Description                                                        |
| ---------------- | ------------- | ------------------------------------------------------------------ |
| `id`             | `str`         | Unique slug                                                        |
| `title`          | `str`         | Display name                                                       |
| `description`    | `str`         | Optional — shown in UI                                             |
| `required_roles` | `list[str]`   | Django permission strings; empty = any authenticated user          |
| `parameters`     | `list[Param]` | Runtime parameters (date pickers, selects, text)                   |
| `pipeline`       | `list[Step]`  | Ordered query steps; each step references a Django queryset method |
| `output_fields`  | `list[Field]` | Column spec for the tabular output — name, label, format, type     |
| `cache_ttl`      | `int`         | Per-definition TTL override (seconds); `null` uses global default  |
| `version`        | `int`         | Auto-incremented on save                                           |

### Parameter Types

| Type        | Widget hint             |
| ----------- | ----------------------- |
| `date`      | Date picker             |
| `daterange` | Start + end date picker |
| `select`    | Dropdown from choices   |
| `text`      | Free text               |
| `integer`   | Numeric input           |
| `boolean`   | Toggle                  |

### Aggregate Pipeline Steps

Each step in the pipeline is a named DSL instruction applied to a Django queryset:

| Step type    | Description                                                    |
| ------------ | -------------------------------------------------------------- |
| `filter`     | Apply `Q()` conditions, optionally using parameter values      |
| `annotate`   | Add computed fields — `Count`, `Sum`, `Avg`, `Max`, `Min`      |
| `date_trunc` | Group by date granularity — `day`, `week`, `month`, `year`     |
| `group_by`   | `.values(*fields)` grouping                                    |
| `order_by`   | Sort on computed or raw fields                                 |
| `limit`      | Cap result set; defaults to 10,000 rows                        |
| `subquery`   | Reference output of a previous named step as a subquery source |

Steps are composable — step N can reference the queryset produced by step N−1 by name.

## Models

| Model              | Description                                                                  |
| ------------------ | ---------------------------------------------------------------------------- |
| `ReportDefinition` | Stores the JSON schema for a report, tenant-scoped; soft-deletable           |
| `ReportVersion`    | Immutable snapshot of `ReportDefinition.schema` at each save                 |
| `ReportParameter`  | Denormalised parameter spec for UI discovery (foreign key to definition)     |
| `ReportRun`        | One execution of a report — links to version, parameters used, status, owner |
| `ReportRunResult`  | JSONB column storing paginated result rows; linked to `ReportRun`            |
| `ScheduledReport`  | Celery Beat entry — cron expression, destination type, recipient list        |

`ReportVersion` records are immutable once created. `ReportRun` always references the version in use
at generation time, so historical runs remain reproducible.

## GraphQL API

### Queries

| Query                                 | Returns                                           |
| ------------------------------------- | ------------------------------------------------- |
| `reports(tenantId)`                   | `[ReportDefinition]` — all visible to caller      |
| `report(id)`                          | `ReportDefinition` — including parameter spec     |
| `runReport(id, parameters)`           | `ReportRun` — initiates async run, returns run ID |
| `reportStatus(runId)`                 | `ReportRun` — current status and progress         |
| `reportResult(runId, page, pageSize)` | `ReportRunResult` — paginated row data            |
| `scheduledReports(tenantId)`          | `[ScheduledReport]`                               |

### Mutations

| Mutation                           | Description                                       |
| ---------------------------------- | ------------------------------------------------- |
| `createReport(input)`              | Create a new `ReportDefinition`                   |
| `updateReport(id, input)`          | Update definition — creates a new `ReportVersion` |
| `deleteReport(id)`                 | Soft-delete                                       |
| `createScheduledReport(input)`     | Create a Celery Beat entry                        |
| `updateScheduledReport(id, input)` | Update schedule or recipients                     |
| `deleteScheduledReport(id)`        | Remove Celery Beat entry                          |
