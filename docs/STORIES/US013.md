## User Story

As a compliance officer, I want an immutable audit log so that all significant actions are recorded
with actor, resource, action, timestamp, IP, and an HMAC chain that detects tampering — with
configurable per-category retention, model auto-tracking, and optional Loki shipping alongside the
append-only database record.

## Configuration Contract

All behaviour is controlled via `SYNTEK_AUDIT` in `settings.py`. Validated at startup in
`AppConfig.ready()`.

```python
SYNTEK_AUDIT = {
    # --- HMAC integrity chain ---
    'HMAC_KEY': env('SYNTEK_AUDIT_HMAC_KEY'),   # str — required; 32+ byte secret for HMAC-SHA256 chain signing.
                                                  # Rotate via key versioning — old entries remain verifiable.

    # --- Retention ---
    # After retention_days, PII fields are anonymised and the entry record is retained for chain integrity.
    # A retention_days of 0 means keep indefinitely (PII never anonymised).
    'RETENTION_DAYS': 730,                        # int — global default (2 years)
    'RETENTION_CATEGORIES': {                     # dict[str, int] — per-action-category overrides
        'login':       365,                       # 1 year for login events
        'data_change': 730,                       # 2 years for data mutations
        'admin':       2555,                      # 7 years for admin actions (financial compliance)
        'gdpr':        2555,                      # 7 years for GDPR-related actions
    },

    # --- Auto-tracking ---
    'TRACK_MUTATIONS': True,                      # bool — automatically record all Strawberry GraphQL mutations
    'EXCLUDE_MUTATIONS': [                        # list[str] — mutation names to exclude from auto-tracking
        'refreshToken',
        'introspect',
        'heartbeat',
    ],
    'TRACK_MODELS': [],                           # list[str] — model dotted paths auto-tracked on save/delete
                                                  # e.g. ['syntek_auth.User', 'myapp.Order']
                                                  # Empty list = no model-level auto-tracking (use @audit_action decorator manually).

    # --- Storage ---
    'STORAGE_BACKEND': 'db',                      # 'db' | 'loki' | 'both'
                                                  # 'db'   — AuditEntry table (append-only, HMAC-chained, queryable)
                                                  # 'loki' — ship to Loki only (no DB table; no queryability via admin)
                                                  # 'both' — write to DB and ship to Loki simultaneously
    'LOKI_INHERIT': True,                         # bool — inherit Loki connection config from SYNTEK_LOGGING
                                                  #        (requires syntek-logging with 'loki' backend configured)
                                                  #        If False, configure LOKI_URL and LOKI_AUTH directly:
    'LOKI_URL': env('AUDIT_LOKI_URL', default=''),   # str — used only when LOKI_INHERIT is False
    'LOKI_AUTH': env('AUDIT_LOKI_AUTH', default=''), # str — used only when LOKI_INHERIT is False
    'LOKI_LABELS': {'app': 'syntek', 'stream': 'audit'},  # dict — additional labels for audit log stream
}
```

**Key rules:**

- `HMAC_KEY` is required and must be at least 32 bytes — shorter keys raise `ImproperlyConfigured`.
- `RETENTION_CATEGORIES` overrides `RETENTION_DAYS` for matching action categories.
- `STORAGE_BACKEND: 'loki'` does not create the `AuditEntry` table — the admin audit log query UI is
  unavailable.
- `STORAGE_BACKEND: 'both'` is recommended for deployments with Loki — DB provides queryability,
  Loki provides centralised log aggregation.
- `LOKI_INHERIT: True` requires `syntek-logging` with `'loki'` in `BACKENDS` — validated at startup.
- The database user used by the application must not have `UPDATE` or `DELETE` on the `AuditEntry`
  table — enforced via a migration that revokes those privileges.

## Acceptance Criteria

```gherkin
# --- Settings validation ---

Given `HMAC_KEY` is missing from `SYNTEK_AUDIT`
When `AppConfig.ready()` runs
Then `ImproperlyConfigured` is raised — HMAC_KEY is required

Given `HMAC_KEY` is fewer than 32 bytes
When `AppConfig.ready()` runs
Then `ImproperlyConfigured` is raised — HMAC_KEY must be at least 32 bytes

Given `STORAGE_BACKEND` is 'loki' or 'both' and `LOKI_INHERIT` is False and `LOKI_URL` is empty
When `AppConfig.ready()` runs
Then `ImproperlyConfigured` is raised — LOKI_URL is required when not inheriting from SYNTEK_LOGGING

Given `LOKI_INHERIT` is True and syntek-logging is not configured with the 'loki' backend
When `AppConfig.ready()` runs
Then `ImproperlyConfigured` is raised — LOKI_INHERIT requires syntek-logging with loki backend

# --- Audit entry creation ---

Given any significant action occurs (login, data change, admin action)
When the audit middleware records it
Then an `AuditEntry` is written with: actor_id, resource_type, resource_id, action, ip, user_agent,
timestamp, tenant_id, and hmac_hash

# --- HMAC chain integrity ---

Given audit entries are written sequentially
When each entry is written
Then its HMAC is computed over (previous_entry_hmac || entry_data) so any tampering with any entry
breaks the chain from that point onwards

Given an audit entry is written
When the batch integrity check management command runs
Then any entry whose HMAC does not match the expected chain value is flagged and logged as tampered

# --- Database immutability ---

Given the application database user attempts to UPDATE an audit entry
When the SQL executes
Then the operation is rejected by a database-level permission denial

Given the application database user attempts to DELETE an audit entry
When the SQL executes
Then the operation is rejected by a database-level permission denial

# --- Auto-tracking mutations ---

Given `TRACK_MUTATIONS` is True and a Strawberry mutation runs
When the mutation completes
Then an AuditEntry is written automatically — no per-resolver boilerplate required

Given `EXCLUDE_MUTATIONS` includes 'refreshToken'
When the refreshToken mutation runs
Then no AuditEntry is written

# --- Model auto-tracking ---

Given `TRACK_MODELS` includes 'syntek_auth.User'
When a User instance is saved with changed fields
Then an AuditEntry is written with action='data_change', resource_type='User', and the changed field names

Given `TRACK_MODELS` is empty
When any model is saved
Then no AuditEntry is written by model tracking (manual @audit_action decorator calls still work)

# --- Retention and anonymisation ---

Given `RETENTION_CATEGORIES` sets 'login' retention to 365 days
When a login AuditEntry is older than 365 days and the retention task runs
Then PII fields (ip, user_agent, actor_id display name) are replaced with '[ANONYMISED]'
and the entry record and its HMAC are retained for chain integrity

Given `RETENTION_DAYS` is 730 and an entry has no matching category override
When the entry is older than 730 days and the retention task runs
Then the global retention policy applies

# --- Storage backends ---

Given `STORAGE_BACKEND` is 'db'
When an AuditEntry is written
Then it is stored in the database table only; no Loki push occurs

Given `STORAGE_BACKEND` is 'both'
When an AuditEntry is written
Then it is stored in the database and simultaneously pushed to Loki with the configured labels

Given `STORAGE_BACKEND` is 'loki'
When an AuditEntry is written
Then it is pushed to Loki only; no database row is created

# --- GDPR retention ---

Given a GDPR erasure request is processed for a user
When the erasure task runs
Then all AuditEntry records where actor_id matches the user have PII fields anonymised immediately
regardless of retention period
```
