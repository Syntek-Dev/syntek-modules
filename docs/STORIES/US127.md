# US127 — `syntek-ai` — AI Conversation Backend

**Last Updated**: 10/03/2026 **Version**: 0.1.0 **Status**: To Do **Story Points**: 8

**Epic**: Epic 3 — Core Backend Modules

## User Story

**As a** developer

**I want** an AI conversation backend module

**So that** I can store chat sessions and message history, stream AI responses from a configurable
provider, apply configurable system prompts, and expose conversation history via GraphQL — without
the consuming project implementing its own AI plumbing.

---

- **Must Have:**

## Architecture

### Core Design

`syntek-ai` stores conversation sessions and messages in Django models. AI provider communication is
abstracted behind an `AIProvider` protocol — consuming projects configure their provider and API key
in `SYNTEK_AI` settings. The module does not ship provider-specific logic directly; it ships
adapters for OpenAI-compatible APIs (which includes OpenAI, Anthropic via compatibility layer,
Ollama, and any OpenAI-compatible endpoint).

Responses are streamed to the frontend via Server-Sent Events (SSE) or WebSocket (Django Channels),
configurable per project. System prompts are configurable via `SYNTEK_AI['DEFAULT_SYSTEM_PROMPT']`;
consuming projects override this per session. All Syntek clients start from prompt templates in the
`syntek-ai` repository (a separate repository of YAML bot definitions and markdown prompts — not
executable code).

Message history is stored encrypted (via `syntek-pyo3` field encryption). Attachments (images,
files) are stored via `syntek-media` or `syntek-documents` depending on type.

### Configuration

```python
SYNTEK_AI = {
    # Provider — OpenAI-compatible endpoint URL + API key
    # Works with OpenAI, Anthropic (via openai-python compatibility), Ollama, any OpenAI-compatible API
    'PROVIDER_BASE_URL': env('SYNTEK_AI_PROVIDER_BASE_URL', default='https://api.openai.com/v1'),
    'API_KEY': env('SYNTEK_AI_API_KEY'),
    'DEFAULT_MODEL': env('SYNTEK_AI_DEFAULT_MODEL', default='gpt-4o'),

    # System prompt — applied to every new session unless overridden
    'DEFAULT_SYSTEM_PROMPT': env('SYNTEK_AI_DEFAULT_SYSTEM_PROMPT', default='You are a helpful assistant.'),

    # Streaming transport — how AI responses reach the frontend
    'STREAM_TRANSPORT': 'sse',   # 'sse' | 'websocket'
    'CHANNEL_LAYER': 'default',  # Used when STREAM_TRANSPORT = 'websocket'

    # Message retention — how long conversation history is kept
    'MESSAGE_RETENTION_DAYS': 90,

    # Attachments — where uploaded files in chat are stored
    'ATTACHMENT_BACKEND': 'media',   # 'media' (syntek-media/Cloudinary) | 'documents' (syntek-documents/MinIO)

    # Per-tenant scoping
    'TENANT_SCOPED': True,

    # Rate limiting — max messages per user per hour (0 = unlimited)
    'RATE_LIMIT_MESSAGES_PER_HOUR': 0,
}
```

### Data Models

**ConversationSession**

- `id` — ULID primary key
- `user` — ForeignKey to User (from `syntek-users`)
- `tenant` — ForeignKey to Tenant (from `syntek-tenancy`)
- `system_prompt` — encrypted TextField (via `syntek-pyo3` field encryption)
- `status` — CharField ('active', 'archived')
- `model_override` — optional CharField (allows per-session model override)
- `started_at` — DateTimeField (auto_now_add)
- `archived_at` — DateTimeField (nullable)
- `metadata` — JSONField (for custom app-specific session data)

**ConversationMessage**

- `id` — ULID primary key
- `session` — ForeignKey to ConversationSession
- `role` — CharField ('user', 'assistant', 'system')
- `content` — encrypted TextField (via `syntek-pyo3` field encryption)
- `attachment_url` — CharField (nullable, reference to media/document URL)
- `attachment_type` — CharField (nullable, 'image', 'document', 'file')
- `created_at` — DateTimeField (auto_now_add)
- `token_count` — IntegerField (for usage tracking, estimated)

### AI Provider Protocol

```python
from typing import AsyncIterator
from abc import ABC, abstractmethod

class AIProvider(ABC):
    """Abstract protocol for AI providers."""

    @abstractmethod
    async def stream_chat(
        self,
        messages: list[dict[str, str]],
        model: str,
        system_prompt: str,
    ) -> AsyncIterator[str]:
        """
        Stream chat response from provider.

        Args:
            messages: List of {role, content} dicts from conversation history
            model: Model identifier (e.g., 'gpt-4o')
            system_prompt: System prompt to inject

        Yields:
            Text chunks of the response

        Raises:
            AIProviderError: If API call fails
        """
        pass


class OpenAICompatibleProvider(AIProvider):
    """OpenAI-compatible provider adapter."""

    def __init__(self, base_url: str, api_key: str):
        self.base_url = base_url
        self.api_key = api_key

    async def stream_chat(
        self,
        messages: list[dict[str, str]],
        model: str,
        system_prompt: str,
    ) -> AsyncIterator[str]:
        # Implementation: HTTP POST to base_url/chat/completions
        # Stream server-sent events or text/event-stream response
        pass
```

### Streaming Transport

**SSE Transport**

- Endpoint: `GET /api/ai/conversations/{session_id}/stream/`
- Content-Type: `text/event-stream`
- Each chunk arrives as `data: <token>\n\n`
- Client subscribes with `EventSource`

**WebSocket Transport**

- Consumer: `ai_chat_consumer` (Django Channels)
- Message format: `{type: 'chat.message', content: '<token>'}`
- Requires `CHANNEL_LAYER` configuration

## Acceptance Criteria

### Scenario 1: Session Creation

**Given** a user initiates a conversation

**When** `startConversation(systemPrompt?)` mutation is called

**Then** a `ConversationSession` is created with:

- Unique ULID identifier
- Configured default system prompt (or override if provided)
- Status set to 'active'
- User FK populated
- Tenant FK populated (from request context)
- `started_at` timestamp set

**And** the response includes the session ID for subsequent messages

---

### Scenario 2: Message Streaming via SSE

**Given** `STREAM_TRANSPORT = 'sse'`

**When** `sendMessage(sessionId, content)` mutation is called

**Then** the system:

- Validates the request (authentication, rate limits)
- Queues the AI provider call (non-blocking)
- Returns an SSE subscription URL immediately
- Opens an SSE stream to `/api/ai/conversations/{session_id}/stream/`
- Streams response chunks from the AI provider as `data: <token>\n\n` events
- Each chunk arrives within 500ms of generation

**And** the client receives all chunks in order

---

### Scenario 3: Message Storage

**Given** a message exchange is complete

**When** the SSE stream ends

**Then** the system:

- Stores the user message as a `ConversationMessage` (role='user', content encrypted)
- Stores the full assistant response as a `ConversationMessage` (role='assistant', content
  encrypted)
- Calculates approximate token count for both messages
- Records `created_at` timestamp for both

**And** both messages are queryable via `conversationHistory(sessionId)`

---

### Scenario 4: Conversation History Retrieval

**Given** a session has previous messages

**When** `conversationHistory(sessionId)` GraphQL query is called

**Then** the system returns:

- All `ConversationMessage` records for the session
- In chronological order (oldest first)
- With role and decrypted content
- With attachment metadata (if present)
- Paginated (configurable limit per request)

**And** only the session owner or an admin can view the history (permission check)

---

### Scenario 5: Attachment in Message

**Given** `ATTACHMENT_BACKEND = 'media'`

**When** `sendMessage` is called with an attached image

**Then** the system:

- Uploads the image via `syntek-media` (Cloudinary)
- Stores the Cloudinary URL in the message record's `attachment_url` field
- Sets `attachment_type = 'image'`
- References the attachment in the message content or separately

**And** the attachment URL is included in `conversationHistory` response

---

### Scenario 6: Message Retention Cleanup

**Given** `MESSAGE_RETENTION_DAYS = 90`

**When** the message retention Celery Beat task runs (e.g., daily)

**Then** the system:

- Queries all `ConversationMessage` records older than 90 days
- Deletes them (soft or hard delete per config)
- Logs the cleanup action via `syntek-logging`

**And** older messages are no longer accessible via queries

---

### Scenario 7: Rate Limiting

**Given** `RATE_LIMIT_MESSAGES_PER_HOUR = 60`

**When** a user sends their 61st message within an hour

**Then** the system:

- Returns a `RateLimitExceeded` error (429 HTTP status)
- Does NOT call the AI provider
- Includes a retry-after header or message

**And** the error is logged via `syntek-logging` for monitoring

## Notes

- Provider selection happens at consume time via `SYNTEK_AI['PROVIDER_BASE_URL']` and
  `SYNTEK_AI['API_KEY']` — no hardcoded provider logic
- Message encryption via `syntek-pyo3` is mandatory (no plaintext storage)
- System prompt injection is a critical security boundary — validate that user overrides cannot
  modify provider instructions
- Attachment storage is optional — consuming projects choose their backend via `ATTACHMENT_BACKEND`
  setting
- Message retention cleanup is non-destructive by default; consuming projects can configure hard vs.
  soft delete
- The `syntek-ai` repository (separate from syntek-modules) will contain YAML bot definitions and
  markdown prompt templates; this module does not include them
