## User Story

As a developer, I want structured JSON logging with request ID propagation so that logs from any
layer of the stack can be correlated and shipped to whichever observability backends the operator
chooses — and a frontend log ingestion endpoint so that browser and mobile logs can be enriched with
verified server-side context (tenant, user) and forwarded through the same pipeline.

## MoSCoW Priority

**Must Have**

## Architecture: Hybrid Logging Model

The preferred logging model separates error tracking from structured log shipping:

```text
Browser / Mobile
  ├── Crashes & errors  →  Sentry / Glitchtip DIRECT (errorBackend)
  │                        Reliable — does not depend on your server being up.
  │
  └── Structured logs   →  POST /api/frontend-logs/  →  Django (syntek-logging)
                           Django enriches with tenant_id + verified user_id,
                           then forwards to all configured SYNTEK_LOGGING backends
                           (Loki, Sentry, stdout, file).

Django server
  └── All server logs   →  SYNTEK_LOGGING backends (Loki, Sentry, stdout, file)
```

This model gives a single unified log store (Loki / Grafana) for structured logs, while keeping
crash reporting on a direct path that survives server downtime.

Other developers who do not use the Syntek infrastructure can:

- Set `logBackend="custom"` to point the frontend at their own endpoint (ELK, Datadog, etc.)
- Set `logBackend="none"` to disable structured log shipping entirely
- Set `errorBackend="sentry"` with any Sentry-compatible DSN (Sentry SaaS, Glitchtip, etc.)

> **Glitchtip**: Self-hosted, Sentry-compatible error tracking. Uses the same Sentry SDK. Point the
> DSN at your Glitchtip instance — no other changes required.
>
> **Grafana**: A visualisation layer, not a log ingestion target. Ship to Loki; view in Grafana.

## Configuration Contract

```python
SYNTEK_LOGGING = {
    # --- Log level ---
    'LOG_LEVEL': 'info',               # 'debug' | 'info' | 'warning' | 'error' | 'critical'
    'DEV_FORMAT': 'console',           # 'console' (coloured human-readable) | 'json'
                                       # Applied only when DEBUG=True; production always uses JSON.

    # --- Backends ---
    # Multiple backends can be active simultaneously.
    # 'stdout'  — JSON to stdout; collected by a log agent (Promtail, Alloy) and shipped to Loki.
    # 'file'    — rotating JSON file on disk.
    # 'sentry'  — Sentry SaaS or Glitchtip (self-hosted, Sentry-compatible). Same SDK for both.
    # 'loki'    — direct HTTP push to the Loki Push API.
    'BACKENDS': ['stdout'],            # list[str] — at least one required.

    # --- Sentry / Glitchtip (required when 'sentry' is in BACKENDS) ---
    'SENTRY_DSN': env('SENTRY_DSN', default=''),
    'SENTRY_TRACES_SAMPLE_RATE': 0.1,
    'SENTRY_PROFILES_SAMPLE_RATE': 0.0,
    'SENTRY_ENVIRONMENT': env('ENVIRONMENT', default='production'),
    'SENTRY_RELEASE': env('RELEASE', default=''),
    'SENTRY_SEND_DEFAULT_PII': False,

    # --- Loki (required when 'loki' is in BACKENDS) ---
    'LOKI_URL': env('LOKI_URL', default=''),
    'LOKI_AUTH': env('LOKI_AUTH', default=''),
    'LOKI_LABELS': {'app': 'syntek'},
    'LOKI_BATCH_SIZE': 100,
    'LOKI_FLUSH_INTERVAL': 5,          # int (seconds)

    # --- File (required when 'file' is in BACKENDS) ---
    'LOG_FILE': '/var/log/syntek/app.log',
    'LOG_FILE_MAX_BYTES': 10_485_760,
    'LOG_FILE_BACKUP_COUNT': 5,

    # --- Request / response middleware ---
    'LOG_REQUESTS': True,
    'LOG_REQUEST_BODY': False,
    'LOG_RESPONSE_BODY': False,
    'LOG_SLOW_REQUESTS_MS': 500,       # int (ms) — 0 = disabled

    # --- Sensitive field redaction ---
    'REDACT_FIELDS': [
        'password', 'token', 'secret', 'api_key', 'card_number', 'cvv', 'ssn',
        'authorization', 'cookie', 'session',
    ],

    # --- Request ID ---
    'REQUEST_ID_HEADER': 'X-Request-ID',

    # --- Frontend log ingestion endpoint ---
    # Receives structured log batches from @syntek/ui-logging and @syntek/mobile-logging.
    # Validates the JWT, enriches with server-side context, forwards to all BACKENDS.
    'FRONTEND_LOGS_ENABLED': True,      # bool — enable POST /api/frontend-logs/ endpoint
    'FRONTEND_LOGS_RATE_LIMIT': '500/minute',  # str — per-user rate limit on the endpoint
                                                #        Uses syntek-cache (requires US077).
    'FRONTEND_LOGS_MAX_LEVEL': 'info',  # str — reject frontend log entries above this level
                                        #        Prevents debug spam flooding production logs.
                                        #        'debug' | 'info' | 'warning' | 'error'
    'FRONTEND_LOGS_REQUIRE_AUTH': True, # bool — False allows unauthenticated log submission
                                        #        (useful for pre-login error capture; set carefully)
}
```

**Key rules:**

- `BACKENDS` must contain at least one entry.
- `'sentry'` requires `SENTRY_DSN`. `'loki'` requires `LOKI_URL`. `'file'` requires `LOG_FILE`.
- `FRONTEND_LOGS_ENABLED: True` requires syntek-cache (US077) for rate limiting — validated at
  startup.
- `FRONTEND_LOGS_MAX_LEVEL` acts as a server-side gate: a frontend posting `debug` logs to a
  production server configured with `'info'` has those entries silently dropped.
- All frontend log entries go through the same `REDACT_FIELDS` filter as server logs.
- The `/api/frontend-logs/` endpoint is REST (not GraphQL) — GraphQL adds overhead for log batches
  and creates a circular dependency when logging GraphQL errors.

## Acceptance Criteria

```gherkin
# --- Settings validation ---

Given `BACKENDS` is an empty list
When `AppConfig.ready()` runs
Then `ImproperlyConfigured` is raised

Given `BACKENDS` includes 'sentry' and `SENTRY_DSN` is empty
When `AppConfig.ready()` runs
Then `ImproperlyConfigured` is raised

Given `BACKENDS` includes 'loki' and `LOKI_URL` is empty
When `AppConfig.ready()` runs
Then `ImproperlyConfigured` is raised

Given `FRONTEND_LOGS_ENABLED` is True and syntek-cache is not installed
When `AppConfig.ready()` runs
Then `ImproperlyConfigured` is raised — frontend log endpoint rate limiting requires syntek-cache

# --- Multi-backend ---

Given `BACKENDS` is ['stdout', 'sentry', 'loki']
When an error is logged
Then it is emitted to all three backends simultaneously

# --- Sentry / Glitchtip ---

Given `SENTRY_DSN` points to a Glitchtip instance
When an exception is raised
Then it appears in Glitchtip with no code change versus Sentry SaaS

Given `SENTRY_SEND_DEFAULT_PII` is False
When any log is sent to Sentry
Then no PII fields are present in the event payload

# --- Stdout / JSON ---

Given `DEBUG` is False
When any log statement runs
Then output is valid JSON with: timestamp, level, message, request_id, tenant_id

Given `DEBUG` is True and `DEV_FORMAT` is 'console'
When a log statement runs
Then output is human-readable with colour-coded level indicators

# --- Request ID ---

Given an HTTP request arrives with no X-Request-ID header
When the logging middleware runs
Then a new UUID is generated and threaded through all log statements for that request

Given a Celery task is dispatched during a request with request_id 'abc-123'
When the task runs on a worker
Then 'abc-123' is present in all task log output

# --- Sensitive field redaction ---

Given `REDACT_FIELDS` includes 'password'
When a log entry containing {"password": "hunter2"} is emitted
Then the output contains {"password": "[REDACTED]"} across all backends

# --- Request logging ---

Given `LOG_SLOW_REQUESTS_MS` is 500 and a request takes 800ms
When the request completes
Then a WARNING is emitted with the path and duration_ms

# --- Frontend log ingestion endpoint ---

Given `FRONTEND_LOGS_ENABLED` is True and an authenticated frontend client POSTs a log batch
When the request is received at POST /api/frontend-logs/
Then each entry is enriched with server-side tenant_id and verified user_id, then forwarded
to all configured BACKENDS via the same pipeline as server logs

Given a frontend client POSTs a log entry with level 'debug' and `FRONTEND_LOGS_MAX_LEVEL` is 'info'
When the entry is received
Then it is silently dropped — no error returned to the client

Given `FRONTEND_LOGS_REQUIRE_AUTH` is True and an unauthenticated request hits /api/frontend-logs/
When the endpoint processes the request
Then a 401 is returned

Given `FRONTEND_LOGS_REQUIRE_AUTH` is False and an unauthenticated request hits /api/frontend-logs/
When the endpoint processes the request
Then the entry is accepted and forwarded without user_id enrichment

Given a frontend client exceeds `FRONTEND_LOGS_RATE_LIMIT`
When further requests arrive within the rate limit window
Then a 429 is returned with Retry-After

Given a frontend log entry contains a field matching `REDACT_FIELDS`
When the entry is processed by the endpoint
Then the field value is replaced with '[REDACTED]' before forwarding to any backend

Given the frontend POSTs a batch of 50 log entries in one request
When the endpoint processes the batch
Then all 50 entries are enriched and forwarded — batch POST is supported
```
