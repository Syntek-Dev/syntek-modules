## User Story

As a developer, I want a notification core module so that I can dispatch notifications to any
channel through a unified API, enforce per-user preferences, track delivery, and enforce
GDPR-compliant consent gating — with channel adapters loaded as optional sub-modules.

## MoSCoW Priority

**Must Have**

## Acceptance Criteria

```gherkin
Given a notification is dispatched to a user who has disabled email for that notification type
When the dispatch runs
Then the email channel is skipped and the in-app notification is sent

Given a notification channel adapter is not installed
When a notification is dispatched to that channel
Then the dispatch is silently skipped and a warning is logged — no exception raised

Given a user updates their notification preferences
When the next notification is dispatched
Then the updated preferences are honoured — no caching lag

Given a notification is dispatched as in-app
When the recipient's browser is connected via WebSocket
Then the notification is delivered in real time without a page refresh

Given a delivery failure occurs on any channel
When the retry policy triggers
Then it retries with exponential back-off up to `RETRY_MAX_ATTEMPTS` times before routing to DLQ

Given a DLQ item exists for a failed notification
When the admin queries `failedNotifications`
Then the record includes channel, error message, attempt count, and last_attempted_at

Given a notification category has requires_explicit_consent = True
When a notification is dispatched to a user who has not consented to that category
Then delivery is blocked with a ConsentNotGivenError and a DeliveryFailure record is written —
     no retry is attempted

Given a user grants consent for a notification category
When the grantNotificationConsent mutation is called
Then a NotificationConsentRecord is created with consent_given_at, ip_hash, and method
And future notifications in that category are dispatched normally

Given a user withdraws consent for a notification category
When the withdrawNotificationConsent mutation is called
Then the NotificationConsentRecord is closed with consent_withdrawn_at
And the next dispatch for that category is blocked immediately — no grace period
```

## GDPR Consent Model

All notification categories define a `legal_basis` and a `requires_explicit_consent` flag. This
determines whether dispatch is automatic (legitimate interest / contract) or gated on an explicit
opt-in.

### Default Notification Categories

| Category Slug     | Display Name       | Legal Basis         | Requires Explicit Consent | Notes                                   |
| ----------------- | ------------------ | ------------------- | ------------------------- | --------------------------------------- |
| `security`        | Security Alerts    | legitimate_interest | No                        | MFA codes, login alerts, breach notices |
| `transactional`   | Transactional      | contract            | No                        | Order confirmation, booking, receipts   |
| `account`         | Account Activity   | legitimate_interest | No                        | Password change, profile update         |
| `product_updates` | Product Updates    | configurable        | Configurable              | Release notes, new features             |
| `marketing_email` | Marketing Emails   | consent             | Yes                       | Newsletters, promotions via email       |
| `marketing_sms`   | Marketing SMS      | consent             | Yes                       | SMS promotions                          |
| `push_marketing`  | Push Notifications | consent             | Yes                       | Marketing push alerts                   |
| `reminders`       | Reminders          | legitimate_interest | No (configurable)         | Appointment reminders, renewals         |

All categories are configurable via `SYNTEK_NOTIFICATIONS['CONSENT_CATEGORIES']`. Consuming projects
can override `legal_basis`, `requires_explicit_consent`, and `display_name` for any category, or add
custom categories.

### `NotificationConsentCategory` Model

One row per configured category per tenant.

| Field                       | Type      | Notes                                          |
| --------------------------- | --------- | ---------------------------------------------- |
| `slug`                      | SlugField | Unique identifier                              |
| `display_name`              | CharField |                                                |
| `legal_basis`               | CharField | `contract` / `legitimate_interest` / `consent` |
| `requires_explicit_consent` | bool      | If True, dispatch is blocked without a record  |
| `tenant`                    | FK (null) | Null = global default; set = tenant override   |

### `NotificationConsentRecord` Model

One record per grant. Consent is never updated — it is closed (withdrawn) and replaced.

| Field                  | Type      | Notes                                                            |
| ---------------------- | --------- | ---------------------------------------------------------------- |
| `user`                 | FK        | Auth user                                                        |
| `category`             | FK        | `NotificationConsentCategory`                                    |
| `consent_given_at`     | datetime  |                                                                  |
| `consent_withdrawn_at` | datetime  | Null until withdrawn                                             |
| `ip_hash`              | CharField | SHA-256 of IP at time of consent — GDPR-safe                     |
| `user_agent_hash`      | CharField | SHA-256 of UA string                                             |
| `method`               | CharField | `registration` / `settings` / `import` / `api`                   |
| `gdpr_consent_ref`     | CharField | Optional reference to `syntek-gdpr` ConsentRecord ID             |
| `legal_doc_version`    | CharField | Optional version string from `syntek-legal` (e.g. `privacy@3.2`) |
| `tenant`               | FK (null) |                                                                  |

      DLQ routing, preference update, WebSocket in-app delivery, retry back-off, consent gate block,

      `DeliveryFailure` models
- [ ] Implement channel registry — adapters register themselves on install via
      `SYNTEK_NOTIFICATION_ADAPTERS`; core dispatches to registered channels only; missing channels
      log a warning and continue

### GDPR Consent

- [ ] Implement `NotificationConsentCategory` and `NotificationConsentRecord` models
- [ ] Populate `NotificationConsentCategory` rows from `SYNTEK_NOTIFICATIONS['CONSENT_CATEGORIES']`
      at startup via `AppConfig.ready()` — idempotent upsert
- [ ] Implement `check_notification_consent(user, category)` — returns True if consent is not
      required or an open consent record exists; raises `ConsentNotGivenError` otherwise
- [ ] Implement consent gate in dispatch pipeline — called before each channel dispatch; skips with
      `ConsentNotGivenError` logged to `DeliveryFailure`; does not retry
- [ ] Implement `syntek-gdpr` SAR hook — `NotificationConsentRecord` registered as a GDPR data class
- [ ] Implement `syntek-gdpr` erasure hook — anonymises `ip_hash` and `user_agent_hash` fields

### Delivery & WebSocket

- [ ] Implement in-app WebSocket delivery (Django Channels) with consumer, group send, and
      reconnection handling
- [ ] Implement delivery tracking and per-channel failure recording
- [ ] Implement Celery retry task with exponential back-off (configurable base, cap, jitter) and DLQ
      routing after `RETRY_MAX_ATTEMPTS`

### GraphQL

      `notificationConsentStatus(userId)`, `failedNotifications`
- [ ] Implement Strawberry mutations: `markAsRead`, `markAllRead`, `updatePreferences`,
      `grantNotificationConsent(category, method)`, `withdrawNotificationConsent(category)`,

      steps
