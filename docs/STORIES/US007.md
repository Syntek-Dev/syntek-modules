# US007 — `syntek-pyo3`: PyO3 Django Bindings

**Last Updated**: 09/03/2026 **Version**: 0.1.1 **Status**: Completed **Epic**: Epic 2 — Rust
Encryption Layer

## User Story

As a backend developer, I want Django `EncryptedField` types and PyO3 bindings so that the ORM layer
safely stores and retrieves ciphertext, with all encryption and decryption handled exclusively by
the GraphQL middleware — no plaintext ever touches Django models or PostgreSQL.

## Architecture Note

The encryption boundary is the **GraphQL layer**, not the Django ORM layer.

| Path             | What happens                                                                                                                                                     |
| ---------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Write (mutation) | GraphQL middleware receives plaintext → calls `encrypt_field` / `encrypt_fields_batch` → passes ciphertext to the Django resolver → model saves ciphertext to DB |
| Read (query)     | Model loads ciphertext from DB → resolver returns ciphertext → GraphQL middleware decrypts → plaintext sent to frontend                                          |

`EncryptedField` is therefore a **storage and validation type only** — it does not encrypt on write
or decrypt on read. Its role is to ensure the DB column never accidentally stores plaintext, and to
act as a schema marker so the GraphQL middleware knows which fields require crypto handling.

Passwords are **never** encrypted with AES-256-GCM. They are one-way Argon2id hashed via
`hash_password` and `verify_password`. `EncryptedField` is not used for password columns.

## Acceptance Criteria

```gherkin
Given the syntek_pyo3 extension is compiled and installed (in .venv for syntek-modules dev; in Docker for consumer projects)
When I run `from syntek_pyo3 import encrypt_field, decrypt_field, encrypt_fields_batch,
decrypt_fields_batch, hash_password, verify_password`
Then the import succeeds with no errors

Given a Django model field declared as `EncryptedField`
When a model instance is saved with a value that is a valid ciphertext string
Then the value is stored as-is in the database column — no additional encryption is applied

Given a Django model field declared as `EncryptedField`
When a model instance is saved with a value that does not match the expected ciphertext format
Then a `ValidationError` is raised before the DB write — plaintext is never written to the DB

Given a Django model instance is fetched from the DB
When the `EncryptedField` attribute is accessed in Python
Then the raw ciphertext string is returned — decryption is NOT performed at this layer

Given an invalid decryption key or tampered ciphertext is passed to `decrypt_field`
When called from any Python context
Then a `DecryptionError` is raised and no partial plaintext is returned

Given `encrypt_fields_batch` is called with a list of (field_name, plaintext) tuples
When called from Python
Then it returns a list of ciphertext strings in the same order, delegating to syntek-crypto

Given `decrypt_fields_batch` is called with a list of (field_name, ciphertext) tuples
When called from Python
Then it returns a list of plaintext strings in the same order, or raises `BatchDecryptionError`
if any field fails

Given maturin develop is run in the syntek-modules .venv (dev only)
When I import syntek_pyo3 in the active .venv
Then the native extension loads without any import errors

Given maturin build produces a wheel
When the wheel is installed inside a consumer Docker image
Then syntek_pyo3 imports and functions correctly in that environment
```
