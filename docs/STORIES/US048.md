## User Story

As a developer, I want an authentication UI package so that I can drop in fully styled login,
registration, MFA, OAuth, and passkey flows that connect directly to the `syntek-auth` GraphQL API ‚Äî
with full control over presentation mode (full-page or modal), routing rules, enabled flows, and
layout via a slot/render-prop API, all configured on the `SyntekAuthProvider` without touching
component internals.

## Configuration Contract

All behaviour is configured via props on `SyntekAuthProvider`, which wraps the consuming
application's root. No `.env` config ‚Äî all values come from props.

```tsx
import { SyntekAuthProvider } from "@syntek/ui-auth";

<SyntekAuthProvider
  // --- Presentation mode ---
  mode="page" // 'page' | 'modal'
  // 'page'  ‚Äî auth flows live at dedicated routes (loginPath, registerPath, etc.)
  // 'modal' ‚Äî auth flows render as an overlay triggered from any page

  // --- Routes (mode: 'page' only) ---
  loginPath="/login" // string ‚Äî route rendered for the login flow
  registerPath="/register" // string ‚Äî route rendered for the registration flow
  forgotPasswordPath="/forgot" // string
  resetPasswordPath="/reset" // string
  // --- Redirects ---
  postLoginRedirect="/" // string | ((user: User) => string) ‚Äî where to send after successful login
  postLogoutRedirect="/login" // string ‚Äî where to send after logout
  postRegisterRedirect="/" // string | null ‚Äî null = show confirmation screen instead of redirect
  // --- Enabled flows ---
  enableRegistration={true} // boolean ‚Äî show/hide registration link and route
  enableForgotPassword={true} // boolean
  enableOAuth={true} // boolean ‚Äî show OAuth provider buttons
  enablePasskeys={true} // boolean ‚Äî show passkey login option
  enableMFA={true} // boolean ‚Äî show MFA setup option in account settings
  // --- Modal options (mode: 'modal' only) ---
  modalTrigger={<Button>Sign in</Button>} // ReactNode ‚Äî element that opens the modal
  closeOnOutsideClick={true} // boolean
  closeOnSuccess={true} // boolean ‚Äî close modal after successful login
  // --- Slots (render-prop API for custom layout wrappers) ---
  // All slots receive the default content as children and can wrap or replace it.
  slots={{
    loginFormWrapper: (props) => <Card className="my-card">{props.children}</Card>,
    registerFormWrapper: (props) => <Card>{props.children}</Card>,
    mfaChallenge: (props) => <CustomMFAScreen {...props} />, // full replacement
    oauthButtons: (props) => <MyOAuthRow {...props} />,
    forgotPasswordForm: (props) => <>{props.children}</>,
  }}

  // --- GraphQL endpoint ---
  // Inherited from @syntek/api-client ‚Äî not set here directly.
>
  <App />
</SyntekAuthProvider>;
```

**Key rules:**

- In `mode: 'page'`, the provider registers routes at `loginPath` etc. using the consuming app's
  router. The consuming app must wrap with the router before `SyntekAuthProvider`.
- In `mode: 'modal'`, no routes are registered. All flows render inside the modal overlay.
- `postLoginRedirect` as a function receives the authenticated user object ‚Äî use this to redirect
  different roles to different dashboards.
- Slots wrap the default content; they do not replace the data logic. Use `mfaChallenge` for a full
  replacement of just the MFA screen layout.
- `enableOAuth` and `enablePasskeys` are client-side toggles ‚Äî the backend still controls which
  methods are actually available via the GraphQL schema.

## Acceptance Criteria

```gherkin
# --- Presentation mode: page ---

Given `mode` is 'page' and `loginPath` is '/auth/login'
When the user navigates to '/auth/login'
Then the login form renders at that route as a full page

Given `mode` is 'page' and the user successfully logs in
When the session is established
Then the user is redirected to `postLoginRedirect`

Given `postLoginRedirect` is a function returning '/admin' for admin users and '/dashboard' for others
When an admin user logs in
Then they are redirected to '/admin'

# --- Presentation mode: modal ---

Given `mode` is 'modal'
When the `modalTrigger` element is clicked
Then the auth modal opens with the login form ‚Äî no route change occurs

Given `mode` is 'modal' and `closeOnSuccess` is True
When the user successfully logs in via the modal
Then the modal closes and the current page re-renders with the authenticated state

Given `mode` is 'modal' and `closeOnOutsideClick` is True
When the user clicks outside the modal
Then the modal closes without completing authentication

# --- Enabled flows ---

Given `enableRegistration` is False
When the login form is rendered
Then no register link or route is present

Given `enableOAuth` is False
When the login form is rendered
Then no OAuth provider buttons are rendered

Given `enableForgotPassword` is False
When the login form is rendered
Then no forgot password link is present

# --- Slots ---

Given a custom `loginFormWrapper` slot is provided
When the login form renders
Then it is wrapped inside the custom component with the default form content as children

Given a custom `mfaChallenge` slot is provided
When MFA is required and the MFA screen is shown
Then the custom MFA component renders in place of the default

# --- Error handling ---

Given a user submits the login form with invalid credentials
When the mutation returns an error
Then the error is displayed inline on the form ‚Äî not as a browser alert ‚Äî and no redirect occurs

# --- MFA ---

Given MFA is required for the user's account
When login credentials are validated
Then the MFA challenge screen is presented and the session is not issued until the TOTP/passkey is verified

# --- OAuth ---

Given an OAuth provider button is clicked
When the OAuth redirect completes
Then the user is logged in, the session is established, and they are redirected to postLoginRedirect

# --- Passkeys ---

Given a passkey is registered for the user
When the passkey login button is clicked
Then the browser's WebAuthn API is invoked and the session is established on successful assertion
```
