# US008 — `syntek-graphql-crypto`: GraphQL Encryption Middleware

**Last Updated**: 09/03/2026 **Version**: 0.1.1 **Status**: To Do **Epic**: Epic 2 — Rust Encryption
Layer

---

## User Story

As a security engineer, I want a GraphQL middleware layer that is the single encryption boundary for
the entire system, so that plaintext from the frontend is encrypted before reaching the ORM on
writes, and ciphertext from the DB is decrypted before reaching the frontend on reads — with no
crypto logic in individual resolvers.

## Architecture Note

This middleware is the **only place** in the system where plaintext and ciphertext are exchanged.

```text
Write path:  Frontend (plaintext) → [middleware encrypts] → Resolver → ORM → DB (ciphertext)
Read path:   DB (ciphertext) → ORM → Resolver → [middleware decrypts] → Frontend (plaintext)
```

Individual resolvers are completely oblivious to encryption. They receive ciphertext on reads and
pass ciphertext to the ORM on writes — the middleware handles all transformation transparently.

Fields are annotated at the schema level:

- `@encrypted` — single field, encrypted/decrypted individually
- `@encrypted(batch: "group_name")` — field belongs to a named group; all fields sharing the same
  `group_name` on the same type are encrypted/decrypted in a single batch Rust call

Choosing individual vs batch per field:

- **Individual**: fields that are queried or mutated in isolation (e.g. `email` on its own)
- **Batch**: fields that are always read or written together (e.g. `first_name`+`last_name`, or
  `address_line_1`+`address_line_2`+`city`+`county`+`postcode`)

Passwords are **never** handled by this middleware — password fields use `hash_password` /
`verify_password` directly in the resolver (one-way hash, not reversible encryption).

## Acceptance Criteria

```gherkin
# --- Write path (mutations) ---

Given a GraphQL mutation input contains a field annotated with `@encrypted`
When the middleware processes the input
Then the plaintext value is encrypted via `encrypt_field` before the resolver runs, and the
resolver receives only the ciphertext string

Given a GraphQL mutation input contains multiple fields sharing `@encrypted(batch: "group_name")`
When the middleware processes the input
Then all fields in the group are encrypted in a single `encrypt_fields_batch` call, and the
resolver receives the ciphertext strings — one Rust call for the whole group

Given a mutation input contains both `@encrypted` individual fields and `@encrypted(batch: ...)`
group fields
When the middleware processes the input
Then individual fields are encrypted individually and batch groups are each encrypted in one
batch call — never mixing groups or combining unrelated fields

# --- Read path (queries) ---

Given a GraphQL resolver returns a field annotated with `@encrypted`
When the middleware processes the response
Then the ciphertext value is decrypted via `decrypt_field` before serialisation, and the
frontend receives the plaintext string

Given a GraphQL resolver returns multiple fields sharing `@encrypted(batch: "group_name")`
When the middleware processes the response
Then all fields in the group are decrypted in a single `decrypt_fields_batch` call before
serialisation

Given a field is NOT annotated with `@encrypted` or `@encrypted(batch: ...)`
When the middleware processes the response
Then it passes through unchanged with no performance overhead

# --- Error handling ---

Given a decryption failure for a single `@encrypted` field
When the middleware processes the response
Then the field is set to `null`, a structured error is appended to the GraphQL `errors` array
(including field path and error type), and the rest of the response is not aborted

Given a decryption failure for any field within an `@encrypted(batch: "group_name")` group
When the middleware processes the response
Then ALL fields in that group are set to `null` and a single structured error is appended to
the `errors` array — partial group results are never returned

Given an encryption failure for any field within an `@encrypted(batch: "group_name")` mutation
When the middleware processes the input
Then the entire mutation is rejected with a structured error — no partial ciphertext is written
to the ORM

# --- Auth guard ---

Given an unauthenticated request attempts to resolve an `@encrypted` or `@encrypted(batch: ...)`
field
When the middleware runs
Then all encrypted fields are set to `null`, an authorisation error is appended to `errors`,
and the attempt is logged via syntek-logging — the request is not aborted entirely

# --- Resolver isolation ---

Given middleware is configured at application startup
When individual resolvers run
Then they require no encryption or decryption logic — all crypto is handled by the middleware
```
