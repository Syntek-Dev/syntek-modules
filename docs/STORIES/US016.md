## User Story

As a developer, I want a per-tenant settings store so that each tenant can have independently
configurable typed values — with a schema registry that modules and consuming projects extend,
custom validators, Redis caching, design token overrides, and a GraphQL API — all controlled from
`settings.py` without touching module source.

## Design Decisions

### Schema registry

Settings are not just stored — they are declared. Every module (syntek-auth, syntek-payments, etc.)
registers its own settings schema via `register_settings()` in its `AppConfig.ready()`. The
consuming project extends the registry with its own custom settings via `SYNTEK_SETTINGS['SCHEMA']`.
This means the admin UI always shows every setting with its type, default, and description — no
guessing what keys exist.

### Type system

Six supported types: `str`, `int`, `float`, `bool`, `json`, `url`, `email`. Type coercion is applied
on read — the raw stored value is a string in the database; `get_setting()` returns the correctly
typed value. Setting a value that cannot be coerced to the declared type raises `SettingTypeError`
before any database write.

### Custom validators

Each setting definition can declare a list of validator callables. Validators receive the
post-coercion value and raise `ValidationError` or return `False` to reject it. This lets consuming
projects enforce domain rules (e.g. max length, allowed values, regex patterns) beyond what the type
system provides.

### Evaluation hierarchy

Settings have three layers, resolved in this order:

1. **Tenant override** — stored in `TenantSetting` model, highest priority
2. **Global default** — declared in the schema, used when no tenant override exists

There is no user-level setting override — settings are tenant-wide. Per-user preferences that vary
within a tenant belong in `syntek-auth` (profile) or a feature flag (US017).

### Caching

Setting reads are hot-path. Every resolved value is cached in Redis (syntek-cache) per tenant per
key with a configurable TTL. Cache is invalidated immediately on write. If syntek-cache is not
installed, caching is silently skipped — settings still work, just slower.

## Configuration Contract

All behaviour is controlled via `SYNTEK_SETTINGS` in `settings.py`. Validated at startup in
`AppConfig.ready()`.

```python
SYNTEK_SETTINGS = {

    # --- Schema registry ---
    # Consuming projects declare custom settings here.
    # Each module also registers its own settings via register_settings() in AppConfig.ready().
    # Keys must be unique — collisions with module-registered keys raise ImproperlyConfigured.
    'SCHEMA': {
        # Format per key:
        # 'my_setting_key': {
        #     'type': 'str',              # 'str' | 'int' | 'float' | 'bool' | 'json' | 'url' | 'email'
        #     'default': '',             # value used when no tenant override exists (must match type)
        #     'nullable': False,          # bool — allow None as a valid value
        #     'description': '',          # str — shown in admin UI and GraphQL schema docs
        #     'group': 'General',         # str — UI grouping for admin panel organisation
        #     'validators': [],           # list[str] — dotted paths to callable(value) -> bool | raises ValidationError
        # }
        #
        # Example:
        # 'max_team_members': {
        #     'type': 'int',
        #     'default': 10,
        #     'nullable': False,
        #     'description': 'Maximum number of team members per tenant',
        #     'group': 'Limits',
        #     'validators': ['myapp.validators.positive_integer'],
        # },
    },

    # --- Caching ---
    'CACHE_ENABLED': True,          # bool — cache resolved setting values in syntek-cache (Redis)
                                     # Silently skipped if syntek-cache is not installed.
    'CACHE_TTL': 300,               # int (seconds) — TTL per key per tenant; invalidated on write

    # --- Design tokens ---
    'DESIGN_TOKENS_ENABLED': True,  # bool — enable TenantDesignToken model and designTokens GraphQL query
                                     # Allows per-tenant CSS custom property overrides (colours, fonts, etc.)
                                     # Frontend applies overrides at :root via @syntek/session

    # --- GraphQL ---
    'EXPOSE_VIA_GRAPHQL': True,     # bool — expose tenantSettings, designTokens queries and mutations
    'REQUIRE_ADMIN_FOR_WRITE': True, # bool — mutations require is_tenant_admin permission
                                      # Set False only for programmatic/service-to-service writes
}
```

**Key rules:**

- `SCHEMA` keys are merged with module-registered keys at startup. A collision raises
  `ImproperlyConfigured` naming the conflicting key and the module that owns it.
- `default` in the schema must be a value that passes type coercion and all declared validators —
  validated at startup, not at first read.
- `CACHE_ENABLED: True` requires syntek-cache (US077). If not installed, caching is silently
  disabled with a startup warning. This is the only setting that degrades gracefully rather than
  raising `ImproperlyConfigured`.
- `DESIGN_TOKENS_ENABLED: False` drops the `TenantDesignToken` model entirely — no migration run. If
  it was previously True, a manual migration to drop the table is required (documented).
- `REQUIRE_ADMIN_FOR_WRITE: True` relies on syntek-permissions (US011). Enforced at the GraphQL
  resolver level using the `@hasPermission('manage_tenant_settings')` directive.

## Python API

```python
from syntek_settings import get_setting, set_setting, register_settings

# Read a setting for the current tenant
max_members = get_setting('max_team_members', tenant=request.tenant)
# Returns the tenant override if set, otherwise the schema default.
# Always returns the correctly typed value (int in this case).

# Write a setting for a tenant
set_setting('max_team_members', value=25, tenant=request.tenant)
# Raises SettingTypeError if value cannot be coerced to declared type.
# Raises ValidationError if a declared validator rejects the value.
# Invalidates the cache key for this tenant immediately.

# Module registration (called in AppConfig.ready())
register_settings({
    'auth_session_timeout': {
        'type': 'int',
        'default': 1800,
        'description': 'Session idle timeout in seconds',
        'group': 'Security',
    },
})
```

## GraphQL API

```graphql
# Query — returns all settings for the current tenant with their resolved values
query TenantSettings {
  tenantSettings {
    key
    value
    type
    description
    group
    isOverridden # true if tenant has set an override; false if using schema default
  }
}

# Query — returns design token overrides for the current tenant
query DesignTokens {
  designTokens {
    property # CSS custom property name e.g. "--color-primary"
    value # CSS value e.g. "#FF6B00"
  }
}

# Mutation — write a setting (requires manage_tenant_settings permission)
mutation UpdateSetting($key: String!, $value: String!) {
  updateSetting(key: $key, value: $value) {
    key
    value
  }
}

# Mutation — write a design token override
mutation UpdateDesignToken($property: String!, $value: String!) {
  updateDesignToken(property: $property, value: $value) {
    property
    value
  }
}

# Mutation — reset a setting to its schema default
mutation ResetSetting($key: String!) {
  resetSetting(key: $key) {
    key
    value
  }
}
```

## Acceptance Criteria

```gherkin
# --- Schema registry ---

Given SCHEMA declares 'max_members' with type 'int' and default 10
When AppConfig.ready() runs
Then the setting is registered with correct type and default

Given a consuming project declares 'max_members' in SCHEMA and syntek-auth also registers 'max_members'
When AppConfig.ready() runs
Then ImproperlyConfigured is raised naming the conflicting key and owning module

Given SCHEMA declares a setting with default 'not-a-number' and type 'int'
When AppConfig.ready() runs
Then ImproperlyConfigured is raised — default does not match declared type

Given SCHEMA declares validators: ['myapp.validators.max_100'] and default is 200
When AppConfig.ready() runs
Then ImproperlyConfigured is raised — default fails validator

# --- get_setting / set_setting ---

Given 'max_members' is declared with type 'int' and default 10
When get_setting('max_members', tenant=tenant_a) is called and tenant_a has no override
Then 10 is returned as an int

Given tenant_a sets 'max_members' to 25
When get_setting('max_members', tenant=tenant_a) is called
Then 25 is returned as an int

Given tenant_a sets 'max_members' to 25 and tenant_b has no override
When get_setting('max_members', tenant=tenant_b) is called
Then 10 is returned — tenant_b is isolated from tenant_a's override

Given set_setting('max_members', value='not-a-number', tenant=tenant_a) is called
When the call is made
Then SettingTypeError is raised and nothing is persisted

Given 'email_from' is declared with type 'email' and a validator rejecting 'example.com' domains
When set_setting('email_from', value='admin@example.com', tenant=tenant_a) is called
Then ValidationError is raised and nothing is persisted

# --- Caching ---

Given CACHE_ENABLED is True and get_setting is called twice within CACHE_TTL
When the second call runs
Then the Redis cache is read — the database is not queried a second time

Given a tenant writes a setting override
When get_setting is called immediately after
Then the updated value is returned — cache was invalidated on write

Given CACHE_ENABLED is True and syntek-cache is not installed
When AppConfig.ready() runs
Then a startup warning is logged and get_setting works without caching

# --- Design tokens ---

Given DESIGN_TOKENS_ENABLED is True and tenant_a writes '--color-primary' = '#FF6B00'
When the designTokens GraphQL query is called for tenant_a
Then [{"property": "--color-primary", "value": "#FF6B00"}] is returned

Given tenant_b has no design token overrides
When the designTokens query is called for tenant_b
Then an empty list is returned — the frontend uses shared/tokens defaults

Given DESIGN_TOKENS_ENABLED is False
When AppConfig.ready() runs
Then the TenantDesignToken model is not available and designTokens query returns null

# --- GraphQL ---

Given EXPOSE_VIA_GRAPHQL is True and a tenant admin calls updateSetting
When the mutation runs with a valid value
Then the setting is updated and the new value is returned

Given REQUIRE_ADMIN_FOR_WRITE is True and a non-admin user calls updateSetting
When the mutation runs
Then a PermissionDenied error is returned — setting is not changed

Given resetSetting is called for a key with a tenant override
When the mutation runs
Then the override is deleted and get_setting returns the schema default

Given EXPOSE_VIA_GRAPHQL is False
When a GraphQL client queries tenantSettings
Then the query returns null — no settings are exposed
```
