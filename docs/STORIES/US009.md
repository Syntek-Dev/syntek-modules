# US009 — `syntek-auth`: Authentication Module

## User Story

As a developer installing `syntek-auth`, I want a fully configured authentication system so that my
Django project supports MFA, passkeys, OIDC, and secure session management with zero custom auth
code — and every authentication behaviour (login fields, password policy, MFA methods, token
lifetimes, account lockout) is configurable from `settings.py` without touching the module source.

## MoSCoW Priority

**Must Have**

## Configuration Contract

All behaviour is controlled via `SYNTEK_AUTH` in `settings.py`. Validated at startup in
`AppConfig.ready()` — any missing required key or invalid value raises `ImproperlyConfigured`.

```python
SYNTEK_AUTH = {
    # --- Identity / login fields ---
    'LOGIN_FIELD': 'email',            # 'email' | 'username' | 'phone' | 'email_or_username' | 'email_or_phone'
    'REQUIRE_EMAIL': True,             # bool — must email be collected on registration?
    'REQUIRE_USERNAME': False,         # bool — must username be collected on registration?
    'REQUIRE_PHONE': False,            # False | 'optional' | 'required'

    # --- Username rules (applies when REQUIRE_USERNAME is not False) ---
    'USERNAME_MIN_LENGTH': 3,          # int
    'USERNAME_MAX_LENGTH': 30,         # int
    'USERNAME_ALLOWED_CHARS': r'^[a-zA-Z0-9_.\-]+$',  # regex — validated at registration
    'USERNAME_RESERVED': ['admin', 'root', 'superuser', 'support', 'syntek'],  # list[str]
    'USERNAME_CASE_SENSITIVE': False,  # bool — False = stored and compared lowercased

    # --- Password policy ---
    'PASSWORD_MIN_LENGTH': 12,         # int
    'PASSWORD_MAX_LENGTH': 128,        # int
    'PASSWORD_REQUIRE_UPPERCASE': True,   # bool
    'PASSWORD_REQUIRE_LOWERCASE': True,   # bool
    'PASSWORD_REQUIRE_DIGITS': True,      # bool
    'PASSWORD_REQUIRE_SYMBOLS': False,    # bool
    'PASSWORD_HISTORY': 5,               # int — previous passwords blocked from reuse (0 = disabled)
    'PASSWORD_EXPIRY_DAYS': 0,           # int — 0 = never expires
    'PASSWORD_BREACH_CHECK': True,       # bool — HaveIBeenPwned check on registration/change

    # --- MFA ---
    'MFA_REQUIRED': False,               # bool — True = all users must configure MFA before accessing protected resources
    'MFA_METHODS': ['totp', 'passkey'],  # list — enabled methods; any subset of: 'totp' | 'sms' | 'email_otp' | 'passkey'
    'MFA_BACKUP_CODES_COUNT': 12,        # int — single-use backup codes generated at MFA setup

    # --- Session / tokens ---
    'ACCESS_TOKEN_LIFETIME': 900,        # int (seconds) — 15 min default
    'REFRESH_TOKEN_LIFETIME': 604800,    # int (seconds) — 7 days default
    'ROTATE_REFRESH_TOKENS': True,       # bool — rotate refresh token on every use

    # --- Account security ---
    'LOCKOUT_THRESHOLD': 5,              # int — consecutive failed attempts before lockout
    'LOCKOUT_DURATION': 900,             # int (seconds) — 0 = until manual unlock
    'LOCKOUT_STRATEGY': 'progressive',   # 'fixed' | 'progressive' (duration doubles each lockout)

    # --- Registration ---
    'REGISTRATION_ENABLED': True,        # bool — False = invite-only or admin-created accounts only
    'EMAIL_VERIFICATION_REQUIRED': True, # bool
    'PHONE_VERIFICATION_REQUIRED': False,# bool

    # --- OAuth / OIDC providers ---
    # Each entry must pass the US076 MFA-enforcing allowlist check at startup.
    'OIDC_PROVIDERS': [
        # {
        #     'id': 'github',
        #     'label': 'GitHub',
        #     'client_id': env('GITHUB_CLIENT_ID'),
        #     'client_secret': env('GITHUB_CLIENT_SECRET'),
        #     'discovery_url': 'https://token.actions.githubusercontent.com/.well-known/openid-configuration',
        #     'mfa_enforced': True,   # required for self-hosted providers not on the built-in allowlist
        # }
    ],
}
```

**Key rules:**

- Secrets always come from environment variables — never hardcoded in settings.
- `LOGIN_FIELD` determines both the registration form field label and the login lookup field.
- `MFA_METHODS` controls which methods are offered to users — at least one must be listed.
- OIDC providers not on the built-in MFA-enforcing allowlist (per US076) must include
  `'mfa_enforced': True` — the operator certifies enforcement.

## Acceptance Criteria

```gherkin
# --- Settings validation ---

Given `SYNTEK_AUTH` is configured in settings.py
When `AppConfig.ready()` runs
Then all required keys are validated and `ImproperlyConfigured` is raised with a clear message for any missing or invalid value

Given `LOGIN_FIELD` is set to 'username' and `REQUIRE_USERNAME` is False
When `AppConfig.ready()` runs
Then `ImproperlyConfigured` is raised — LOGIN_FIELD 'username' requires REQUIRE_USERNAME to be True

Given `MFA_METHODS` is an empty list
When `AppConfig.ready()` runs
Then `ImproperlyConfigured` is raised — at least one MFA method must be enabled

# --- Login field behaviour ---

Given `LOGIN_FIELD` is 'email'
When a user submits the login mutation
Then the lookup is performed against the email field and username/phone are not checked

Given `LOGIN_FIELD` is 'email_or_username'
When a user submits the login mutation with their username
Then the lookup checks both email and username fields and returns the matching account

Given `LOGIN_FIELD` is 'phone' and `REQUIRE_PHONE` is 'required'
When a user registers without a phone number
Then registration is rejected with a validation error

# --- Password policy ---

Given `PASSWORD_MIN_LENGTH` is 14 and `PASSWORD_REQUIRE_SYMBOLS` is True
When a user registers with a 10-character password containing no symbols
Then registration is rejected with a validation error naming both failures

Given `PASSWORD_HISTORY` is 5
When a user resets their password to one of their last 5 passwords
Then the reset is rejected with a password history error

Given `PASSWORD_BREACH_CHECK` is True
When a user registers with a password found in the HaveIBeenPwned breach database
Then registration is rejected with a breach warning (no specific breach count revealed)

Given `PASSWORD_EXPIRY_DAYS` is 90
When a user's password was last changed 91 days ago
Then login returns a `password_expired` flag and the user is directed to change it before proceeding

# --- MFA ---

Given `MFA_METHODS` includes 'totp' but not 'sms'
When a user opens their MFA settings screen
Then only TOTP and backup code options are presented — SMS OTP is not offered

Given `MFA_REQUIRED` is True and a user has not configured MFA
When the user logs in with valid credentials
Then a partial session with `mfa_setup_required: true` is issued and all protected resources return 401 until MFA setup completes

Given `MFA_REQUIRED` is True and a user authenticates via OIDC with a valid `amr` claim containing 'mfa'
When the OIDC callback is processed
Then a full session is issued — the provider's MFA satisfies the requirement and no TOTP prompt is shown

# --- Account lockout ---

Given `LOCKOUT_THRESHOLD` is 5 and `LOCKOUT_STRATEGY` is 'progressive'
When a user fails login 5 times
Then the account is locked for `LOCKOUT_DURATION` seconds

Given `LOCKOUT_STRATEGY` is 'progressive'
When a user is locked out a second time after the first lockout expires
Then the lockout duration doubles

# --- Registration ---

Given `REGISTRATION_ENABLED` is False
When the register mutation is called
Then a clear error is returned — registration is disabled for this deployment

Given `EMAIL_VERIFICATION_REQUIRED` is True
When a user registers with a valid email
Then an account is created in an unverified state and a verification email is sent; protected resources are inaccessible until verified

# --- Token rotation ---

Given `ROTATE_REFRESH_TOKENS` is True and a valid refresh token is used
When the refreshToken mutation is called
Then a new access token and a new refresh token are issued; the old refresh token is invalidated

# --- Existing criteria ---

Given a user registers with a valid email and password
When the registration mutation is called
Then the password is hashed via Argon2id (through syntek-pyo3) and no plaintext is stored

Given a valid JWT access token has expired
When the client sends the refresh token
Then a new access token is issued

Given an unauthenticated request hits any protected endpoint
When the request is processed
Then a 401 is returned with no user data or stack trace leaked

Given a passkey is registered for a user
When the user authenticates with that passkey
Then a full session is issued without a password prompt

Given `SYNTEK_AUTH` is configured with an OIDC provider (e.g. Defguard, Authentik, Keycloak)
When a user initiates login via that provider
Then the module performs OIDC discovery, validates the ID token, and issues a session

Given an OIDC provider's ID token is missing or has an invalid `amr` claim
When token validation runs
Then the login is rejected with a clear error and no session is issued

Given `SYNTEK_AUTH['OIDC_PROVIDERS']` lists multiple providers
When `AppConfig.ready()` runs
Then each provider passes the US076 allowlist check — any blocked provider raises `ImproperlyConfigured`

# --- Email verification ---

Given `EMAIL_VERIFICATION_REQUIRED` is True and a user registers with a valid email
When the `verifyEmail` mutation is called with a valid, unexpired token
Then the account is marked as verified and a full session can be issued

Given `EMAIL_VERIFICATION_REQUIRED` is True
When the `verifyEmail` mutation is called with an expired token
Then the mutation returns an error and instructs the user to request a new verification email

Given `EMAIL_VERIFICATION_REQUIRED` is True
When the `verifyEmail` mutation is called with a token that has already been used
Then the mutation returns an error — tokens are single-use

Given `EMAIL_VERIFICATION_REQUIRED` is True and a user's account is unverified
When the user attempts to access any protected resource
Then a 401 is returned with a `email_not_verified` error code

Given `EMAIL_VERIFICATION_REQUIRED` is True
When the `resendVerificationEmail` mutation is called for an already-verified account
Then the mutation returns an error — email is already verified

# --- Phone verification ---

Given `PHONE_VERIFICATION_REQUIRED` is True and a user has provided a phone number
When the `verifyPhone` mutation is called with a valid, unexpired OTP
Then the phone number is marked as verified

Given `PHONE_VERIFICATION_REQUIRED` is True
When the `verifyPhone` mutation is called with an incorrect OTP three times in a row
Then the OTP is invalidated and a new one must be requested — brute-forcing OTPs is not possible

Given `PHONE_VERIFICATION_REQUIRED` is True
When the `verifyPhone` mutation is called with an expired OTP
Then the mutation returns an error and the user is prompted to request a new OTP

Given `REQUIRE_PHONE` is 'required' and `PHONE_VERIFICATION_REQUIRED` is True
When a user registers without providing a phone number
Then registration is rejected with a validation error

# --- Forgot password / password reset ---

Given `EMAIL_VERIFICATION_REQUIRED` is True and a user requests a password reset for an unverified account
When the `resetPasswordRequest` mutation is called
Then a success response is returned (no account enumeration) but no email is sent

Given a user submits a valid email address to the `resetPasswordRequest` mutation
When the mutation is processed
Then a password reset email is dispatched with a single-use, time-limited token; the response is always a generic success (no account enumeration)

Given a user submits an email address that does not exist to the `resetPasswordRequest` mutation
When the mutation is processed
Then the response is identical to the success case — no account enumeration

Given a valid reset token is used in the `resetPasswordConfirm` mutation with a new password that meets policy
When the mutation is processed
Then the password is updated, all existing refresh tokens for the account are invalidated, and a new session may be issued

Given a valid reset token is used in the `resetPasswordConfirm` mutation with a password found in `PASSWORD_HISTORY`
When the mutation is processed
Then the reset is rejected with a password history error

Given a reset token has expired (default TTL: 1 hour)
When the `resetPasswordConfirm` mutation is called
Then the mutation returns a token-expired error — no password change occurs

Given a reset token is used successfully once
When the same token is used again in `resetPasswordConfirm`
Then the mutation returns a token-already-used error — tokens are single-use

# --- Authenticated password change ---

Given an authenticated user submits their current password and a new password that meets policy
When the `changePassword` mutation is called
Then the password is updated, all other active refresh tokens are invalidated (force re-login on other devices), and the current session remains valid

Given an authenticated user submits an incorrect current password to `changePassword`
When the mutation is processed
Then the mutation is rejected with an invalid-credentials error and the failed attempt is counted toward `LOCKOUT_THRESHOLD`

Given `PASSWORD_HISTORY` is 5 and an authenticated user attempts to reuse a recent password via `changePassword`
When the mutation is processed
Then the change is rejected with a password history error

# --- Logout and session invalidation ---

Given an authenticated user calls the `logout` mutation
When the mutation is processed
Then the current refresh token is invalidated and the access token is added to a short-lived denylist (until natural expiry); subsequent requests with the old tokens return 401

Given `logout` is called with an already-invalidated refresh token
When the mutation is processed
Then a 401 is returned — replayed tokens are rejected

Given an admin revokes a user's session via the admin mutation `revokeAllSessions`
When the revocation is processed
Then all refresh tokens for that user are invalidated and all subsequent access token requests return 401
```
