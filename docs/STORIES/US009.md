# US009 — `syntek-auth`: Authentication Module

**Last Updated**: 06/03/2026 **Version**: 0.1.0 **Status**: To Do **Epic**: Epic 3 — Core Backend

## User Story

As a developer installing `syntek-auth`, I want a fully configured authentication system so that my
Django project supports MFA, passkeys, OIDC, and secure session management with zero custom auth
code — including out-of-the-box support for self-hosted identity providers such as Defguard,
Authentik, and Keycloak via standard OIDC.

**Must Have**

## Acceptance Criteria

```gherkin
Given `SYNTEK_AUTH` is configured in settings.py
When `AppConfig.ready()` runs
Then all required settings are validated and `ImproperlyConfigured` is raised for any missing required value

Given a user registers with a valid email and password
When the registration mutation is called
Then the password is hashed via Argon2id (through syntek-pyo3) and no plaintext is stored

Given a user has not enabled MFA and `SYNTEK_AUTH['MFA_REQUIRED']` is `False` (default)
When the user logs in with valid credentials
Then a full session is issued immediately — MFA is not prompted

Given a user has enabled a TOTP authenticator app on their account
When the user submits valid first-factor credentials
Then a partial session token is returned and the TOTP code is required before a full session is issued

Given `SYNTEK_AUTH['MFA_REQUIRED']` is `True` and a user has not yet configured MFA
When the user logs in with valid credentials
Then a partial session is issued with a `mfa_setup_required` flag and the user is directed to
set up an authenticator app before any protected resource is accessible

Given `SYNTEK_AUTH['MFA_REQUIRED']` is `True` and the user authenticates via OIDC/SSO
When the OIDC callback is processed and the `amr` claim confirms MFA was used
Then a full session is issued — the provider's MFA satisfies the requirement and no additional
TOTP prompt is shown

Given 5 consecutive failed login attempts from one account
When a 6th attempt is made within the lockout window
Then the account is locked and a 429 is returned with `Retry-After`

Given a valid JWT access token has expired
When the client sends the refresh token
Then a new access token is issued and the refresh token is rotated

Given an unauthenticated request hits any protected endpoint
When the request is processed
Then a 401 is returned with no user data or stack trace leaked

Given a passkey is registered for a user
When the user authenticates with that passkey
Then a full session is issued without a password prompt

Given `SYNTEK_AUTH` is configured with an OIDC provider (e.g. Defguard, Authentik, Keycloak)
When a user initiates login via that provider
Then the module performs OIDC discovery against the provider's `.well-known/openid-configuration`
endpoint, validates the ID token (signature, issuer, audience, expiry), and issues a session

Given an OIDC provider returns an ID token
When the token is validated
Then the `amr` claim is inspected and login is only completed if it contains `mfa` or `totp` —
confirming the provider enforced MFA during the authentication flow

Given an OIDC provider's ID token is missing or has an invalid `amr` claim
When token validation runs
Then the login is rejected with a clear error and no session is issued

Given `SYNTEK_AUTH['OIDC_PROVIDERS']` lists multiple providers
When `AppConfig.ready()` runs
Then each provider's discovery URL is reachable and each provider entry passes the US076
allowlist check — any misconfigured or blocked provider raises `ImproperlyConfigured`
```

## Tasks

- [ ] **Red phase:** write failing tests — all acceptance criteria + IDOR, brute-force, session
      fixation, token leakage, OIDC discovery failure, invalid `amr` claim rejection
- [ ] Implement `SYNTEK_AUTH` settings schema with `AppConfig.ready()` validation, including
      `OIDC_PROVIDERS` list and provider allowlist check per US076
- [ ] Implement `User` model with `EncryptedField` for PII (email, phone)
- [ ] Replace Django's default password hasher with Argon2id via syntek-pyo3
- [ ] Implement `SYNTEK_AUTH['MFA_REQUIRED']` setting (default `False`):
  - `False`: authenticator app is opt-in — users may enable TOTP on their account voluntarily
  - `True`: users without MFA configured are issued a `mfa_setup_required` partial session and
    blocked from protected resources until they complete TOTP setup
  - OIDC/SSO users with a valid `amr` claim satisfy `MFA_REQUIRED` without an additional prompt
- [ ] Implement TOTP authenticator app flow (RFC 6238) — QR code provisioning, code verification,
      backup codes (8 single-use codes generated at setup)
- [ ] Implement SMS OTP and email OTP as secondary MFA options (lower assurance than TOTP —
      available only when `MFA_REQUIRED` is `False` or explicitly enabled by operator)
- [ ] Implement WebAuthn/FIDO2 passkey registration and authentication
- [ ] Implement generic OIDC client:
  - OIDC discovery — fetch and cache `.well-known/openid-configuration` at startup
  - ID token validation — signature (RS256/ES256), issuer, audience, expiry
  - `amr` claim enforcement — reject if `mfa` or `totp` not present
  - Userinfo endpoint fetch for profile data (name, email)
  - Configurable via `SYNTEK_AUTH['OIDC_PROVIDERS']` — supports multiple providers simultaneously
- [ ] Implement named OAuth/OIDC adapters for the MFA-enforcing provider allowlist (per US076):
      GitHub (OAuth 2.0 + 2FA platform-mandatory), GitLab (OIDC, instance MFA enforced), Okta
      (OIDC), Microsoft Entra ID / Azure AD (OIDC, Conditional Access MFA)
- [ ] Implement provider allowlist validation in `AppConfig.ready()` — raises `ImproperlyConfigured`
      if a blocked provider (Google, Facebook, Instagram, LinkedIn, Twitter/X, Apple consumer,
      Discord) is configured
- [ ] Implement Strawberry mutations for OIDC: `oidcAuthUrl` (returns redirect URL), `oidcCallback`
      (validates token, issues session)
- [ ] Implement JWT: RS256 signing, short-lived access tokens (15 min), rotating refresh tokens (7
      days)
- [ ] Implement account lockout with progressive back-off (Redis-backed per account + per IP)
- [ ] Implement HaveIBeenPwned breach detection on registration
- [ ] Implement Strawberry mutations: `register`, `login`, `logout`, `refreshToken`, `enableMfa`,
      `verifyMfa`, `registerPasskey`, `authenticatePasskey`, `oidcAuthUrl`, `oidcCallback`
- [ ] Run all tests green, then `ruff check` and `mypy`
- [ ] Create `syntek.manifest.toml` — module ID, available install options, all `SYNTEK_<MODULE>`
      settings keys with types and defaults, and `INSTALLED_APPS` entry
- [ ] Build Rust CLI installer binary — reads manifest, presents install options interactively,
      writes `SYNTEK_<MODULE>` config block to the project's `settings.py`, adds app to
      `INSTALLED_APPS`, and prints the URL pattern snippet and migration command as copy-paste next
      steps
- [ ] **Green phase:** all tests pass including security negatives
