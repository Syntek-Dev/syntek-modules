# US079 — `@syntek/mobile-logging`: React Native Observability Package

**Last Updated**: 10/03/2026 **Version**: 0.2.0 **Status**: To Do **Epic**: Epic 8 — Mobile Packages

---

## User Story

As a developer, I want a React Native observability package so that crashes go directly to Sentry or
Glitchtip for reliability, while structured logs are shipped to the Django backend (or any custom
endpoint) for enrichment and unified log storage — with a `useLogger` hook, breadcrumb tracking, and
performance monitoring, all configured via `SyntekMobileLoggingProvider`.

## MoSCoW Priority

**Must Have**

## Architecture: Hybrid Model

```text
Mobile device
  ├── Crashes & errors  →  errorBackend (direct to Sentry/Glitchtip)
  │                        Queued locally if offline; sent on next launch.
  │
  └── Structured logs   →  logBackend
        'syntek'  →  POST /api/frontend-logs/ (Django enriches + forwards to Loki etc.)
        'custom'  →  POST to any endpoint you control
        'none'    →  disabled
```

## Configuration Contract

```tsx
import { SyntekMobileLoggingProvider } from "@syntek/mobile-logging";

<SyntekMobileLoggingProvider
  // ─── Error backend (direct, for reliability) ──────────────────────────────

  errorBackend="sentry" // 'sentry' | 'console' | 'none'
  // Sentry / Glitchtip options (errorBackend: 'sentry')
  dsn={process.env.EXPO_PUBLIC_SENTRY_DSN}
  environment={process.env.EXPO_PUBLIC_ENV}
  release={process.env.EXPO_PUBLIC_RELEASE} // app version + build number
  tracesSampleRate={0.1}
  enableCrashReporting={true} // boolean — unhandled JS exceptions + native crashes
  enablePerformanceMonitoring={true} // boolean — screen loads, JS operations
  enableNetworkTracking={false} // boolean — outbound HTTP request timing (use cautiously)
  sendDefaultPii={false}
  // ─── Log backend (structured logs) ────────────────────────────────────────

  logBackend="syntek" // 'syntek' | 'custom' | 'none'
  // 'syntek' — POST to /api/frontend-logs/ with JWT attached.
  // 'custom' — POST to any endpoint you control.
  // 'none'   — disable structured log shipping.

  // Custom endpoint options (logBackend: 'custom')
  logEndpoint=""
  logHeaders={{}}
  logBatchSize={50}
  logFlushInterval={10} // number (seconds)
  // ─── Shared options ────────────────────────────────────────────────────────

  logLevel="warn" // Minimum level for logBackend. Does not affect errorBackend.
  breadcrumbLimit={100} // number — ring-buffer; oldest dropped when full.
  // Breadcrumbs are attached to crash reports automatically.

  ignoreErrors={[]} // list of strings or regex
>
  <NavigationContainer>
    <AppNavigator />
  </NavigationContainer>
</SyntekMobileLoggingProvider>;
```

**Key rules:**

- `errorBackend: 'sentry'` requires `dsn`. Missing DSN logs a warning and disables gracefully.
- `logBackend: 'syntek'` requires `@syntek/api-client` to be configured. JWT attached automatically.
- `logBackend: 'custom'` requires `logEndpoint`. Missing endpoint logs a warning and disables
  gracefully.
- `enableCrashReporting: true` installs global handlers for unhandled JS promise rejections and
  native crashes. Native crash reports are queued locally and sent on the next app launch.
- `enableNetworkTracking: true` instruments all outbound requests — ensure `sendDefaultPii: false`
  as request URLs may contain sensitive data.
- `logLevel` applies only to `logBackend`. The `errorBackend` always captures crashes.
- Offline behaviour: `logBackend: 'syntek'` and `'custom'` queue entries in device storage when
  offline and flush when connectivity is restored.

## Acceptance Criteria

```gherkin
# --- Configuration ---

Given `errorBackend` is 'sentry' and `dsn` is not provided
When the provider mounts
Then a warning is logged and the error backend is disabled — the app does not crash

Given `logBackend` is 'custom' and `logEndpoint` is empty
When the provider mounts
Then a warning is logged and the log backend is disabled — the app does not crash

# --- Crash reporting ---

Given `enableCrashReporting` is True and an unhandled JS exception occurs
When the exception propagates
Then it is captured and sent to errorBackend with stack trace, breadcrumbs, release, and environment

Given `enableCrashReporting` is True and a native crash occurs
When the app restarts after the crash
Then the queued crash report is sent to errorBackend on the next launch

Given `enableCrashReporting` is False
When an unhandled exception occurs
Then the observability package does not capture it

# --- Glitchtip compatibility ---

Given `dsn` points to a Glitchtip instance
When a crash is captured
Then it appears in Glitchtip — no code change needed versus Sentry SaaS

# --- Performance monitoring ---

Given `enablePerformanceMonitoring` is True
When a screen transition occurs and renders completely
Then a performance span is recorded and sampled per `tracesSampleRate`

Given `enableNetworkTracking` is True and an outbound HTTP request is made
When the request completes
Then URL, method, status code, and duration are recorded as a span

Given `enableNetworkTracking` is False
When an outbound request is made
Then no network span is recorded

# --- logBackend: syntek ---

Given `logBackend` is 'syntek' and the user is authenticated
When useLogger emits an entry at or above `logLevel`
Then it is POSTed to /api/frontend-logs/ with the JWT in the Authorization header

Given the device is offline and logBackend is 'syntek'
When a log entry is emitted
Then it is queued in local device storage and sent when connectivity is restored

# --- logBackend: custom ---

Given `logBackend` is 'custom'
When useLogger emits an entry
Then it is POSTed to `logEndpoint` with `logHeaders`

# --- logLevel ---

Given `logLevel` is 'warn'
When useLogger emits a 'debug' entry
Then it is suppressed — not sent to logBackend

Given `logLevel` is 'warn' and a native crash occurs
When the crash is captured
Then it is still sent to errorBackend — logLevel never suppresses crash reporting

# --- Breadcrumbs ---

Given `breadcrumbLimit` is 100 and 101 breadcrumbs have been recorded
When a crash occurs
Then the 100 most recent breadcrumbs are attached; the oldest is dropped

# --- ignoreErrors ---

Given `ignoreErrors` includes 'Network request failed'
When that error occurs
Then it is not sent to errorBackend

# --- useLogger hook ---

Given a component calls `const { error } = useLogger('PaymentScreen')`
When `error('Card declined', { code: 'insufficient_funds' })` is called
Then the entry is sent to logBackend with 'PaymentScreen' as context
```
