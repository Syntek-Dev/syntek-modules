## User Story

As a developer, I want a runtime feature flag module so that I can roll out features to specific
users, tenants, or percentages of traffic without code deployments — with a static flag registry in
settings, dynamic DB overrides, deterministic percentage rollout, a GraphQL API, Redis caching, and
audit trail — all controlled from `settings.py` without touching module source.

## MoSCoW Priority

**Must Have**

## Design Decisions

### Static registry + dynamic overrides

Flags have two layers:

1. **Static definitions** — declared in `SYNTEK_FLAGS['FLAGS']` in `settings.py`. These are the
   baseline state of every flag. Consuming projects define all their flags here so the registry is
   always complete and auditable via version control.
2. **Dynamic overrides** — stored in the `FeatureFlag` database model. A tenant admin (or the
   GraphQL API) writes a runtime override that takes precedence over the static default.

This means flags work at startup even with no DB record, and can be changed at runtime without a
deployment.

### Evaluation hierarchy

Flags are evaluated in strict precedence order (highest first):

```text
per-user override (FeatureFlagUserOverride)
    ↓ (if not set)
per-tenant override (FeatureFlag DB record: enabled true/false)
    ↓ (if not set or tenant has no record)
global percentage rollout (static FLAGS definition + optional DB override)
    ↓ (if percentage is 0)
global enabled/disabled (static FLAGS definition default)
```

A tenant-level `enabled: False` is an absolute kill switch — it overrides everything including
per-user overrides. This is intentional: tenant admins must be able to disable a flag for their
entire organisation regardless of what individual user overrides exist.

### Deterministic percentage rollout

Percentage rollout uses HMAC-SHA256 over `flag_name + user_id + ROLLOUT_HASH_SALT` to produce a
stable 0–99 bucket for each (flag, user) pair. A user in bucket < rollout_percentage sees the flag
enabled. The same user always gets the same bucket for the same flag — rollout is sticky.

Changing `ROLLOUT_HASH_SALT` reshuffles all buckets, effectively re-randomising the rollout. This is
the escape hatch if a cohort needs to be changed without changing the percentage.

### Caching

Flag evaluations are cached in Redis per (tenant, user, flag_name). TTL is deliberately short
(default 60 seconds) so that runtime changes propagate quickly without requiring a restart. Cache is
invalidated immediately on any DB write to a flag record.

### Audit

All flag create/update/delete operations emit an audit entry via syntek-audit (US013) when
`AUDIT_ENABLED: True`. The entry records: actor, flag name, old state, new state, timestamp, tenant.
Required for compliance — toggling a flag that gates a billable feature must be auditable.

## Configuration Contract

All behaviour is controlled via `SYNTEK_FLAGS` in `settings.py`. Validated at startup in
`AppConfig.ready()`.

```python
SYNTEK_FLAGS = {

    # --- Static flag registry ---
    # All flags that exist in the system must be declared here.
    # Flags not in this dict cannot be evaluated — is_enabled() returns False and logs a warning.
    # Dynamic DB overrides take precedence over these defaults at runtime.
    'FLAGS': {
        # Format per flag:
        # 'flag_name': {
        #     'enabled': False,              # bool — global default when no DB override exists
        #     'description': '',             # str — shown in admin UI and GraphQL schema
        #     'rollout_percentage': 0,       # int (0–100) — percentage of users who see the flag enabled
        #                                    #   0 = no one (unless per-user override); 100 = everyone
        #                                    #   Only evaluated when enabled: True at the active layer
        #     'allowed_tenants': [],         # list[str] — tenant IDs/slugs; empty = all tenants allowed
        #                                    #   If non-empty, the flag is unavailable for unlisted tenants
        #     'allowed_users': [],           # list[str] — user IDs; empty = any user (within enabled tenants)
        #                                    #   If non-empty, acts as an explicit allowlist on top of percentage
        # }
        #
        # Example:
        # 'new_checkout_flow': {
        #     'enabled': True,
        #     'description': 'Redesigned checkout with one-page layout',
        #     'rollout_percentage': 20,
        #     'allowed_tenants': [],
        #     'allowed_users': [],
        # },
        # 'beta_reporting': {
        #     'enabled': True,
        #     'description': 'New reporting dashboard — beta only',
        #     'rollout_percentage': 100,
        #     'allowed_tenants': ['beta-tenant-1', 'beta-tenant-2'],
        #     'allowed_users': [],
        # },
    },

    # --- Percentage rollout ---
    'ROLLOUT_HASH_SALT': env('FLAG_ROLLOUT_SALT', default=''),
    # str — mixed into HMAC-SHA256 hash for deterministic bucket assignment.
    # Changing this reshuffles all rollout buckets (re-randomises cohorts).
    # Use a stable random string in production; rotate only when intentionally re-randomising.

    # --- Caching ---
    'CACHE_ENABLED': True,          # bool — cache flag evaluations in syntek-cache (Redis)
    'CACHE_TTL': 60,                # int (seconds) — short TTL so runtime changes propagate quickly
                                     # Invalidated immediately on any DB write

    # --- Audit ---
    'AUDIT_ENABLED': True,          # bool — write audit entries on flag changes via syntek-audit
                                     # Silently skipped if syntek-audit is not in INSTALLED_APPS

    # --- GraphQL ---
    'EXPOSE_VIA_GRAPHQL': True,     # bool — expose featureFlags, flagStatus queries and mutations
    'REQUIRE_ADMIN_FOR_WRITE': True, # bool — create/update/delete mutations require manage_flags permission
}
```

**Key rules:**

- `is_enabled()` for an unknown flag (not in `FLAGS`) returns `False` and emits a warning via
  syntek-logging. It never raises — callers should not need to handle exceptions for flag checks.
- `ROLLOUT_HASH_SALT` should be set in production. An empty salt still works but reduces rollout
  isolation between different deployments sharing the same user IDs.
- `CACHE_ENABLED: True` requires syntek-cache (US077). Silently skipped if not installed.
- `AUDIT_ENABLED: True` silently skips if syntek-audit (US013) is not in `INSTALLED_APPS` — flags
  work without audit, audit is an enhancement not a hard dependency.
- `allowed_tenants` in the static definition is a design-time allowlist. Dynamic per-tenant
  overrides in the DB can still enable/disable the flag for any tenant regardless — the DB record is
  the runtime source of truth; the static definition is the default.

## Python API

```python
from syntek_flags import is_enabled

# Evaluate a flag for a specific user and tenant
if is_enabled('new_checkout_flow', user=request.user, tenant=request.tenant):
    return new_checkout_response()
else:
    return legacy_checkout_response()

# Evaluate without a user (anonymous or background task context)
if is_enabled('maintenance_mode', tenant=request.tenant):
    return maintenance_response()

# Evaluate globally (no tenant or user context — e.g. server startup)
if is_enabled('enable_new_migrations'):
    run_new_migration_path()
```

```python
from syntek_flags import register_flags

# Module-level flag registration (called in AppConfig.ready())
# Merges into the global registry — raises ImproperlyConfigured on collision
register_flags({
    'syntek_auth_passkey_login': {
        'enabled': False,
        'description': 'Passkey (WebAuthn) login support',
        'rollout_percentage': 0,
    },
})
```

## GraphQL API

```graphql
# Query — all flags visible to the current user/tenant with their evaluated state
query FeatureFlags {
  featureFlags {
    name
    enabled # evaluated result for the current user+tenant (true/false)
    description
    rolloutPercentage
    isOverridden # true if a DB override exists for this tenant
  }
}

# Query — evaluate a single flag for the current user/tenant
query FlagStatus($name: String!) {
  flagStatus(name: $name) {
    name
    enabled
  }
}

# Mutation — create or update a flag override for the current tenant
mutation UpdateFlag($name: String!, $enabled: Boolean!, $rolloutPercentage: Int) {
  updateFlag(name: $name, enabled: $enabled, rolloutPercentage: $rolloutPercentage) {
    name
    enabled
  }
}

# Mutation — delete the tenant override (reverts to static definition default)
mutation ResetFlag($name: String!) {
  resetFlag(name: $name) {
    name
    enabled
  }
}

# Mutation — set a per-user override for a flag (admin only)
mutation SetUserFlagOverride($name: String!, $userId: ID!, $enabled: Boolean!) {
  setUserFlagOverride(name: $name, userId: $userId, enabled: $enabled) {
    name
    userId
    enabled
  }
}
```

## Acceptance Criteria

```gherkin
# --- Flag registry ---

Given FLAGS declares 'new_checkout_flow' with enabled: False
When AppConfig.ready() runs
Then the flag is in the registry with correct defaults

Given is_enabled('unknown_flag', user=user, tenant=tenant) is called
When it runs
Then False is returned and a warning is logged

Given a module calls register_flags with a key that collides with FLAGS
When AppConfig.ready() runs
Then ImproperlyConfigured is raised naming the conflicting key

# --- Evaluation hierarchy ---

Given 'new_checkout' has rollout_percentage: 20 and a user is in the top 80% bucket
When is_enabled('new_checkout', user=user, tenant=tenant) is called
Then False is returned

Given the same user has a per-user override of enabled: True
When is_enabled('new_checkout', user=user, tenant=tenant) is called
Then True is returned — per-user override wins over percentage

Given a tenant has a DB record setting 'new_checkout' enabled: False
When is_enabled('new_checkout', user=user, tenant=tenant) is called for any user of that tenant
Then False is returned — tenant kill switch wins over per-user override

Given 'beta_flag' has allowed_tenants: ['tenant-a'] and tenant-b evaluates it
When is_enabled('beta_flag', user=user, tenant=tenant_b) is called
Then False is returned — tenant not in allowlist

# --- Deterministic percentage rollout ---

Given 'flag_x' has rollout_percentage: 20 and 10,000 different users evaluate it
When results are collected
Then approximately 20% return True (within ±2% tolerance)

Given the same user evaluates 'flag_x' 100 times
When all results are collected
Then all 100 return the same value — rollout is sticky

Given ROLLOUT_HASH_SALT changes from 'salt-a' to 'salt-b'
When the same user evaluates the same flag
Then the bucket assignment may differ — cohort re-randomised

# --- Caching ---

Given CACHE_ENABLED is True and is_enabled is called twice within CACHE_TTL
When the second call runs
Then Redis is read — the DB is not queried a second time

Given a tenant DB record is updated
When is_enabled is called immediately after
Then the updated value is returned — cache was invalidated on write

Given CACHE_ENABLED is True and syntek-cache is not installed
When AppConfig.ready() runs
Then a startup warning is logged and is_enabled works without caching

# --- Audit ---

Given AUDIT_ENABLED is True and a tenant admin calls updateFlag
When the mutation runs
Then an audit entry is created with: actor, flag name, old state, new state, tenant, timestamp

Given AUDIT_ENABLED is True and syntek-audit is not in INSTALLED_APPS
When a flag is updated
Then no error is raised — audit write is silently skipped

Given AUDIT_ENABLED is False
When a flag is updated
Then no audit entry is created

# --- GraphQL ---

Given EXPOSE_VIA_GRAPHQL is True and the current user is in the 20% rollout for 'new_checkout'
When featureFlags is queried
Then 'new_checkout' appears with enabled: true

Given REQUIRE_ADMIN_FOR_WRITE is True and a non-admin calls updateFlag
When the mutation runs
Then PermissionDenied is returned — flag is not changed

Given resetFlag is called for a flag with a tenant DB override
When the mutation runs
Then the override is deleted and is_enabled returns the static definition default

Given setUserFlagOverride sets a user to enabled: True for a disabled flag
When is_enabled is called for that user
Then True is returned — per-user override in effect
```
