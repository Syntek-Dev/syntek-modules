# US017 — `syntek-flags`: Feature Flags Module

**Last Updated**: 06/03/2026 **Version**: 0.1.0 **Status**: To Do **Epic**: Epic 3 — Core Backend
Modules

---

## User Story

As a developer, I want runtime feature flags so that I can roll out features to specific tenants,
users, or percentages of traffic without code deployments.

## MoSCoW Priority

**Must Have**

## Acceptance Criteria

```gherkin
Given a flag `new_checkout` is configured with 20% rollout
When `is_enabled("new_checkout", user)` is called for 10,000 different users
Then approximately 20% return True, consistently for the same user on repeat calls

Given a flag is disabled at the tenant level
When any user of that tenant calls `is_enabled`
Then it returns False regardless of any user-level or percentage override

Given an admin updates a flag value in the Django admin
When the next request calls `is_enabled`
Then the new value is used — no deployment or restart required

Given `syntek-audit` is installed
When a flag value is changed
Then an audit entry is written recording the actor, flag name, old value, and new value
```

## Dependencies

- US016 — `syntek-settings`
- US013 — `syntek-audit`

- [ ] Implement `is_enabled(flag, user, tenant)` with hash-based deterministic rollout
- [ ] Implement tenant-level override taking precedence over percentage
- [ ] Implement per-user override for individual targeting
- [ ] Implement Redis caching of flag values per tenant (invalidated on change)
- [ ] Implement audit log integration on flag change
- [ ] Implement Strawberry query: `featureFlags` returning all active flags for current user/tenant
- [ ] Implement Strawberry mutations: `createFlag`, `updateFlag`, `deleteFlag`

      steps
