## User Story

As a developer, I want a live location tracking module so that I can broadcast a worker's or
vehicle's real-time position to authorised viewers, share a tracking link with end users (customers
tracking a delivery, owners tracking a dog walker), trigger geofence alerts, and store position
history ŌĆö all configurable for any tracking use case.

## Architecture

`syntek-tracking` is domain-agnostic. A "tracker" can be a delivery driver, a dog walker, a field
engineer, or any mobile worker. A "viewer" is anyone with the tracking link. The module has no
business-type assumptions.

Position data flows through three tiers:

**Tier 1 ŌĆö Live stream (Redis + Django Channels)** Every GPS ping from the tracker's device is
published to a Redis PubSub channel (`tracking_{session_id}`). No database write occurs at this
tier. A Redis hash (`tracking:last:{session_id}`) stores the most recent position so that viewers
receive it instantly on WebSocket connect. This tier is always active.

**Tier 2 ŌĆö Sampled history (TrackingPoint, optional TimescaleDB hypertable)** `PING_STORAGE_MODE`
controls whether pings are written to the `TrackingPoint` table. In `'sampled'` mode, only 1 in
`PING_SAMPLE_RATE` pings are persisted ŌĆö at the default interval (15s) and sample rate (3), one row
is written every 45 seconds. Raw position records are retained for `POSITION_HISTORY_RETENTION_DAYS`
(default 90 days), then dropped by a Celery Beat task or ŌĆö when `USE_TIMESCALEDB = True` ŌĆö by a
TimescaleDB automatic drop policy. This tier enables live trail replay during the session and
post-session replay within the retention window.

**Tier 3 ŌĆö Permanent route summary (RouteSnapshot, PostGIS)** On session completion,
`endTrackingSession` computes a `RouteSnapshot`: all sampled `TrackingPoint` records for the session
are read, their geometries assembled into a PostGIS `LineString`, and simplified using the
Ramer-Douglas-Peucker algorithm (`tolerance = ROUTE_SIMPLIFICATION_TOLERANCE`, default 0.0001┬░ Ōēł 11
m). The snapshot stores `distance_metres`, `duration_seconds`, `avg_speed_kmh`, `max_speed_kmh`, and
`point_count`. The snapshot is retained indefinitely and is not subject to the Tier 2 retention
window. On GDPR erasure request, the geometry is set to null and metrics zeroed. `RouteSnapshot` is
standard PostgreSQL (not a time-series table ŌĆö no hypertable needed).

## Configuration

```python
SYNTEK_TRACKING = {
    # Position update interval ŌĆö how frequently the tracker's device sends updates (seconds)
    'POSITION_UPDATE_INTERVAL_SECONDS': 15,

    # Tier 1 ŌĆö Redis live stream (Django Channels)
    # Every ping is published to Redis PubSub. No DB write per ping at this tier.
    # Redis hash stores last known position for instant viewer connect.
    'CHANNEL_LAYER': 'default',   # Must match Django Channels config

    # Tier 2 ŌĆö Sampled TrackingPoint writes
    # PING_STORAGE_MODE: 'all' = every ping stored | 'sampled' = 1 in N | 'none' = stream only, no DB
    'PING_STORAGE_MODE': 'sampled',
    'PING_SAMPLE_RATE': 3,           # Store 1 in every N pings (e.g. 3 ŌåÆ one write every 45s at 15s interval)
    'POSITION_HISTORY_RETENTION_DAYS': 90,

    # Tier 3 ŌĆö RouteSnapshot (permanent PostGIS summary, computed on session end)
    'ROUTE_SUMMARY_ENABLED': True,
    'ROUTE_SIMPLIFICATION_TOLERANCE': 0.0001,   # Ramer-Douglas-Peucker tolerance (degrees, Ōēł 11m)
    'ROUTE_SUMMARY_REQUIRE_POSTGIS': True,       # Raises ImproperlyConfigured if PostGIS unavailable

    # Tracking links ŌĆö shareable links for end-user viewers
    'TRACKING_LINK_ENABLED': True,
    'TRACKING_LINK_EXPIRY_HOURS': 24,
    'TRACKING_LINK_BASE_URL': env('TRACKING_LINK_BASE_URL'),

    # Geofencing
    'GEOFENCE_ENABLED': True,
    'GEOFENCE_NOTIFICATION_ON_ENTER': True,
    'GEOFENCE_NOTIFICATION_ON_EXIT': True,

    # Privacy ŌĆö whether viewers can see exact position or just a fuzzy area
    'VIEWER_POSITION_PRECISION': 'exact',   # 'exact' | 'approximate' (blurs to ~500m radius)

    # TimescaleDB ŌĆö for Tier 2 TrackingPoint hypertable
    # Requires syntek-timescaledb (US118). When True, retention is handled by TimescaleDB drop policy.
    'USE_TIMESCALEDB': False,
    'TIMESCALEDB_CHUNK_INTERVAL': '1 day',
    'TIMESCALEDB_COMPRESS_AFTER': '7 days',
}
```

## Acceptance Criteria

```gherkin
Scenario 1: Session creation with tracking link
Given a tracking session is started for a worker
When startTrackingSession(workerId, sessionType) is called
Then a TrackingSession is created with a unique ULID, a shareable tracking link is generated (if TRACKING_LINK_ENABLED = True), and the session is ready to receive position updates

Scenario 2: Position update storage and broadcast
Given a tracker sends a position update
When updatePosition(sessionId, lat, lng, heading, speed) mutation is called
Then a TrackingPoint is stored and the new position is broadcast to all subscribed WebSocket viewers of that session in under 2 seconds

Scenario 3: Viewer receives current position on connect
Given a viewer has the tracking link
When they connect to the WebSocket channel
Then they receive the current position immediately on connect, then receive live updates as the tracker moves

Scenario 4: Approximate position blurring
Given VIEWER_POSITION_PRECISION = 'approximate'
When a viewer queries the tracker's position
Then the coordinates are blurred to approximately 500 metres from the actual position

Scenario 5: Geofence enter notification
Given a Geofence is defined around a destination
When the tracker's position enters the geofence boundary
Then a geofence.entered event is fired via syntek-bus and a notification is dispatched via syntek-notifications

Scenario 6: Session completion
Given a tracking session ends
When endTrackingSession(sessionId) is called
Then the session status transitions to completed, the tracking link becomes invalid, and the position history is retained for POSITION_HISTORY_RETENTION_DAYS

Scenario 7: Position history retention
Given POSITION_HISTORY_RETENTION_DAYS = 30
When the retention cleanup task runs
Then TrackingPoint records older than 30 days are deleted

Scenario 8: TimescaleDB hypertable conversion
Given USE_TIMESCALEDB = True and syntek-timescaledb is installed
When the TrackingPoint migration runs
Then the syntek_tracking_trackingpoint table is converted to a TimescaleDB hypertable on the recorded_at column, with a compression policy for chunks older than 7 days and a retention policy matching POSITION_HISTORY_RETENTION_DAYS

Scenario 9: Sampled ping storage (Tier 2)
Given PING_STORAGE_MODE = 'sampled' and PING_SAMPLE_RATE = 3
When 9 position updates are received
Then exactly 3 TrackingPoint records are stored (1 in 3 pings persisted) and all 9 positions are broadcast to viewers via Redis PubSub

Scenario 10: Last-known position from Redis on viewer connect
Given a tracker has sent 20 position updates during an active session
When a new viewer connects to the WebSocket channel
Then they receive the last known position from the Redis hash immediately, without querying the database

Scenario 11: RouteSnapshot computed on session end (Tier 3)
Given a completed tracking session with TrackingPoint records
When endTrackingSession is called
Then a RouteSnapshot is created with a simplified PostGIS LineString, distance_metres, duration_seconds, avg_speed_kmh, max_speed_kmh, and point_count

Scenario 12: Stream-only mode (PING_STORAGE_MODE = 'none')
Given PING_STORAGE_MODE = 'none'
When 100 position updates are received
Then no TrackingPoint records are stored, all positions are broadcast via Redis PubSub, and no RouteSnapshot is computed on session end
```
