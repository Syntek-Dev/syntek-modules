# US058 — `@syntek/mobile-auth`: Mobile Authentication

**Last Updated**: 10/03/2026 **Version**: 0.2.0 **Status**: To Do **Epic**: Epic 8 — Mobile Packages

---

## User Story

As a developer, I want a mobile authentication package so that I can add biometric login, social
OAuth, deep link handling, and passkey support to React Native applications backed by `syntek-auth`
— with full control over enabled flows, biometric prompts, session timeout, storage backend, and
screen customisation configured via `SyntekMobileAuthProvider` props.

**Must Have**

## Configuration Contract

All behaviour is configured via props on `SyntekMobileAuthProvider`. No `.env` config.

```tsx
import { SyntekMobileAuthProvider } from "@syntek/mobile-auth";

<SyntekMobileAuthProvider
  // --- Enabled flows ---
  enableBiometric={true} // boolean — enable Face ID / Touch ID / fingerprint unlock
  enablePasskeys={true} // boolean — enable platform passkey (WebAuthn) registration and login
  enableOAuth={true} // boolean — show OAuth provider buttons (Expo AuthSession)
  enableMagicLink={false} // boolean — enable magic link / email OTP deep link flow
  enableRegistration={true} // boolean — show registration screen
  // --- Biometric options ---
  biometricPrompt="Sign in to your account" // string — prompt text shown in the biometric dialog
  biometricFallback="Use password" // string — fallback button label if biometric fails
  biometricReauthAfter={900} // number (seconds) — idle time before biometric re-prompt
  //   0 = always require on app foreground

  // --- Storage ---
  storageBackend="secure_store" // 'secure_store' | 'keychain'
  // 'secure_store' — Expo SecureStore (cross-platform, recommended)
  // 'keychain'     — native iOS Keychain / Android Keystore directly

  // --- Session ---
  sessionTimeout={0} // number (seconds) — 0 = no idle timeout; positive = lock after inactivity
  // --- Redirects ---
  postLoginScreen="Home" // string — navigation target after login (React Navigation screen name)
  postLogoutScreen="Login" // string
  // --- Custom screens (full replacement) ---
  screens={{
    Login: CustomLoginScreen, // optional — replace the default login screen component entirely
    Register: CustomRegisterScreen,
    MFA: CustomMFAScreen,
    Biometric: CustomBiometricScreen,
  }}

  // --- GraphQL endpoint ---
  // Inherited from @syntek/api-client — not set here directly.
>
  <NavigationContainer>
    <AppNavigator />
  </NavigationContainer>
</SyntekMobileAuthProvider>;
```

**Key rules:**

- `storageBackend: 'secure_store'` is recommended for cross-platform compatibility (iOS + Android).
- `storageBackend: 'keychain'` gives direct access to native secure storage but requires additional
  native configuration for each platform.
- `biometricReauthAfter: 0` means biometric is prompted every time the app comes to foreground with
  an active session.
- Custom `screens` receive the same props as the default screens — they replace presentation only;
  auth logic remains in the package.
- `enableBiometric` is a client-side toggle. If the device has no biometric hardware enrolled,
  biometric prompts are automatically skipped regardless of this setting.

## Acceptance Criteria

```gherkin
# --- Enabled flows ---

Given `enableBiometric` is False
When the login screen renders
Then no biometric prompt option is shown

Given `enableOAuth` is False
When the login screen renders
Then no OAuth provider buttons are rendered

Given `enableRegistration` is False
When the login screen renders
Then no register link or screen is accessible

Given `enableMagicLink` is True
When the user requests a magic link and opens the email link
Then the deep link is intercepted, the token is validated with the backend, and the user is logged in

# --- Biometric ---

Given `enableBiometric` is True and biometrics are enrolled on the device
When the user opens the app with a valid session token stored in the secure storage backend
Then Face ID / Touch ID / fingerprint is used to unlock the session without re-entering credentials

Given `biometricReauthAfter` is 900 and the app has been in the background for 800 seconds
When the app is brought to the foreground
Then no biometric prompt is shown — the session is still within the re-auth window

Given `biometricReauthAfter` is 900 and the app has been in the background for 1000 seconds
When the app is brought to the foreground
Then biometric re-authentication is required before the session is restored

Given biometric authentication fails and `biometricFallback` is 'Use password'
When the biometric prompt is dismissed
Then a fallback button labelled 'Use password' is shown and leads to the password login screen

Given the device has no biometric hardware or no biometrics enrolled
When the login screen renders with `enableBiometric: true`
Then biometric option is not shown — the device capability check overrides the setting

# --- Storage ---

Given `storageBackend` is 'secure_store'
When a session token is persisted after login
Then it is stored using Expo SecureStore — not AsyncStorage

Given `storageBackend` is 'keychain'
When a session token is persisted after login
Then it is stored in the native iOS Keychain / Android Keystore directly

# --- Session timeout ---

Given `sessionTimeout` is 300 (5 minutes) and the user has been idle for 301 seconds
When the user next interacts with the app
Then the session is considered expired, tokens are cleared, and the user is routed to postLogoutScreen

# --- OAuth ---

Given a social OAuth button is tapped
When the OAuth flow completes in the Expo AuthSession in-app browser
Then the deep link returns the auth code, the backend mutation exchanges it, and the session is established

# --- Passkeys ---

Given the user registers a passkey
When the WebAuthn ceremony completes on device
Then the passkey credential is stored in the device's platform authenticator and usable for future logins

# --- Custom screens ---

Given a custom Login screen is provided in `screens`
When the user navigates to the login screen
Then the custom component renders — auth mutations and session logic are still handled by the package

# --- Redirects ---

Given `postLoginScreen` is 'Dashboard'
When the user successfully logs in
Then the navigator routes to the 'Dashboard' screen
```
