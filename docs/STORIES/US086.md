## User Story

As a developer, I want a user profile module so that I can define exactly what my application
collects about its users — customers, clients, members, patients, students, or any other domain
person — with a schema-driven custom field system, Cloudinary avatar support, a configurable status
lifecycle, field-level encryption for sensitive data, and a full GraphQL API — all controlled from
`settings.py` without touching module source.

## MoSCoW Priority

**Must Have**

## Design Decisions

### Separation from `syntek-auth`

`syntek-auth` (US009) owns the **identity record**: login credentials, password hash, MFA state,
session tokens. It does not know or care what your application calls its users or what data it
collects about them.

`syntek-users` owns the **domain profile**: the fields your application actually needs (name,
address, date of birth, dietary requirements, NHS number, membership tier, etc.). The profile is
linked to the auth `User` via a OneToOne relationship.

This separation means:

- The auth module never changes when business requirements change (e.g. adding an address field)
- The profile module can be replaced with a custom model in extreme cases — the auth module is
  unaffected
- Consuming projects never need to subclass or monkey-patch the auth `User` model

### Custom fields via validated JSONField

Profile fields declared in `PROFILE_SCHEMA` are stored in a single `JSONField` column
(`custom_fields`) on the `UserProfile` model. This avoids code generation and dynamic migrations
while keeping the API clean. The module validates types, required constraints, and field-level
encryption rules on every write.

The GraphQL layer introspects `PROFILE_SCHEMA` at startup and exposes each declared field as a
first-class typed scalar — consumers query `profile { firstName, nhsNumber, membershipTier }` not
`profile { customFields }`.

Built-in fields (`status`, `avatar_url`, `user`, timestamps) are concrete model columns — not in
`custom_fields`.

### Field-level encryption

Fields declared with `'encrypted': True` in `PROFILE_SCHEMA` are encrypted at rest using AES-256-GCM
via `syntek-pyo3` before the value is written to `custom_fields`. Decryption is transparent on read.
This requires `syntek-pyo3` and the `SYNTEK_FIELD_KEY_*` environment variables defined by
`syntek-graphql-crypto`.

### Status lifecycle

The status field is a string column with a configurable set of valid values and configurable allowed
transitions. Invalid transitions are rejected at the model layer before any DB write. The default
status on profile creation is the first entry in `SYNTEK_USERS['STATUSES']`.

## Configuration Contract

All behaviour is controlled via `SYNTEK_USERS` in `settings.py`. Validated at startup in
`AppConfig.ready()`.

```python
SYNTEK_USERS = {

    # --- Profile display label ---
    # Controls how this module's records are labelled in the Django admin, GraphQL schema
    # docs, and error messages. Does not affect the underlying model or table name.
    'PROFILE_LABEL': 'User Profile',    # str — e.g. 'Customer', 'Member', 'Patient', 'Client'
    'PROFILE_LABEL_PLURAL': 'User Profiles',  # str — plural form for list contexts

    # --- Profile schema (custom fields) ---
    # Declares every custom field collected by this application.
    # Fields are stored in a validated JSONField and exposed as first-class GraphQL scalars.
    # Built-in fields (status, avatar_url, user FK, created_at, updated_at) are always present.
    #
    # Field definition format:
    # 'field_name': {
    #     'type': str,       # REQUIRED — 'str' | 'text' | 'int' | 'float' | 'bool'
    #                        #            | 'date' | 'datetime' | 'email' | 'phone'
    #                        #            | 'url' | 'json' | 'choice'
    #     'required': bool,  # REQUIRED — True = must be set on creation; False = nullable
    #     'default': Any,    # Optional — value used when not provided; must pass type coercion
    #     'max_length': int, # Optional — str/text fields only; no max if omitted
    #     'choices': list,   # Required for 'choice' type — list of (value, label) tuples
    #     'encrypted': bool, # Optional — True = AES-256-GCM encrypted at rest via syntek-pyo3
    #                        #            Requires SYNTEK_FIELD_KEY_* env vars
    #     'description': str,# Optional — label shown in admin UI and GraphQL schema docs
    #     'searchable': bool, # Optional — include in full-text search (syntek-search integration)
    # }
    'PROFILE_SCHEMA': {
        # Examples (uncomment and customise for your application):
        # 'first_name': {
        #     'type': 'str', 'required': True, 'max_length': 100, 'description': 'First name',
        # },
        # 'last_name': {
        #     'type': 'str', 'required': True, 'max_length': 100, 'description': 'Last name',
        # },
        # 'date_of_birth': {
        #     'type': 'date', 'required': False, 'description': 'Date of birth',
        # },
        # 'address_line_1': {'type': 'str', 'required': False, 'max_length': 255},
        # 'address_line_2': {'type': 'str', 'required': False, 'max_length': 255},
        # 'city':           {'type': 'str', 'required': False, 'max_length': 100},
        # 'postcode':        {'type': 'str', 'required': False, 'max_length': 10},
        # 'nhs_number': {
        #     'type': 'str', 'required': False, 'max_length': 10, 'encrypted': True,
        #     'description': 'NHS number',
        # },
        # 'dietary_requirements': {'type': 'text', 'required': False},
        # 'membership_tier': {
        #     'type': 'choice',
        #     'choices': [('standard', 'Standard'), ('premium', 'Premium'), ('vip', 'VIP')],
        #     'required': False, 'default': 'standard',
        # },
    },

    # --- Status lifecycle ---
    'STATUSES': ['active', 'inactive', 'archived'],
    # list[str] — REQUIRED; first entry is the default status on profile creation.
    # Define any statuses that make sense for your domain:
    # e.g. ['pending_approval', 'active', 'suspended', 'deactivated'] for a membership platform
    # e.g. ['prospective', 'active', 'lapsed', 'archived'] for a client CRM
    # e.g. ['enrolled', 'active', 'graduated', 'withdrawn'] for an education platform

    'STATUS_TRANSITIONS': {
        # dict[str, list[str]] — allowed transitions from each status.
        # A status not listed here can transition to anything (unrestricted).
        # An empty list makes the status a terminal state (no transitions out).
        # Example:
        # 'active':   ['inactive', 'archived'],
        # 'inactive': ['active', 'archived'],
        # 'archived': [],    # terminal — cannot be reactivated (use soft-delete + new record)
    },

    # --- Avatar ---
    'AVATAR_ENABLED': True,             # bool — enable avatar upload via syntek-media (Cloudinary)
                                         # Silently disabled with a warning if syntek-media is not installed
    'AVATAR_MAX_SIZE_BYTES': 5_242_880, # int — 5 MB default
    'AVATAR_ALLOWED_FORMATS': ['jpeg', 'png', 'webp'],  # list[str]
    'AVATAR_TRANSFORM': {               # dict — Cloudinary transformation applied on upload
        'width': 400,                    # Crop to a square, face-centered, at 400×400px
        'height': 400,
        'crop': 'fill',
        'gravity': 'face',
    },

    # --- Soft delete ---
    'SOFT_DELETE': True,                # bool — 'archived' profiles are soft-deleted:
                                         # hidden from default querysets, accessible with manager flag
                                         # False = no soft delete; archived records remain fully visible

    # --- GraphQL ---
    'EXPOSE_VIA_GRAPHQL': True,         # bool — expose userProfile, userProfiles queries and mutations
    'SEARCH_ENABLED': True,             # bool — expose userProfiles(search: String) for text search
                                         # Uses searchable: True fields in PROFILE_SCHEMA
    'REQUIRE_ADMIN_FOR_USER_WRITE': True,
    # bool — True = only users with 'manage_users' permission can update another user's profile
    # Users can always update their own profile regardless of this setting
    'EXPOSE_SENSITIVE_TO_SELF': True,
    # bool — True = users can read their own encrypted fields
    # False = encrypted fields are only accessible to admins (useful for e.g. internal reference numbers)
}
```

**Key rules:**

- `PROFILE_SCHEMA` is validated at startup. A field with `required: True` and no `default` must be
  provided on profile creation — validated in the `createUserProfile` mutation, not at the DB layer.
- `encrypted: True` fields require `syntek-pyo3` (US007) and a `SYNTEK_FIELD_KEY_PROFILE` env var.
  Missing dependencies raise `ImproperlyConfigured` at startup.
- `STATUSES` must contain at least one entry. `STATUS_TRANSITIONS` entries that reference unknown
  statuses raise `ImproperlyConfigured`.
- `AVATAR_ENABLED: True` requires `syntek-media` (US030). Silently disabled with a warning if not
  installed — avatar-related GraphQL fields return null.
- `SOFT_DELETE: True` means the `UserProfile` manager's default queryset excludes archived profiles.
  Use `UserProfile.all_objects.filter(status='archived')` to access them.
- The GraphQL type name for the profile is derived from `PROFILE_LABEL` (spaces removed,
  PascalCase). E.g. `'Customer Profile'` → `CustomerProfileType`. This controls what consuming
  projects see in their GraphQL schema.

## Python API

```python
from syntek_users import get_profile, create_profile, update_profile, set_status
from syntek_users.schema import get_field_value, set_field_value

# Read a profile for the current user
profile = get_profile(user=request.user)

# Create a profile for a user (called after registration)
profile = create_profile(user=user, fields={
    'first_name': 'Alice',
    'last_name': 'Smith',
    'membership_tier': 'standard',
})
# Raises: FieldValidationError (required field missing, type mismatch, invalid choice)
# Raises: ProfileAlreadyExistsError (user already has a profile)

# Update specific fields
update_profile(profile=profile, fields={
    'postcode': 'SW1A 1AA',
    'dietary_requirements': 'Vegan',
})

# Transition status
set_status(profile=profile, new_status='inactive', actor=request.user)
# Raises: InvalidStatusTransitionError if transition is not in STATUS_TRANSITIONS

# Read a single declared field (handles decryption transparently)
nhs_number = get_field_value(profile, 'nhs_number')

# Write a single field (handles encryption transparently)
set_field_value(profile, 'nhs_number', '123 456 7890', actor=request.user)
```

## GraphQL API

The GraphQL type name, field names, and descriptions are driven by `PROFILE_SCHEMA` and
`PROFILE_LABEL`. The examples below use the default label.

```graphql
# Query — current user's own profile
query MyProfile {
  myProfile {
    status
    avatarUrl
    createdAt
    # Custom fields from PROFILE_SCHEMA are first-class scalars:
    firstName
    lastName
    membershipTier
    # encrypted: True fields are returned as null unless caller has permission
    nhsNumber
  }
}

# Query — look up any user's profile (requires manage_users permission for other users)
query UserProfile($userId: ID!) {
  userProfile(userId: $userId) {
    status
    avatarUrl
    firstName
    lastName
  }
}

# Query — paginated list with optional text search across searchable fields
query UserProfiles($search: String, $status: String, $first: Int, $after: String) {
  userProfiles(search: $search, status: $status, first: $first, after: $after) {
    edges {
      node {
        firstName
        lastName
        status
        avatarUrl
      }
    }
    pageInfo {
      hasNextPage
      endCursor
    }
    totalCount
  }
}

# Mutation — create a profile (typically called after registration)
mutation CreateUserProfile($fields: UserProfileInput!) {
  createUserProfile(fields: $fields) {
    status
    firstName
  }
}

# Mutation — update own profile (or another user's with manage_users permission)
mutation UpdateUserProfile($userId: ID, $fields: UserProfileInput!) {
  updateUserProfile(userId: $userId, fields: $fields) {
    firstName
    lastName
  }
}

# Mutation — transition user status
mutation SetUserStatus($userId: ID!, $status: String!) {
  setUserStatus(userId: $userId, status: $status) {
    status
  }
}

# Mutation — upload avatar
mutation UploadAvatar($file: Upload!) {
  uploadAvatar(file: $file) {
    avatarUrl
  }
}
```

## Acceptance Criteria

```gherkin
# --- Settings validation ---

Given PROFILE_SCHEMA declares 'nhs_number' with encrypted: True and syntek-pyo3 is not installed
When AppConfig.ready() runs
Then ImproperlyConfigured is raised — encrypted fields require syntek-pyo3

Given STATUSES is an empty list
When AppConfig.ready() runs
Then ImproperlyConfigured is raised — at least one status is required

Given STATUS_TRANSITIONS references a status not in STATUSES
When AppConfig.ready() runs
Then ImproperlyConfigured is raised naming the unknown status

Given PROFILE_SCHEMA declares a 'choice' field with no choices list
When AppConfig.ready() runs
Then ImproperlyConfigured is raised

Given PROFILE_SCHEMA declares a field with required: True, no default, and type: 'int'
When a profile is created without that field
Then FieldValidationError is raised — required field missing

# --- Profile schema ---

Given PROFILE_SCHEMA declares 'first_name' with type: 'str' and max_length: 100
When createUserProfile is called with first_name='A' * 101
Then FieldValidationError is raised — max_length exceeded

Given PROFILE_SCHEMA declares 'membership_tier' with choices: [('standard',..),('premium',..)]
When updateUserProfile is called with membership_tier='gold'
Then FieldValidationError is raised — 'gold' is not a declared choice

Given PROFILE_SCHEMA declares 'date_of_birth' with type: 'date'
When updateUserProfile is called with date_of_birth='not-a-date'
Then FieldValidationError is raised — type coercion failed

# --- Encrypted fields ---

Given 'nhs_number' is declared with encrypted: True
When set_field_value is called with a valid NHS number
Then the value is AES-256-GCM encrypted before storage in custom_fields

Given the encrypted 'nhs_number' is read by the owning user
When get_field_value('nhs_number') is called
Then the plaintext NHS number is returned — decryption is transparent

Given EXPOSE_SENSITIVE_TO_SELF is False and the owning user queries nhsNumber via GraphQL
When the resolver runs
Then null is returned — users cannot read their own encrypted fields in this mode

Given an admin with manage_users permission queries nhsNumber
When the resolver runs
Then the decrypted value is returned

# --- Status lifecycle ---

Given STATUSES is ['active', 'inactive', 'archived'] and default is 'active'
When createUserProfile is called without specifying status
Then the profile is created with status 'active'

Given STATUS_TRANSITIONS has 'archived': [] (terminal)
When setUserStatus is called with status='active' for an archived profile
Then InvalidStatusTransitionError is raised — no transitions out of archived

Given STATUS_TRANSITIONS is empty (no restrictions)
When setUserStatus is called with any valid status
Then the transition succeeds

# --- Avatar ---

Given AVATAR_ENABLED is True and a valid JPEG is uploaded via uploadAvatar
When the mutation completes
Then the image is uploaded to Cloudinary with AVATAR_TRANSFORM applied and avatarUrl is returned

Given AVATAR_ENABLED is True and a file exceeding AVATAR_MAX_SIZE_BYTES is uploaded
When the mutation runs
Then it is rejected with a file size error — Cloudinary is not called

Given AVATAR_ENABLED is True and a BMP file is uploaded (not in AVATAR_ALLOWED_FORMATS)
When the mutation runs
Then it is rejected with a format error

Given AVATAR_ENABLED is False
When uploadAvatar mutation is called
Then a disabled error is returned — Cloudinary is not called

# --- Soft delete ---

Given SOFT_DELETE is True and a profile's status is set to 'archived'
When UserProfile.objects.all() is queried
Then the archived profile is not returned

Given SOFT_DELETE is True and a profile's status is set to 'archived'
When UserProfile.all_objects.filter(status='archived') is queried
Then the archived profile is returned

# --- GraphQL permissions ---

Given REQUIRE_ADMIN_FOR_USER_WRITE is True and a non-admin user calls updateUserProfile
      with a userId that is not their own
When the mutation runs
Then PermissionDenied is returned — profile is not changed

Given REQUIRE_ADMIN_FOR_USER_WRITE is True and a user calls updateUserProfile
      with no userId (updating their own profile)
When the mutation runs
Then the update succeeds — users can always update their own profile

# --- Tenant isolation ---

Given two tenants with users of the same email address
When userProfiles is queried by a user of Tenant A
Then only Tenant A's profiles are returned — no cross-tenant leakage

# --- GraphQL type naming ---

Given PROFILE_LABEL is 'Customer'
When the GraphQL schema is generated
Then the type is named CustomerType (not UserProfileType)
```

## Dependencies

- US009 — `syntek-auth` (provides the auth User model; UserProfile has OneToOne FK to User)
- US010 — `syntek-tenancy` (tenant FK on UserProfile; all queries are tenant-scoped)
- US007 — `syntek-pyo3` (required when any field has `encrypted: True`)
- US008 — `syntek-graphql-crypto` (provides SYNTEK*FIELD_KEY*\* env vars for encryption)
- US011 — `syntek-permissions` (REQUIRE_ADMIN_FOR_USER_WRITE enforcement)
- US030 — `syntek-media` (optional — AVATAR_ENABLED requires syntek-media; degrades gracefully)
