## User Story

As a developer, I want a multi-regime data protection module so that I can handle subject access
requests, right to erasure, data portability, consent management, and retention policies —
configurable for any national data protection regulation (GDPR, UK-GDPR, CCPA, LGPD, PIPEDA, POPIA,
PDPA) without code changes.

## Architecture

`syntek-gdpr` handles data **processing consent** and **data subject rights** (GDPR Articles 6, 17,
20 and equivalents). Legal document management (Privacy Policy, T&Cs, DPAs) is handled by
`syntek-legal` (US091). The two modules integrate via a string-reference convention —
`ConsentRecord` may optionally store a `legal_document_version_id` string but carries no FK across
module boundaries.

## Configuration

```python
SYNTEK_GDPR = {
    # Active compliance regimes — determines which rights and obligations apply
    # Supported: 'gdpr', 'uk_gdpr', 'ccpa', 'lgpd', 'pipeda', 'popia', 'pdpa'
    # Multiple regimes can be active simultaneously (e.g. a SaaS serving EU and California)
    'COMPLIANCE_REGIMES': ['gdpr'],

    # Data residency — informational; enforced at infrastructure level, recorded for audit
    'DATA_RESIDENCY_REGION': 'eu',   # 'eu' | 'uk' | 'us' | 'ca' | 'br' | 'za' | 'sg' | custom

    # SAR / data export
    'SAR_EXPORT_FORMATS': ['json', 'csv'],   # Formats offered to the data subject
    'SAR_COMPLETION_DAYS': 30,               # Statutory deadline (GDPR=30, CCPA=45, etc.)
    'SAR_STORAGE_BACKEND': 'minio',          # 'minio' | 'local'

    # Erasure
    'ERASURE_ANONYMISE_FIELDS': True,        # True = anonymise PII; False = hard delete where permitted
    'ERASURE_PRESERVE_AUDIT_CHAIN': True,    # Always True in production — audit records retained

    # Retention
    'RETENTION_POLICIES': {
        # category: days — 0 = no limit; -1 = delete immediately after purpose fulfilled
        'user_profile': 365 * 3,
        'payment_metadata': 365 * 7,    # UK legal requirement
        'audit_logs': 365 * 7,
        'marketing_data': 365,
    },

    # Consent
    'CONSENT_REQUIRED_FOR_PROCESSING': True,  # False for regimes using legitimate interest default
    'CONSENT_EXPIRY_DAYS': 365,               # Days before re-consent is required (0 = never)

    # CCPA-specific
    'CCPA_DO_NOT_SELL_ENABLED': False,        # Enable "Do Not Sell My Personal Information" flow

    # DPA tracking (integrates with syntek-legal)
    'DPA_DOCUMENT_SLUG': 'data-processing',   # Slug of the DPA document type in syntek-legal

    # Notifications
    'SAR_NOTIFICATION_EMAIL': True,           # Notify data subject when SAR export is ready
    'ERASURE_NOTIFICATION_EMAIL': True,       # Notify data subject when erasure is complete
}
```

## Acceptance Criteria

```gherkin
Scenario 1: Subject Access Request collects data across all registered modules
Given COMPLIANCE_REGIMES = ['gdpr'] and a SAR is submitted
When the export task runs
Then all PII across every registered syntek module is collected, structured as a JSON package, and stored with a pre-signed download URL — completing within SAR_COMPLETION_DAYS

Scenario 2: CCPA Do Not Sell flow
Given COMPLIANCE_REGIMES = ['ccpa'] and CCPA_DO_NOT_SELL_ENABLED = True
When a data subject submits a "Do Not Sell" request
Then a DoNotSellRecord is created, all data-sale processing is halted, and the data subject is notified within 15 days

Scenario 3: Right to Erasure with audit chain preservation
Given a Right to Erasure request is submitted
When the erasure task runs
Then PII fields are anonymised across all registered modules (not deleted) and the audit chain records are preserved intact

Scenario 4: Consent enforcement on processing
Given a user's consent for purpose marketing is withdrawn
When any module attempts to process data under that purpose
Then the operation is rejected with a ConsentWithdrawnError

Scenario 5: Retention policy enforcement
Given RETENTION_POLICIES = {'user_profile': 1095}
When the retention task runs
Then user profile records older than 1095 days have PII anonymised and a RetentionLog entry is written

Scenario 6: Auto-discovery of new registered modules
Given a new syntek-* module is installed and registers its GDPR data classes via GDPRRegistry.register()
When a SAR or erasure task runs
Then the new module's data is automatically included with no settings change

Scenario 7: Regime-specific consent configuration
Given COMPLIANCE_REGIMES = ['gdpr', 'ccpa']
When the consent banner configuration is queried
Then a regime-specific consent configuration object is returned that the frontend can use to render the correct banner for each user's jurisdiction

Scenario 8: Data portability export (GDPR Article 20)
Given a data portability request is submitted
When the portability export runs
Then the data is exported in a machine-readable structured format (JSON) covering only data the subject actively provided
```

## Dependencies

- US104 — `syntek-consent` (consent record management — syntek-gdpr depends on syntek-consent for
  ConsentRecord model)
- US013 — `syntek-audit` (immutable audit chain preserved on erasure)
- US016 — `syntek-settings` (per-tenant GDPR configuration)
- US010 — `syntek-tenancy` (tenant-scoped consent records)
- US015 — `syntek-tasks` (async SAR/erasure/retention Celery tasks)
- US087 — `syntek-notifications-email` (SAR ready notification, erasure confirmation)
- US031 — `syntek-documents` (SAR export file storage in MinIO; optional)
