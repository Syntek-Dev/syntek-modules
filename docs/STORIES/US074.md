# US074 — `syntek-manifest`: Module Manifest Spec & CLI Shared Framework

**Last Updated**: 06/03/2026
**Version**: 0.1.0
**Status**: To Do
**Epic**: Epic 1 — Foundation & Tooling

---

## User Story

As a module author, I want a standardised manifest format and a shared Rust CLI library so that every Syntek module can ship its own self-contained installer binary without independently inventing the scaffolding protocol.

## MoSCoW Priority

**Must Have**

## Acceptance Criteria

```gherkin
Given a valid `syntek.manifest.toml` file
When the CLI framework parses it
Then all declared fields (module ID, options, settings keys, entry points) are validated and any missing required fields raise a descriptive error

Given a developer runs a module's installer binary with no arguments
When the binary starts
Then it presents an interactive prompt for each install option declared in the manifest

Given a backend module installer runs and the developer selects options
When the installer completes
Then the correct `SYNTEK_<MODULE>` config block is written to `settings.py` and the app is added to `INSTALLED_APPS` with no duplicate entries

Given a frontend module installer runs and the developer selects options
When the installer completes
Then the project's app root is wrapped with required providers and a minimal usage stub is written to the declared entry file

Given any installer completes
When post-install steps exist (URL patterns, migration commands, native build steps)
Then they are printed as formatted copy-paste snippets — never written to files automatically

Given a `settings.py` already contains a `SYNTEK_<MODULE>` block
When the installer runs again
Then it detects the existing block, warns the developer, and does not overwrite without explicit confirmation
```
