## User Story

As a developer, I want an outbound and inbound webhook module so that I can receive signed inbound
webhooks from external services and dispatch signed outbound events to consumer endpoints — with
automatic retry, dead letter queue, and full delivery logging.

## MoSCoW Priority

**Should Have**

## Architecture — Webhooks vs Events

`syntek-webhooks` handles **external HTTP delivery**. `syntek-bus` (US090) handles **internal domain
event dispatch**. These are complementary layers:

```text
Domain event fired (syntek-bus)
  └─→ Async subscriber: syntek-webhooks outbound dispatcher
        └─→ HTTP POST to registered consumer endpoint (HMAC-signed)
```

When both modules are installed, `syntek-webhooks` auto-subscribes to `syntek-events` and dispatches
outbound HTTP calls for any registered event type. Consuming projects register webhook endpoints via
the GraphQL API and subscribe to event type patterns — no code change required.

## Acceptance Criteria

```gherkin
Given an inbound webhook arrives with an invalid HMAC signature
When the ingestion endpoint processes it
Then it returns 401 and the payload is not processed or stored

Given an inbound webhook has an idempotency key seen in the last 24 hours
When the ingestion endpoint processes it
Then it returns 200 without re-processing the payload

Given an outbound webhook delivery fails
When the retry policy triggers
Then it retries up to 10 times with exponential back-off and jitter before routing to DLQ

Given a domain event is fired via syntek-events
When syntek-webhooks has an outbound rule matching that event type
Then the payload is dispatched to all registered endpoints, each signed with HMAC-SHA256

Given a delivery attempt completes (success or failure)
When the delivery log is queried
Then the full request body, response status, headers, and duration are available

Given a DLQ item exists for a permanently failed delivery
When an admin queries `webhookDeadLetterQueue`
Then the item includes event type, endpoint URL, last error, and attempt count

Given a campaign event is fired by syntek-email-marketing
When a consuming project has registered a webhook for `campaign.send_completed`
Then the event is dispatched to their endpoint signed with HMAC-SHA256
```

## Campaign Webhook Events

`syntek-webhooks` dispatches all events registered by `syntek-email-marketing` (US038) to consuming
project endpoints. No additional configuration is required in US038 — events are fired via
`syntek-events` and `syntek-webhooks` handles HTTP delivery automatically.

Events available for subscription (registered by US038 via US090):

| Event Type                      | Fired When                                      |
| ------------------------------- | ----------------------------------------------- |
| `campaign.send_started`         | Campaign fan-out begins                         |
| `campaign.send_completed`       | All send records created (counts available)     |
| `subscriber.joined`             | Double opt-in confirmed                         |
| `subscriber.unsubscribed`       | Unsubscribe link clicked or API call            |
| `subscriber.bounced.hard`       | Hard bounce confirmed via provider webhook      |
| `automation.step_sent`          | Automation sequence step delivered              |
| `automation.sequence_completed` | All steps in a sequence delivered to subscriber |
