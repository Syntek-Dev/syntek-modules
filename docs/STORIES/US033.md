## User Story

As a developer, I want a pluggable integration framework so that I can register third-party service
adapters, handle OAuth2 PKCE flows, store credentials encrypted per tenant, and receive webhooks
from external services — with consuming projects registering only the integrations they need via
`SYNTEK_INTEGRATIONS` settings.

## Architecture

`syntek-integrations-core` provides the plumbing only — the integration registry, OAuth2 flow,
encrypted credential storage, and webhook dispatch infrastructure. No specific integrations (Slack,
Google, accounting services, etc.) are bundled. Consuming projects register their own adapters, or
install dedicated adapter packages such as `syntek-integrations-accounting` (US109) or
`syntek-integrations-automation` (US108). This keeps the core lean and avoids shipping credentials
configuration for services the project does not use.

## Configuration

```python
SYNTEK_INTEGRATIONS = {
    # List of adapter dotted-path strings to register at startup
    # Each adapter must implement the IntegrationAdapter protocol
    # Example: 'myproject.integrations.MyServiceAdapter'
    'ADAPTERS': [],

    # Encryption key reference for credential storage (resolved from environment)
    # Uses syntek-pyo3 AES-256-GCM field encryption
    'CREDENTIAL_KEY_ENV': 'SYNTEK_INTEGRATION_KEY',

    # Webhook receiver path prefix (mounted in URL conf)
    'WEBHOOK_PATH_PREFIX': '/webhooks/integrations/',

    # HMAC verification — default algorithm for inbound webhook signature checks
    'DEFAULT_HMAC_ALGORITHM': 'sha256',

    # Token refresh — how many seconds before expiry to proactively refresh
    'TOKEN_REFRESH_BUFFER_SECONDS': 300,

    # Audit all OAuth connect/disconnect events via syntek-audit
    'AUDIT_OAUTH_EVENTS': True,
}
```

## Acceptance Criteria

```gherkin
Given an IntegrationAdapter is registered in SYNTEK_INTEGRATIONS['ADAPTERS']
When AppConfig.ready() runs
Then the adapter is available in the IntegrationRegistry and its supported scopes and webhook event types are discoverable via GraphQL

Given an integration OAuth2 flow is initiated
When the consumer calls initiateOAuth(provider)
Then a PKCE code verifier and challenge are generated, state is stored per-session, and the authorisation URL is returned

Given an OAuth2 callback arrives with a valid code and state
When the callback is processed
Then the access and refresh tokens are encrypted via syntek-pyo3 and stored in OAuthCredential linked to the tenant

Given a stored access token is within TOKEN_REFRESH_BUFFER_SECONDS of expiry
When an integration request is made
Then the token is refreshed automatically using the stored refresh token before the request proceeds

Given a tenant disconnects an integration
When the disconnect mutation is called
Then all stored credentials are deleted (not soft-deleted) and the integration is marked inactive

Given two tenants configure the same integration type
When each tenant's integration runs
Then they use their own credentials and cannot access each other's data (tenant isolation enforced at query level)

Given an inbound webhook arrives at the receiver path
When HMAC signature verification passes
Then the event is dispatched to the registered adapter's handle_webhook method

Given an inbound webhook arrives with an invalid HMAC signature
When verification runs
Then the request is rejected with 401 and not dispatched
```

- US090 — `syntek-bus` (fires `integration.connected` and `integration.disconnected` events)
