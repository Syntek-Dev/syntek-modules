# US011 — `syntek-permissions`: RBAC Module

**Last Updated**: 10/03/2026 **Version**: 0.2.0 **Status**: To Do **Epic**: Epic 3 — Core Backend
Modules

## User Story

As a developer, I want a permissions module so that I can define roles, assign permissions, and
enforce access control at the ORM layer — with full control over the role strategy (group-based,
per-user, or hybrid), role names and labels, object-level grants, and optional field-level
visibility — all configured from `settings.py` without touching module source.

## MoSCoW Priority

**Must Have**

## Configuration Contract

All behaviour is controlled via `SYNTEK_PERMISSIONS` in `settings.py`. Validated at startup in
`AppConfig.ready()`.

```python
SYNTEK_PERMISSIONS = {
    # --- Role strategy ---
    'ROLE_STRATEGY': 'group',        # 'group' | 'per_user' | 'hybrid'
                                     # 'group'    — roles assigned to groups; users inherit from their group
                                     # 'per_user' — each user gets an individually assigned permission set
                                     # 'hybrid'   — group roles + per-user overrides (additions or restrictions)

    # --- Role definitions ---
    # If empty, built-in defaults apply (admin / manager / member).
    # Operators may define their own named roles with custom labels and inheritance.
    'ROLES': [
        {'name': 'admin',   'label': 'Administrator', 'level': 100, 'inherits': []},
        {'name': 'manager', 'label': 'Manager',       'level': 50,  'inherits': ['member']},
        {'name': 'member',  'label': 'Member',         'level': 10,  'inherits': []},
    ],

    # --- Role management ---
    'ALLOW_CUSTOM_ROLES': True,      # bool — operators can create additional roles at runtime via admin/API
    'DEFAULT_ROLE': 'member',        # str — role automatically assigned to new users on registration

    # --- Granularity ---
    'OBJECT_LEVEL': True,            # bool — enable per-object permission grants (ObjectPermission model)
    'FIELD_LEVEL': False,            # bool — enable per-field visibility control
                                     #        (expensive; only enable if the application requires it)

    # --- Enforcement ---
    'SUPERUSER_BYPASS': True,        # bool — superusers bypass all permission checks
    'AUDIT_CHECKS': False,           # bool — log every permission evaluation via syntek-audit
                                     #        (expensive in high-traffic systems; disabled by default)
}
```

**Key rules:**

- `ROLES` entries must have unique `name` values. Circular inheritance raises
  `ImproperlyConfigured`.
- `DEFAULT_ROLE` must match a name in `ROLES` (or a built-in default) — validated at startup.
- `ROLE_STRATEGY: 'per_user'` disables group-based inheritance; every user has their own permission
  set.
- `ROLE_STRATEGY: 'hybrid'` allows per-user grants to add to or restrict group permissions.
- `FIELD_LEVEL: True` enables the `FieldPermission` model; resolvers must call
  `has_field_permission(user, model, field)`.

## Acceptance Criteria

```gherkin
# --- Settings validation ---

Given `DEFAULT_ROLE` is set to 'editor' and 'editor' is not in `ROLES`
When `AppConfig.ready()` runs
Then `ImproperlyConfigured` is raised — DEFAULT_ROLE must match a defined role name

Given `ROLES` contains a circular inheritance chain (e.g. 'a' inherits 'b', 'b' inherits 'a')
When `AppConfig.ready()` runs
Then `ImproperlyConfigured` is raised naming the cycle

# --- Group-based strategy ---

Given `ROLE_STRATEGY` is 'group' and a user belongs to the 'manager' group
When a permission check runs for a 'manager'-level resource
Then access is granted via group membership

Given `ROLE_STRATEGY` is 'group' and a permission bundle is assigned at the group level
When a group member makes a request
Then they inherit all permissions in the bundle without individual assignment

# --- Per-user strategy ---

Given `ROLE_STRATEGY` is 'per_user'
When a new user registers
Then they are assigned the DEFAULT_ROLE permission set directly (no group required)

Given `ROLE_STRATEGY` is 'per_user' and a user is granted 'orders:write'
When the orders:write resolver runs for that user
Then access is granted regardless of any group membership

# --- Hybrid strategy ---

Given `ROLE_STRATEGY` is 'hybrid' and a user's group has 'member' role
When a per-user override grants 'reports:read' to that specific user
Then the user has both 'member' group permissions AND 'reports:read'

Given `ROLE_STRATEGY` is 'hybrid' and a user's group has 'admin' role
When a per-user restriction removes 'billing:write' from that specific user
Then the user cannot write billing records even though their group role would allow it

# --- Custom roles ---

Given `ALLOW_CUSTOM_ROLES` is True
When an admin creates a role named 'auditor' via the createRole mutation
Then the role is persisted and can be assigned to users or groups

Given `ALLOW_CUSTOM_ROLES` is False
When the createRole mutation is called
Then a clear error is returned — runtime role creation is disabled for this deployment

# --- Object-level ---

Given `OBJECT_LEVEL` is True and a user with 'member' role attempts to access a resource owned by another user
When the ORM-layer permission check runs
Then access is denied and a 403 is returned

Given `OBJECT_LEVEL` is True and an explicit ObjectPermission grant exists for a user on a specific record
When that user accesses the record
Then access is granted even if their role would not normally allow it

# --- Field-level ---

Given `FIELD_LEVEL` is True and a user does not have 'read' permission on the 'salary' field of Employee
When a GraphQL query requests the salary field
Then null is returned for that field and a field-permission error is appended to the errors array

Given `FIELD_LEVEL` is False
When the FieldPermission model is queried
Then an error is raised — field-level permissions are not enabled for this deployment

# --- Tenant scoping ---

Given tenant isolation is active
When any permission check runs
Then it is always scoped to the active tenant — cross-tenant grants cannot be created or honoured

# --- Escalation ---

Given a privilege escalation is attempted (a 'member' attempting a 'superuser' action)
When the permission check runs
Then it is denied and the attempt is logged via syntek-audit

Given `SUPERUSER_BYPASS` is True and a superuser makes any request
When the permission middleware runs
Then the check is skipped entirely for that user

Given `SUPERUSER_BYPASS` is False
When a superuser makes a request
Then the full permission check runs — superuser status alone is not sufficient
```
