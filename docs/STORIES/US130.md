## User Story

As an operator deploying `syntek-auth`, I want provider-specific OAuth/OIDC adapters for popular
social login services (Google, Apple, Facebook, Instagram, LinkedIn, Microsoft MSA, Discord,
Twitter/X, Slack, Zoom, Patreon, Twitch, TikTok) so that users can authenticate via their existing
accounts without custom integration code â€” with each provider's unique protocol details (client
secret generation, token validation, user profile extraction, MFA gating) handled transparently by
the module.

## MoSCoW Priority

**Must Have**

- Google adapter with proper OIDC discovery and user profile extraction
- Apple adapter with signed JWT client secret generation and ES256 token handling
- Facebook adapter with Graph API user info endpoint
- Instagram adapter (Facebook Graph API, same credentials as Facebook)
- Account linking and MFA-gated flow (reusing existing `PendingOAuthSession`)

**Should Have**

- LinkedIn adapter (OIDC-compliant)
- Microsoft MSA adapter (v2.0 endpoint)
- Discord adapter (OAuth2 + Graph API)
- Slack adapter (OAuth2 + identity API)
- Zoom adapter (OIDC-compliant)

**Could Have**

- Twitter/X adapter (OAuth 2.0 with PKCE)
- Patreon adapter (OAuth2)
- Twitch adapter (OAuth2 + Client-ID header)
- TikTok adapter (non-standard OAuth2, PKCE-like flow)

**Won't Have**

- Provider-specific UI (social buttons) â€” this is a separate frontend story (`@syntek/ui-auth`)
- Custom webhook event handling per provider
- Deep linking or provider-specific session recovery

## Configuration Contract

All providers are configured via `SYNTEK_AUTH['OAUTH_PROVIDERS']` in `settings.py`. Each adapter
consumes this config at instantiation and must validate all required keys at startup.

### Adapter Interface Contract

All provider adapters must implement:

```python
from dataclasses import dataclass
from typing import Dict, Any, Optional

@dataclass
class SocialUserProfile:
    provider_id: str              # 'google', 'apple', etc.
    email: str                    # Must be present (or error at exchange time)
    name: Optional[str]           # Full name or None
    avatar_url: Optional[str]     # Profile picture URL or None
    raw: Dict[str, Any]          # Raw provider response for debugging

class OAuthAdapter(ABC):
    """Base adapter class that all providers inherit from."""

    @abstractmethod
    def get_auth_url(self, redirect_uri: str, state: str, nonce: str) -> str:
        """
        Return the provider's OAuth/OIDC authorization URL.

        Args:
            redirect_uri: The callback URL registered with the provider
            state: CSRF token (stored in session or temporary table)
            nonce: OIDC nonce (for token validation)

        Returns:
            Full URL to send the user to
        """
        pass

    @abstractmethod
    async def exchange_code(self, code: str, redirect_uri: str) -> Dict[str, Any]:
        """
        Exchange authorization code for tokens.

        Args:
            code: Authorization code from provider callback
            redirect_uri: Must match the registered redirect_uri

        Returns:
            Dict containing at minimum: 'access_token', 'id_token' (if OIDC), 'token_type', 'expires_in'
        """
        pass

    @abstractmethod
    async def get_user_info(self, token_response: Dict[str, Any]) -> SocialUserProfile:
        """
        Extract user profile from token response.

        Args:
            token_response: Dict returned by exchange_code()

        Returns:
            SocialUserProfile with provider_id, email, name, avatar_url, raw

        Raises:
            OAuthError: If email is missing or token validation fails
        """
        pass

class GoogleOAuthAdapter(OAuthAdapter):
    """Google OIDC adapter."""
    pass

class AppleOAuthAdapter(OAuthAdapter):
    """Apple Sign in adapter with JWT client secret."""
    pass

class FacebookOAuthAdapter(OAuthAdapter):
    """Facebook OAuth2 + Graph API adapter."""
    pass

# ... LinkedIn, Microsoft, Discord, Twitter adapters follow same interface
```

### Key Rules for Configuration

- All secrets (`client_secret`, `private_key`) come from environment variables â€” never hardcoded.
- `mfa_enforced: True` means the adapter routes through `PendingOAuthSession` before issuing full
  tokens.
- Each provider config must pass validation in `AppConfig.ready()` â€” missing required keys raise
  `ImproperlyConfigured`.
- Apple's `private_key` is a PEM-encoded PKCS#8 string; the adapter generates a signed JWT client
  secret valid for 1 hour (max 6-month key lifespan per Apple policy).
- Twitter/X requires PKCE; the `code_verifier` is stored in `PendingOAuthSession.metadata` and
  re-used during code exchange.
- Account enumeration must be prevented: if a provider user already has a local account (matched by
  email), return the same success response as a new account flow (no error).

# --- MFA gating for all adapters ---

Given a provider has `mfa_enforced: True` and the user's account is not yet created
When the OAuth callback is processed
Then a `PendingOAuthSession` is created with the user's provider_id, email, name, avatar_url,
and a temporary session token is returned to the client

Given a user has a pending OAuth session
When they authenticate via local MFA (TOTP, passkey, etc.)
Then the `PendingOAuthSession` is consumed, a full session token pair is issued,
and the user's avatar_url and name are stored (overwriting any previous OIDC profile data)

Given a provider has `mfa_enforced: False` and is a trusted provider (GitHub, GitLab, etc.)
When the OAuth callback is processed
Then a full session token pair is issued immediately (no MFA gate)

Given a provider has `mfa_enforced: True` but the user already has an account (email match)
When the OAuth callback is processed
Then a `PendingOAuthSession` is still issued (MFA enforcement is not bypassed for existing accounts)

# --- Account linking and PII encryption ---

Given a user logs in via Google and their email matches an existing local account
When the login flow completes
Then the OAuth profile data (email, name, avatar_url) is encrypted via `encrypt_field` before being
stored in the user record; the `OAuthProfile` table links the provider_id to the user

Given two different OAuth providers return the same email for a user
When the user authenticates via the second provider
Then both provider profiles are linked to the same user account (no duplicate created)

Given an OAuth adapter extracts email, name, and avatar_url from the provider
When the user record is updated
Then email and name are encrypted via `EncryptedField` / `encrypt_field` at the Django model layer;
avatar_url is also encrypted (it is PII when associated with a user)

# --- Account enumeration prevention ---

Given a user signs up via Google with email test@example.com
When someone later attempts to sign up via Google with the same email
Then the response is identical to the first signup response â€” no error indicating the email exists

Given a user's email does not exist in the system and they authenticate via an OAuth provider
When the provider returns their email
Then a new `PendingOAuthSession` is created (if MFA-gated) or a full session is issued (if not)

# --- Error handling ---

Given an OAuth adapter receives an invalid client_id from the provider
When the authorization URL is being generated
Then a clear `OAuthError` is raised with context (provider name, reason)

Given the provider's discovery URL returns a 404
When OIDC discovery runs
When a clear `OAuthError` is raised â€” discovery failed; check the provider configuration

Given a code exchange request times out (provider is slow/offline)
When the `oidcCallback` mutation is called
Then the mutation returns a 504 error (service unavailable) and the user is directed to retry

Given a user's email is missing from the provider's response
When the profile extraction completes
Then the mutation returns `oauth_error: 'missing_email'` with a user-friendly message

Given the provider returns an ID token but validation fails (expired, invalid signature)
When token validation runs
Then the login is rejected with `oauth_error: 'token_invalid'` and no user data is revealed

# --- Existing PendingOAuthSession wiring (from US009) ---

Given a user has initiated OAuth login and received a temporary session token
When they authenticate via their configured local MFA
Then the `PendingOAuthSession` is automatically consumed by the MFA verification mutation

Given a PendingOAuthSession has expired (default TTL: 1 hour)
When a user attempts to complete their OAuth + MFA flow
Then the flow is rejected with `session_expired` error

Given a PendingOAuthSession is created during OAuth callback
When the `metadata` field stores `{'avatar_url': '...', 'name': '...', ...}`
Then this metadata is persisted and available to the MFA verification mutation for final profile creation

## Story Points

**26**

**Rationale:** Thirteen provider adapters with varying protocol requirements (OIDC, OAuth2, Graph
APIs, JWT generation, PKCE, non-standard flows), full error handling, account linking, MFA gating
integration, PII encryption, account enumeration prevention, allowlist update, and comprehensive
test coverage across all success and error paths. Estimates assume use of existing libraries
(python-jose, requests) and Django ORM. Apple adapter's JWT generation adds complexity; Twitter's
and TikTok's PKCE flows add protocol complexity; Twitch's mandatory Client-ID header and TikTok's
non-standard `client_key` naming each add implementation overhead. High test count due to
provider-specific requirements. Delivery across three sprints (06A, 06B, 06C) for an experienced
team.

## Notes

- **Backwards compatibility:** US009 is complete and in production; this story only _adds_ new
  adapters, it does not modify the existing OIDC client or `PendingOAuthSession` model.
- **No UI in this story:** Social buttons, provider selection flows, and MFA UX are separate
  frontend stories (`@syntek/ui-auth` adaptive form component).
- **Environment variable naming:** Adopters define secrets via `env()` (django-environ or similar);
  example keys given here are conventional but not prescriptive.
- **Testing credentials:** Implement adapters with sandbox/test credentials from each provider;
  actual provider applications must be created by the operator deploying the module.
- **PKCE for Twitter/X:** Required by provider policy; implementation must use SHA256 challenge
  method.
- **Account enumeration:** Golden rule â€” if email matches an existing account, the response must be
  identical to a new account signup. Log the match server-side but never reveal it to the client.
- **Encryption:** All PII from provider profiles (email, name, avatar URL) must be encrypted before
  storage. Use `encrypt_field()` at the service layer, or `EncryptedField` at the model layer.
- **Provider allowlist:** All MFA-gated providers must be on the US076 allowlist or have
  `mfa_enforced: True` in config. Built-in providers (GitHub, GitLab, Okta, Entra, Authentik,
  Keycloak, Defguard) are already allowlisted and can be configured via generic OIDC. Instagram is
  already present in `MFA_GATED_PROVIDERS`; Slack, Zoom, Patreon, Twitch, and TikTok must be added
  as part of this story's allowlist update task.
- **Thirteen adapters total:** Google, Apple, Facebook, Instagram (Must Have) + LinkedIn, Microsoft
  MSA, Discord, Slack, Zoom (Should Have) + Twitter/X, Patreon, Twitch, TikTok (Could Have).
  Delivered across SPRINT-06A, SPRINT-06B, and SPRINT-06C.
- **Language:** British English (en_GB).
- **Timezone:** Europe/London (UTC+0 or UTC+1 depending on DST).
